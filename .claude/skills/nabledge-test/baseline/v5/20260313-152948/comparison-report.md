## 分析を受けた仮説

**仮説1: qa-001 の低検出はバッチ起動知識ファイルの具体的コマンド例不足が原因**

- **データ根拠**: ベンチマーク mean=55.0%（std=18.8%）、広域チェック 3/8（37.5%）。未検出キーワードは「コマンドライン引数」「アクションのクラス名」「リクエストID」「javaコマンド」「ハンドラキュー」のすべてが起動コマンドの具体的手順。qa-004「バッチエラーハンドリング」が前回57%→今回100%に改善したのと対照的。
- **実装箇所**: `.claude/skills/nabledge-5/` 以下のバッチ関連知識ファイル（`batch-*.json` 等）
- **予測**: 知識ファイルにバッチ起動コマンド例（`-requestPath`、`java` コマンド形式）を追加することで、次回ベンチマーク mean が 80%+ に改善し std も低下する。

**仮説2: CA Processing Flow の utility class 名未記述はコード解析テンプレートの記述指示不足**

- **データ根拠**: ca-004（mean 91.5%）・ca-005（mean 90.6%）で Processing Flow の `SessionUtil`（ca-004: 2/10試行で検出, ca-005: 1/10試行で検出）と `BeanUtil`（ca-004: 3-4/10, ca-005: 4/10）が一貫して未検出。クラス名を `BeanUtil.copy()` 形式で書いた試行のみ検出成功。
- **実装箇所**: `.claude/skills/nabledge-5/workflows/code-analysis.md` の Processing Flow 生成指示
- **予測**: テンプレートに「Processing Flow では利用する Nablarch ユーティリティクラス名をメソッド呼び出し形式（`BeanUtil.copy()`、`SessionUtil.delete()`）で明示すること」の指示を追加することで、次回ベンチマーク mean が ca-004/ca-005 ともに 95%+ に改善する。
