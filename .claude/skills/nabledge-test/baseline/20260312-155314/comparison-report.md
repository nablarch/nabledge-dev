## 分析を受けた仮説

### 仮説1: qa-004の後退はナレッジ再生成で特定キーワードのhints/セクション構造が変化したことによる

**データ根拠**: qa-004が 7/7→4/7 (-3項目)。未検出の `TransactionAbnormalEnd`・`ProcessAbnormalEnd`・`ResumeDataReader` はバッチエラーハンドリング固有のクラス名。

**実装参照**: `.claude/skills/nabledge-6/knowledge/` 内のバッチ関連ナレッジファイル (features/processing/ など)。

**予測**: qa-004の回答に上記キーワードが含まれていない場合、知識検索が到達できていないことが確定する。知識ファイルのhints に `TransactionAbnormalEnd`・`ResumeDataReader` を追加することで検出率が回復するはず。確認方法: qa-004の `response.md` を手動確認。

### 仮説2: CAの速度改善はhints追加によるpre-filterの効率化に起因する

**データ根拠**: CAの平均実行時間が -41.5%。特に知識検索ステップが速い試行では「hints pre-filter」で不要セクションを早期排除できていると推測される (ca-004の試行では知識検索2〜10秒の変動)。

**実装参照**: `.claude/skills/nabledge-6/workflows/code-analysis.md` の知識検索ステップと section-judgement ワークフロー内 Step 0。

**予測**: hintsが充実したナレッジファイルほど pre-filter で無関係セクションが排除されるため、知識検索 (Step 2) が短縮される。逆に hints が薄いファイルは full-text search に依存し時間がかかる。

### 仮説3: ca-003の後退はProjectSearchAction特有のナレッジファイル構造変化による

**データ根拠**: ca-003のみ 33/33→30/33 に後退。ca-001・ca-002は横ばい、ca-004・ca-005は改善。ProjectSearchActionが参照するナレッジセクションで再生成後に構造・内容が変わった箇所があると推測。

**予測**: ca-003の `output/` ファイルを前回の baseline の対応ファイルと比較することで、どのセクションの内容が薄くなったかを特定できる。`ca-003 response.md` または output ファイルを確認し、未検出の `BeanUtil`・`SessionUtil` (ProcessingFlow) と `UniversalDao` (NablarchUsage) のセクション記載状況を調べる。
