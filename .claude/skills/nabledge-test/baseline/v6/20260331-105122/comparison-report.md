## 分析を受けた仮説

**仮説A: qa-001 general_select_tagアスペクトの非決定的インクルージョン**
- 根拠: 3試行中1回のみn:select比較セクションを含む（mean=75%）
- 実装: スキルの回答生成プロセスでn:codeSelect専用回答で十分と判断した場合にn:select比較を省略する
- 予測: qa-001期待値からgeneral_select_tagを削除するか、質問を「コード値プルダウン vs 一般プルダウン」に変更すれば安定100%が期待できる

**仮説B: ca-002の実行時間増加は並列実行時のファイルシステム競合**
- 根拠: 前回101秒→今回193秒（+91%）。前回は単独実行、今回はca-001, ca-002, ca-003を並列実行
- 実装: `.nabledge/`出力ファイル書き込み時に複数エージェントが同時アクセス
- 予測: シリアル実行で再計測すれば前回と同等の100-120秒程度に収束する

**仮説C: CA Overview は主要クラス2-3個に絞り込む設計**
- 根拠: ca-001でSessionUtil（補助ユーティリティ）がOverviewに出ない、ca-002でZipCodeData（DTOクラス）が出ない。一方ca-003はメイン処理クラスが全て出る（100%）
- 実装: スキルの概要生成ロジックが処理の中心クラスを選択的に列挙する
- 予測: CA Overview 期待値をアーキテクチャ的に中心的なクラス（Action、主要Nablarchクラス）に限定すれば、ca-001/ca-002のOverview検出率も100%に改善できる
