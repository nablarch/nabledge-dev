## 分析を受けた仮説

**H1: 22-B-12 変更は v6 に破壊的影響を与えていない**
- データ根拠: CA 3 シナリオ全てで検出率完全一致 (37/37, 31/32, 36/37)。22-B-12 の create-side 変更 (resolver, _sub_header, rst_normaliser) はいずれも v1.x が対象で、v6 の generated content は前回と同一のはず。
- 関連実装: `tools/rbkc/scripts/create/resolver.py` (v6 は include パターンを使わないため挙動変化なし)、`scripts/create/converters/xlsx_common.py` (v6 は .xlsx bessatsu のみ / preamble 1 セル問題は .xls 固有)。
- 予測: 同コミットで再計測すれば ca-* は同値、qa-* も mean ±1件 の範囲で揺らぐ。

**H2: qa-001 ベンチマークの variance 0 は artifact で、本来 variance は前回値に近い**
- データ根拠: 前回 3 試行で 3/4, 2/4, 3/4 の分布 (SD=21.6pp)、今回 6/8 x3 (SD=0)。3 試行では分布幅が安定推定できない。
- 関連実装: qa-001 は `n:select` の OR 条件検出が LLM 応答に揺さぶられる (memory 参照: hints 欠落で qa-001 benchmark 試行 1/2 で 62.5% が既知)。
- 予測: 5〜10 試行に増やすと variance は前回水準 (20pp 程度) に戻る可能性が高い。variance が縮んだまま安定すれば、改善と判定できる。

**H3: ca-003 の -69 秒は Claude Sonnet の内部キャッシュ効果 (同一 session / 近接時刻) 由来**
- データ根拠: 検出率は完全不変だが実行時間だけ -19.2%。
- 関連実装: 計測側は caching なし (isolated subagent)。ただし Anthropic API の prompt cache で short-term 繰り返し呼出が速くなる可能性あり。
- 予測: 日を跨いで再計測すると前回値に戻る可能性がある。これを劣化改善シグナルとみなすには 1 週間後の再計測で裏取りが必要。
