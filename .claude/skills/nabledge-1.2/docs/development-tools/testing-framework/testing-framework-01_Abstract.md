# 自動テストフレームワーク

## 

自動テストフレームワークの特徴:

- **JUnit4ベース**: JUnit4のアノテーション（`@Test`、`@Before`等）、assertメソッド、Matcherクラスなど、JUnit4の機能を使用する。
- **テストデータの外部化**: テストデータをExcelファイルに記述できる。DB準備データや期待テスト結果をExcelに記載し、自動テストフレームワークAPIを通じて使用できる。
- **Nablarch特化のテスト補助機能**: トランザクション制御やシステム日付設定など、Nablarchアプリケーションに特化したAPIを提供する。

**推奨規約**:
- 1テストメソッドにつき1シート用意する
- シート名はテストメソッド名と同名にする（例: `@Test public void testInsert()` → シート名 `testInsert`）

> **注意**: これは制約ではなく推奨事項。テストメソッド名とExcelシート名が同名でなくても正しく動作する。今後の機能追加はこの規約をデフォルトとして開発されるため準拠を推奨する。規約を変更する場合はプロジェクト内で統一すること。

## 自動テストフレームワークの構成

| 構成物 | 説明 | 作成者 |
|---|---|---|
| テストクラス | テスト処理を記述する | アプリケーションプログラマ |
| テスト対象クラス | テスト対象となるクラス | アプリケーションプログラマ |
| Excelファイル | テストデータを記載する。自動テストフレームワークAPIでデータを読み取れる | アプリケーションプログラマ |
| コンポーネント設定ファイル・環境設定ファイル | テスト実行時の各種設定を記載する | アプリケーションプログラマ（個別テストに固有の設定が必要な場合） |
| 自動テストフレームワーク | テストに必要な機能を提供する | — |
| Nablarch Application Framework | フレームワーク本体（本機能の対象外） | — |

## シート内の構造

データの種類に依らない共通書式:
- データ1行目は「データタイプ=値」の形式でデータタイプと値を記載する
- 2行目以降の書式はデータタイプにより異なる
- データの個数は複数記述できる

**データタイプ一覧**:

| データタイプ名 | 説明 | 設定する値 |
|---|---|---|
| SETUP_TABLE | テスト実行前にDBに登録するデータ | 登録対象のテーブル名 |
| EXPECTED_TABLE | テスト実行後の期待するDBデータ。省略したカラムは比較対象外 | 確認対象のテーブル名 |
| EXPECTED_COMPLETE_TABLE | テスト実行後の期待するDBデータ。省略したカラムには:ref:`default_values_when_column_omitted`が設定されているものとして扱う | 確認対象のテーブル名 |
| LIST_MAP | `List<Map<String,String>>`形式のデータ | シート内で一意になるID（期待値のID、任意の文字列） |
| SETUP_FIXED | 事前準備用の固定長ファイル | 準備ファイルの配置場所 |
| EXPECTED_FIXED | 期待値を示す固定長ファイル | 比較対象ファイルの配置場所 |
| SETUP_VARIABLE | 事前準備用の可変長ファイル | 準備ファイルの配置場所 |
| EXPECTED_VARIABLE | 期待値を示す可変長ファイル | 比較対象ファイルの配置場所 |
| MESSAGE | メッセージング処理のテストで使用するデータ | 固定値（`setUpMessages` または `expectedMessages`） |
| EXPECTED_REQUEST_HEADER_MESSAGES | 要求電文（ヘッダ）の期待値を示す固定長ファイル | リクエストID |
| EXPECTED_REQUEST_BODY_MESSAGES | 要求電文（本文）の期待値を示す固定長ファイル | リクエストID |
| RESPONSE_HEADER_MESSAGES | 応答電文（ヘッダ）を示す固定長ファイル | リクエストID |
| RESPONSE_BODY_MESSAGES | 応答電文（本文）を示す固定長ファイル | リクエストID |

## コメント

セル内に `//` から開始する文字列を記載した場合、そのセルから右のセルは全て読み込み対象外となる。

## マーカーカラム

カラム名が**半角角括弧で囲まれている**場合（例: `[no]`）、そのカラムは「マーカーカラム」とみなされ、テスト実行時には読み込まれない。コメントは右端方向のセルしか無効化できないため左端・中央のセルには使えないが、マーカーカラムはその制約がない。全データタイプで使用可能。

## セルの書式

- 文字列のみ使用する。テストデータ作成前に全セルの書式を文字列に設定しておくこと。
- 罫線やセルの色付けは任意。

> **警告**: Excelファイルに文字列以外の書式でデータを記述した場合、正しくデータが読み取れなくなる。

## セルへの特殊な記述方法

| 記述方法 | 自動テスト内での値 | 説明 |
|---|---|---|
| `null`/`Null`/`NULL` | null | null（半角、大文字小文字不問）と記述するとnull値として扱う |
| `"null"`/`"NULL"` | 文字列のnull | ダブルクォート（半角・全角問わず）で囲まれた場合は前後のクォートを除去した文字列として扱う。文字列中のダブルクォートはエスケープ不要 |
| `"1⊔"` | 1⊔ | セルの値にスペースがあることを解りやすくする目的で使用できる（⊔ = 半角スペース） |
| `"⊔"` | ⊔ | セルの値が半角スペースであることを明示する |
| `"１△"` | １△ | セルの値に全角スペースがあることを解りやすくする目的で使用できる（△ = 全角スペース） |
| `"△△"` | △△ | セルの値が全角スペース2つであることを明示する |
| `${systemTime}` | システム日時 | SystemTimeProvider実装クラスから取得したTimestampの文字列形式（例: `2011-04-11 01:23:45.0`） |
| `${updateTime}` | システム日時 | `${systemTime}`の別名。特にDBのタイムスタンプ更新時の期待値として使用 |
| `${setUpTime}` | コンポーネント設定ファイルに記載された固定値 | DBセットアップ時のタイムスタンプに決まった値を使用したい場合 |
| `${文字種,文字数}` | 指定した文字種を指定した文字数分まで増幅した値 | 使用可能な文字種: 半角英字、半角数字、半角記号、半角カナ、全角英字、全角数字、全角ひらがな、全角カタカナ、全角漢字、全角記号その他、外字。単独・組み合わせ使用可能 |
| `${binaryFile:ファイルパス}` | BLOB列に格納するバイナリデータ | BLOB列にファイルのデータを格納したい場合。ファイルパスはExcelファイルからの相対パスで記述 |

> **凡例**: ⊔ は半角スペースの意、△は全角スペースの意

## テストメソッド記述方法

テストメソッドに`@Test`アノテーションを付与する（JUnit4のアノテーションを使用）。

```java
public class SampleTest {

    @Test
    public void testSomething() {
        // テスト処理
    }
}
```

> **注意**: `@Before`や`@After`などのアノテーションも使用できる。テストメソッド前後にリソースの取得解放などの共通処理を記述できる。詳細は [using_junit_annotation](testing-framework-03_Tips.md) を参照。

## テストメソッドの実行順序に依存しないテストを作成する

- 実行順序によらず同じ結果になるようにする。クラス単体・複数まとめて実行しても同じ結果でなければならない
- 本フレームワークではテスト中にコミットが行われるため、前後のテストでDBの内容が変更される可能性が高い。自テストクラスで必要な事前条件は全て自テストクラス内で準備すること
- これにより得られる効果:
  - テストの実行順序によって偶然テストが失敗したり偶然成功する、という事態を防ぐ
  - そのテストのデータまたはソースコードだけで、事前条件が分かる
- マスタデータのような読み取り専用テーブルの準備は、共通のExcelファイルに記載しテスト実行前に1回だけ実行するか、事前準備済みの前提でテストを実行する
- 共通Excelファイル運用のメリット:
  - マスタ系のデータをプロジェクト全体で再利用できる
  - テストデータのメンテナンスが容易になる
  - テスト実行速度が上がる
- マスタデータ投入には[master_data_setup_tool](../toolbox/toolbox-01_MasterDataSetupTool.md)を使用する。[04_MasterDataRestore](testing-framework-04_MasterDataRestore.md)によりテスト内で発生したマスタデータの変更をテスト終了時に自動的に元の状態に戻せる

## テストデータは全てExcelシートに記述する

- テストソースコード中にはテストデータを記載せず、全てExcelシートに記載すること
- Excelシートを見ればテストケースのバリエーションを把握できる
- テストデータはExcelシート、テストロジックはテストソースコードと役割分担が明確になる
- Excelシートを読み込む構造にしておくことで、容易にテストケースを追加できる
- テストソースコードの重複を大幅に削減できる

## Excelによるテストデータ記述

DB準備データやDB検索結果など、テストデータの記述にExcelファイルを使用できる。JavaソースコードよりExcelのほうが可読性・編集しやすさで有利。

**命名規約**: ExcelファイルとExcelファイルパスには推奨規約がある。規約に従うことで、テストクラスでディレクトリ名・ファイル名を明示指定せずにファイルを読み込める。明示的なパス指定で任意の場所のExcelファイルを読み込むことも可能。

## パス、ファイル名に関する規約

Excelファイルのファイル名・パスに関する推奨規約:

- Excelファイル名はテストソースコードと同じ名前（拡張子のみ異なる）にする
- Excelファイルはテストソースコードと同じディレクトリに配置する

| ファイルの種類 | 配置ディレクトリ | ファイル名 |
|---|---|---|
| テストソースファイル | `<PROJECT_ROOT>/test/jp/co/tis/example/db/` | `ExampleDbAcessTest.java` |
| Excelファイル | （同上） | `ExampleDbAcessTest.xls` |

ExcelファイルはExcel2003以前の形式（拡張子 `xls`）とExcel2007以降の形式（拡張子 `xlsx`）の両方に対応している。
