## Nablarch のバージョンアップ方針

Nablarchが提供するコンテンツ(詳細は [Nablarch の製品構成](../../about/about-nablarch/about-nablarch-contents.md#nablarch-の製品構成) をご参照ください)に対するバージョンアップ方針について説明します。

### 更新内容

Nablarchが提供するコンテンツに対して実施されるバージョンアップでは、下記の更新が実施されます。

| コンテンツに対して実施する更新内容 |  |  | 説明 | リリース頻度 |
|---|---|---|---|---|
| 不具合対応 | Nablarch Application FW(NAF)、 Nablarch Testing FW(NTF)の不具合 | セキュリティ・運用レベルに致命的な不具合対応 | NAF、NTFの不具合の中でも、 セキュリティや運用レベルに致命的な影響を与える不具合対応です。セキュリ ティホールが検出された場合などが該当します。 | 不具合検出後直ちに対応し、随時リリースします。 |
|  |  | その他の不具合対応 | 通常の一般的な不具合です。NAFだけではなく、例えば不成功となるべきテス トが成功してしまうようなNTFの不具合なども該当します。 | 不具合検出後、影響範囲を見定めリリーススケ ジュールを策定し、リリースします。 |
|  | NAF、NTF以外の不具合 |  | Nablarch標準の改訂や、チュートリアル用サンプルの不具合対応など、製造成 果物に直接影響しない不具合対応です。 | 半期に一度程度の頻度でリリースを予定してい ます。 |
| 機能追加 |  |  | NAFやNTFに対する機能追加、Nablarch標準の追加、 Nablarch Toolbox へのツール追加などです。 | 半期に一度程度の頻度でリリースを予定してい ます。 |
| アーキテクチャデザインの変更(API変更を伴う) |  |  | Nablarchの全面改訂です。APIが変更される場合もあります。 | 特にサイクルはありません。 技術革新等、大きな外部環境の変化に追随する場 合、ソフトウェアやサービスの新規追加に伴って アーキテクチャデザインを見直す場合等 が想定されます。 |

各バージョンアップ時にお客様に行っていただく作業は、 [バージョンアップ時に必要なお客様のアプリケーションへの適用作業](../../about/about-nablarch/about-nablarch-versionup-policy.md#バージョンアップ時に必要なお客様のアプリケーションへの適用作業) をご参照ください。

### 後方互換性ポリシー

Nablarchの後方互換性ポリシーについて説明します。

#### 後方互換性を維持する範囲

Nablarchライブラリとして提供するコンテンツ(NAF、NTF)に対して、 [後方互換性の例外](../../about/about-nablarch/about-nablarch-versionup-policy.md#後方互換性の例外) に該当する場合を除き、
 [後方互換性維持の内容](../../about/about-nablarch/about-nablarch-versionup-policy.md#後方互換性維持の内容) に記載の内容で、後方互換性を維持いたします。

なお、Nablarchライブラリ以外のコンテンツに対する後方互換性の維持は考慮されません。

#### 後方互換性維持の内容

Nablarch では、下記のポリシーで後方互換性を維持するよう目指しています。

* Nablarchライブラリのバージョンアップを行った場合に、出来る限り下記を発生させないよう考慮します。

  * 既存のアプリケーションコードの修正。
  * 既存の自動テストコードの修正。
  * 既存の自動テストデータの修正。

この後方互換性維持の方針によりNablarchライブラリは、基本的にオブジェクトコード(jarファイル)の差し替えと設定ファイルの変更のみでバージョンアップできます。

この後方互換性維持の方針は、NAF/NTFのAPIのうち、Nablarchが定める公開APIを対象にしています。
Nablarchが定める公開APIとはjavadocにより仕様を公開しているAPIになります。
公開APIに対して、javadocにより仕様を公開していないAPIは、非公開APIになります。

非公開APIは、事前通知なく後方互換性が維持されないバージョンアップを行う場合がありますので、プロジェクトにて非公開APIを使用しないでください。

Nablarchでは、非公開APIの使用を検知するツールを提供しています。
プロジェクトにてこのツールを使用して非公開APIが使用されないように運用してください。
ツールの詳細は、下記を参照してください。

　プログラミング・単体テストガイド＞プログラミング工程で使用するツール＞Java静的解析ツール＞使用不許可APIチェックツール

#### 後方互換性の例外

下記内容に該当する場合は、事前通知なく後方互換性が維持されないバージョンアップを行う場合があります。

* Nablarchライブラリが出力するログのレベル、文言に対する変更。
* アーキテクチャデザインの変更を目的とするバージョンアップを行った場合。 [1]

下記内容に該当する場合は、事前通知の上で後方互換性が維持されないバージョンアップを行う場合があります。

* Nablarchの不具合が検出され、その対応が後方互換性を維持したまま実施できない場合。
* Nablarchを動作させる環境である、JDKのバージョンアップに起因する問題が発生し、その対応が後方互換性を維持したまま実施できない場合。

このバージョンアップが実施されるのは、メジャーバージョンアップ時のみです。
リビジョンアップ、マイナーバージョンアップの際は、アーキテクチャデザインの変更は実施されません。

### EOSLポリシー

Nablarchは、基幹システムのアプリケーション開発・実行基盤として安心してご利用いただくために、EOSL [2] の期間は設けません。
ご利用中のお客様がいるバージョンに対しては、基本サービスを提供いたします。

基本サービスの内容については、 [基本サービスの内容](../../about/about-nablarch/about-nablarch-support-service.md#基本サービスの内容) をご覧ください。

EOSL：End Of Service Lifeの略で、製品に対するサポートやサービスの提供を終了することです。

> **Note:**
> * >   新規に導入いただく場合は、原則として、契約時点での最新版をご導入いただきます。
> * >   不具合対応などのバージョンアップはお客様がご利用中のマイナーバージョンの最新のリビジョンに対してのみ実施されます。
> * >   マイナーバージョンアップでは、既にご利用中のお客様がいなくなったバージョンはお選びいただけません。

> バージョン番号、リビジョンについては、 [バージョン番号体系](../../about/about-nablarch/about-nablarch-versionup-policy.md#バージョン番号体系) をご参照ください。

### バージョン番号体系

Nablarchのバージョン番号は、メジャー番号、マイナー番号、リビジョン番号の3つの部分から成り立っています。Nablarchの
バージョン番号は、下記のようにこれら3つを"."でつないだ、 *x.y.z* で表されます。

```html
{メジャー番号}.{マイナー番号}.{リビジョン番号}
```

#### コンテンツとバージョン番号の関係

Nablarchのコンテンツにはさまざまな物が含まれます(詳細は [Nablarch の製品構成](../../about/about-nablarch/about-nablarch-contents.md#nablarch-の製品構成) をご参照ください)。これら一式に対して1つ
のバージョン番号が割り振られます。

#### バージョン番号と更新内容の対応

バージョン番号の各部分は、 [更新内容](../../about/about-nablarch/about-nablarch-versionup-policy.md#更新内容) と以下のような対応を持っています。また、各部分がインクリメントする
バージョンアップのことを、バージョンアップの名称欄に記載した名称で呼びます。各番号の詳細は、下表の後の記述をご参照ください。

| 番号 | 対応する対応内容 | バージョンアップの名称 | リリース頻度 |
|---|---|---|---|
| リビジョン番号 | NAF、NTFの不具合対応 | リビジョンアップ | 随時 |
| マイナー番号 | 機能追加またはNAF、NTF以外の不具合対応 | マイナーバージョンアップ | 半期に一度 |
| メジャー番号 | アーキテクチャデザインの更新 | メジャーバージョンアップ | 数年に一度 |

##### リビジョン番号

リビジョン番号がインクリメントするバージョンアップをリビジョンアップと呼びます。リビジョンアップは不具合対応を目的としています。
中でも、製造成果物の品質に影響を与える不具合への対応を行います。

リビジョンアップが行われても、マイナー番号、メジャー番号は変化しません。

##### マイナー番号

マイナー番号がインクリメントするバージョンアップをマイナーバージョンアップと呼びます。マイナーバージョンアップは機能追加と不具合対応を目的としています。
不具合対応については、製造成果物に影響を与えない不具合の対応を行います。また、前回のマイナーバージョンアップ以降に発生した、リビジョンアップの内
容も含みます。

マイナーバージョンアップが行われると、リビジョン番号は0にリセットされます。メジャー番号は変化しません。

##### メジャー番号

メジャー番号がインクリメントするバージョンアップをメジャーバージョンアップと呼びます。メジャーバージョンアップはNablarch自身のアーキテクチャデザインの更新
(API変更を伴います)を目的としています。

メジャーバージョンアップが行われると、リビジョン番号、メジャー番号は共に0にリセットされます。

### バージョンアップ時に必要なお客様のアプリケーションへの適用作業

Nablarchは、不具合修正や機能追加を実施する場合にバージョンアップを行います(詳細は [更新内容](../../about/about-nablarch/about-nablarch-versionup-policy.md#更新内容) をご参照ください)。
バージョンアップには、リビジョンアップ、マイナーバージョンアップ、メジャーバージョンアップの3種類があり(詳細は [バージョン番号体系](../../about/about-nablarch/about-nablarch-versionup-policy.md#バージョン番号体系) をご参照ください)、
お客様に実施していただく作業に以下のような違いがあります。

#### リビジョンアップ時に必要な作業

オブジェクトコード(jarファイル)の差し替えと設定ファイルの書き換えでバージョンアップとなります。

> **Note:**
> 原則として [3]  、お客様のアプリケーションへの影響はありません。

#### マイナーバージョンアップ時に必要な作業

オブジェクトコード(jarファイル)の差し替えと設定ファイルの書き換えでバージョンアップとなります。

> **Note:**
> 原則として [3] 、既存のお客様のアプリケーションオブジェクトコードへの影響はありません。

> また、新機能を使用する際には、設定ファイルの修正(追記)が必要になります。

#### メジャーバージョンアップ時に必要な作業

お客様のアプリケーションの修正、設定ファイルの修正、必要であればモデルの修正、オブジェクトコード(jarファイル)の差し替えでバージョンアップとなります。

> **Note:**
> 後方互換性は保障されないため、お客様のアプリケーションに修正が必要です(システム再構築と同等となる場合もあります)。

リビジョンアップとマイナーバージョンアップの場合も、例外的に不具合対応のため、やむを得ず後方互換性が保てない場合があります。この場合、テストコード、テストデータ、
お客様のアプリケーションの修正が必要になります(詳細は [後方互換性の例外](../../about/about-nablarch/about-nablarch-versionup-policy.md#後方互換性の例外) をご参照ください)。
