# 業務コンポーネントの責務配置

## 業務コンポーネントの責務配置

業務処理の複雑な責務をこれらのクラスに対してどのように配置するかについては様々な考え方があり、特定の方法に絞ることはできない。以下では、典型的なプロジェクトで採用することができる責務配置の1例を示す。

| 名称 (クラス接尾辞) | 責務 |
|---|---|
| 業務アクションハンドラ (Action) | フレームワークから直接コールバックされる業務処理のエントリーポイント。取引ごとに1つ作成する。単純な業務処理はActionに直接実装してよい。複雑な処理や複数業務・処理形態（バッチと画面オンラインなど）から共用される処理は、別途業務共通コンポーネントを作成してそこに実装する。 |
| 業務共通コンポーネント (Component) | 業務ロジックを実装するクラス。HTTPリクエストオブジェクトや実行コンテキストなどの実行制御基盤オブジェクトに直接依存してはならない。 |
| 業務フォーム (Form) | データ保持と外部入力値の精査を担う。単項目入力精査および項目間入力精査を実装する。DBへのアクセスが必要な精査処理はActionで実装する。 |
| 業務画面 (View) | 画面オンライン処理のユーザインタフェース（通常はJSP）。業務処理は実装せず、Actionから渡された処理結果を表示するのみ。 |

> **注意**: 業務アクションハンドラはHTTPリクエストオブジェクト等のフレームワーク依存オブジェクトに依存するため、自動テストはリクエスト単体テストで行う必要がある。複雑な内部条件を持つ業務ロジックをActionに実装するとテストデータのセットアップが煩雑になるため、その場合は業務処理部分を業務共通コンポーネントとして切り出してクラス単体テストを行うこと。

**処理フロー（画面オンライン処理）**:

1. View（Webクライアント）からリクエストが送られる
2. アプリケーションフレームワークがリクエストを受信し、Actionを呼び出す
3. ActionはバリデーションとForm生成を行い、Componentを呼び出す
4. Componentはビジネスロジックを実行し、結果をActionに返す
5. Actionは必要に応じてComponentの戻り値を処理（リクエストスコープへの値の格納など）し、フレームワークに返す
6. フレームワークはViewを処理（JSP→HTML変換など）し、クライアントにレスポンスを返す
