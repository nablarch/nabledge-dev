# 自動テストフレームワーク

## 自動テストフレームワークの構成

自動テストフレームワークの構成要素：

| 構成物 | 説明 | 作成者 |
|---|---|---|
| テストクラス | テスト処理を記述 | アプリケーションプログラマ |
| テスト対象クラス | テスト対象となるクラス | アプリケーションプログラマ |
| Excelファイル | テストデータを記載。フレームワークAPIでデータ読み取り可能 | アプリケーションプログラマ |
| コンポーネント設定ファイル・環境設定ファイル | テスト実行時の各種設定 | アプリケーションプログラマ（個別テストに固有設定が必要な場合） |
| 自動テストフレームワーク | テスト機能を提供 | — |
| Nablarch Application Framework | フレームワーク本体（本機能の対象外） | — |

![構成図](../../knowledge/development-tools/testing-framework/assets/testing-framework-01_Abstract/abstract_structure.png)

## テストメソッド記述方法

JUnit4のアノテーションを使用。テストメソッドに`@Test`アノテーションを付与。

```java
public class SampleTest {
    @Test
    public void testSomething() {
        // テスト処理
    }
}
```

> **補足**: `@Before`や`@After`も使用可能。テストメソッド前後のリソース取得解放などの共通処理については :ref:`using_junit_annotation` を参照。

## Excelによるテストデータ記述

データベース準備データや検索結果などをExcelファイルのスプレッドシート形式で扱える。

### 命名規約

Excelファイル名・パスの推奨規約に従うことで、テストクラスでディレクトリ名やファイル名の明示的指定不要（簡潔な記述が可能）。明示的パス指定で任意の場所のExcelファイルも読み込み可能。

## パス、ファイル名に関する規約

**ファイル名・パス規約**：

- Excelファイル名はテストソースコードと同じ名前（拡張子のみ異なる）
- Excelファイルをテストソースコードと同じディレクトリに配置

**配置例**：

| ファイルの種類 | 配置ディレクトリ | ファイル名 |
|---|---|---|
| テストソースファイル | <PROJECT_ROOT>/test/jp/co/tis/example/db/ | ExampleDbAcessTest.java |
| Excelファイル | 〃 | ExampleDbAcessTest.xlsx [1] |

[1] Excel2003以前（xls）およびExcel2007以降（xlsx）の形式に対応

## Excelシート名に関する規約

**規約**:
- １テストメソッドにつき１シート用意する
- シート名はテストメソッド名と同名にする

**例**:

| テストメソッド | Excelシート名 |
|---|---|
| @Test public void testInsert() | testInsert |

> **補足**: シート名とテストメソッド名が同名でなくても動作するが、規約に準拠することを推奨する（今後の機能追加はこの規約をデフォルトとして開発される）。プロジェクト内で統一すること。

## シート内の構造

![シート記述例](../../knowledge/development-tools/testing-framework/assets/testing-framework-01_Abstract/sheet_example.JPG)

**データタイプの概念**: シート内にはデータベースデータ、検索結果など様々なデータを記載可能。テストデータの種類を判別するため「データタイプ」を付与する。データタイプは、そのテストデータが何を表しているかを示す。

**データタイプ一覧**:

| データタイプ名 | 説明 | 設定する値 |
|---|---|---|
| SETUP_TABLE | テスト実行前にDBに登録するデータ | 登録対象のテーブル名 |
| EXPECTED_TABLE | テスト実行後の期待するDBデータ（省略カラムは比較対象外） | 確認対象のテーブル名 |
| EXPECTED_COMPLETE_TABLE | テスト実行後の期待するDBデータ（省略カラムには:ref:`default_values_when_column_omitted`が設定されているものとして扱う） | 確認対象のテーブル名 |
| LIST_MAP | List<Map<String,String>>形式のデータ | シート内で一意になるID（期待値のID） |
| SETUP_FIXED | 事前準備用の固定長ファイル | 準備ファイルの配置場所 |
| EXPECTED_FIXED | 期待値を示す固定長ファイル | 比較対象ファイルの配置場所 |
| SETUP_VARIABLE | 事前準備用の可変長ファイル | 準備ファイルの配置場所 |
| EXPECTED_VARIABLE | 期待値を示す可変長ファイル | 比較対象ファイルの配置場所 |
| MESSAGE | メッセージング処理のテストで使用するデータ | 固定値（setUpMessagesまたはexpectedMessages） |
| EXPECTED_REQUEST_HEADER_MESSAGES | 要求電文（ヘッダ）の期待値を示す固定長ファイル | リクエストID |
| EXPECTED_REQUEST_BODY_MESSAGES | 要求電文（本文）の期待値を示す固定長ファイル | リクエストID |
| RESPONSE_HEADER_MESSAGES | 応答電文（ヘッダ）を示す固定長ファイル | リクエストID |
| RESPONSE_BODY_MESSAGES | 応答電文（本文）を示す固定長ファイル | リクエストID |

**また、データの個数も複数記述できる。**

**書式**:
- 1行目: `データタイプ=値` の形式
- 2行目以降: データタイプにより異なる

**例** (SETUP_TABLE):
```
SETUP_TABLE=COMPOSER

NO | FIRST_NAME | LAST_NAME
00001 | Steve | Reich
00002 | Phillip | Glass
```

## コメント

セル内に`//`から開始する文字列を記載した場合、そのセルから右のセルは全て読み込み対象外となる。可読性向上のために付加情報を記載する場合に使用可能。

**例**:
```
EXPECTED_TABLE=PLAYER

NO | FIRST_NAME | LAST_NAME | ADDRESS |
//番号 | 名 | 姓 | 住所 |
0001 | Andres | Segovia | Spain |
0002 | Julian | Bream | England | // このレコードが追加される
```

## マーカーカラム

テストデータの見出し行において、カラム名が半角角括弧で囲まれている場合、そのカラムは「マーカーカラム」とみなされる。マーカーカラムに該当する列はテスト実行時には読み込まれない。

**用途**: 実際のデータには含めたくないがExcelシート上には記述しておきたい情報を記載する場合。前述の「コメント」を使用することにより、実際のデータには含まれない情報を記述できるが、「コメント」には、そのセルから右のセルを読み込み対象外にするという性質があるため、左端（または中央）のセルにはコメントを使用できない。このような場合にマーカーカラムを使用する。

**例**:
```
LIST_MAP=EXAMPLE_MARKER_COLUMN

[no] | id | name
1 | U0001 | 山田
2 | U0002 | 田中
```

上記は、`[no]`が無視されるため、以下と等価:
```
id | name
U0001 | 山田
U0002 | 田中
```

**適用範囲**: 全データタイプで使用可能。

## セルの書式

**必須設定**: セルの書式には文字列のみを使用する。テストデータ作成前に、全てのセルの書式を文字列に設定すること。

**任意設定**: 罫線やセルの色付けは任意。データの見やすさ向上により、レビュー品質や保守性の向上が期待できる。

> **重要**: 文字列以外の書式でデータを記述した場合、正しくデータが読み取れなくなる。

## 日付の記述方法

**対応形式**:
- `yyyyMMddHHmmssSSS`
- `yyyy-MM-dd HH:mm:ss.SSS`

**省略ルール**:

| 省略方法 | 動作 |
|---|---|
| ミリ秒を省略（yyyyMMddHHmmss / yyyy-MM-dd HH:mm:ss） | ミリ秒として0を指定したものとして扱われる |
| 時刻全部を省略（yyyyMMdd / yyyy-MM-dd） | 時刻として0時0分0秒000を指定したものとして扱われる |

**例**:

| 記述例 | 評価結果 |
|---|---|
| 20210123123456789 | 2021年1月23日 12時34分56秒789 |
| 20210123123456 | 2021年1月23日 12時34分56秒000 |
| 20210123 | 2021年1月23日 00時00分00秒000 |
| 2021-01-23 12:34:56.789 | 2021年1月23日 12時34分56秒789 |
| 2021-01-23 12:34:56 | 2021年1月23日 12時34分56秒000 |
| 2021-01-23 | 2021年1月23日 00時00分00秒000 |

## セルへの特殊な記述方法

| 記述方法 | 自動テスト内での値 | 説明 |
|---|---|---|
| null / Null | null | セル内に「null」（半角、大文字小文字区別なし）と記述した場合、null値として扱う。DBにnull値を登録したい場合や、期待値でnull値を設定したい場合に使用。 |
| "null" / "NULL" / "1 " / " " / "１　" / "　　" | 文字列のnull / NULL / 1  /   / １　 / 　　 | 前後がダブルクォート（半角、全角問わず）で囲われている場合、前後のダブルクォートを取り除いた文字列を扱う。「null」や「NULL」を文字列として扱う場合や、セルの値にスペースがあることを解りやすくする場合に使用。文字列中のダブルクォートはエスケープ不要。 |
| """ | " | ダブルクォート文字そのもの |
| "" | 空文字列 | 空文字列（空行を表す場合に使用） |
| ${systemTime} / ${updateTime} | システム日時 | コンポーネント設定ファイルで設定されたSystemTimeProvider実装クラスから取得したTimestampの文字列形式（例: `2011-04-11 01:23:45.0`）。${updateTime}はDB更新時のタイムスタンプ期待値として特に使用。 |
| ${setUpTime} | コンポーネント設定ファイルに記載された固定値 | データベースセットアップ時のタイムスタンプに決まった値を使用したい場合に使用。 |
| ${文字種,文字数} | 指定した文字種を指定した文字数分まで増幅した値 | 使用可能な文字種: 半角英字、半角数字、半角記号、半角カナ、全角英字、全角数字、全角ひらがな、全角カタカナ、全角漢字、全角記号その他、外字。単独でも組み合わせでも使用可能（例: `${半角英字,5}` → `geDSfe`、`${半角数字,2}-{半角数字4}` → `37-3425`） |
| ${binaryFile:ファイルパス} | BLOB列に格納するバイナリデータ | BLOB列にファイルのデータを格納したい場合に使用。ファイルパスはExcelファイルからの相対パスで記述。 |
| \\r | CR | 改行コードCR(0x0D)を明示的に記述する場合に使用。 |
| \\n | LF | 改行コードLF(0x0A)を明示的に記述する場合に使用。Excelセル内の改行（Alt+Enter）はLFとして扱われる（Excelの仕様）。 |

**ダブルクォート記法の例**:

| 記述例 | 扱われる値 | 説明 |
|---|---|---|
| "ab"c" | ab"c | 前後のダブルクォートが除去される |
| "abc"" | abc" | 前後のダブルクォートが除去される |
| ab"c | ab"c | 前後がダブルクォートではないため、そのまま扱われる |
| abc" | abc" | 前後がダブルクォートではないため、そのまま扱われる |

**改行コード記法の例**:

| 記述例 | 変換される値 | 説明 |
|---|---|---|
| こんにちは（改行）さようなら | こんにちはLFさようなら | セル内の改行（Alt+Enter）はLF(0x0A)となる |
| こんにちは\\nさようなら | こんにちはLFさようなら | \\nは本機能によりLF(0x0A)に変換される |
| こんにちは\\r（改行）さようなら | こんにちはCRLFさようなら | \\rはCR(0x0D)に変換され、セル内改行はLF(0x0A)となる |

## 注意事項

### テストメソッドの実行順序に依存しないテストを作成する

テストメソッドの実行順序によってテスト結果が変わらないようにすること。クラス単体でテストしても、複数まとめてテストしても同じ結果でなければならない。

本フレームワークではテスト中にコミットが行われるため、前後のテストによってDBの内容が変更される可能性が高い。自テストクラスで必要な事前条件は、全て自テストクラス内で準備すること。

マスタデータのような読み取り専用テーブルの準備は、共通のExcelファイルを用意してそこに記載すること。テスト実行前に1回だけ実行するか、事前準備済みという前提でテストを実行する。

> **補足**: マスタデータの投入には、:ref:`master_data_setup_tool`を使用する。:doc:`04_MasterDataRestore`により、テスト内で発生したマスタデータの変更をテスト終了時に自動的に元の状態に戻すことができる。これにより、マスタデータに変更が必要なテストケースであっても、他のテストケースに影響無く実行できる。

### テストデータは全てExcelシートに記述する

テストソースコード中にテストデータを記載せず、テストデータは全てExcelシートに記載すること。混在すると可読性、保守性が低下する。

### 複数のデータタイプ使用時はデータタイプごとにまとめてデータを記述する

複数のデータタイプを使用する場合、データタイプごとにまとめてデータを記述すること。混在させると、データの読み込みが途中で終了しテストが正しく実行されない。

例えば、以下のようにデータタイプを記述した場合、`TABLE2`までのデータしか評価されず、`TABLE3`以降のデータに誤りがあってもテストは成功してしまう：

```text
EXPECTED_TABLE=TABLE1
:
EXPECTED_COMPLETE_TABLE=TABLE2
:
EXPECTED_TABLE=TABLE3
:
EXPECTED_COMPLETE_TABLE=TABLE4
:
```

正しくは、データタイプごとにまとめて記述する：

```text
EXPECTED_TABLE=TABLE1
:
EXPECTED_TABLE=TABLE3
:
EXPECTED_COMPLETE_TABLE=TABLE2
:
EXPECTED_COMPLETE_TABLE=TABLE4
:
```

## JUnit 5で自動テストフレームワークを動かす

### JUnit Vintage

JUnit VintageはJUnit 5上でJUnit 4のテストを実行する機能を提供する。これにより自動テストフレームワークをJUnit 5上で動かすことができる。

> **重要**: この機能は、あくまでJUnit 4のテストをJUnit 4として動かしているにすぎない。JUnit 4のテストの中でJUnit 5の機能が使えるわけではない。JUnit 4からJUnit 5への移行を段階的進めるための補助として利用できる。JUnit 4からJUnit 5に移行するときの手順については、[公式のガイド(外部サイト、英語)](https://junit.org/junit5/docs/5.11.0/user-guide/#migrating-from-junit4)を参照。

> **補足**: JUnit 5のテストで自動テストフレームワークを使用する方法については、:ref:`ntf_junit5_extension`を参照。

### 前提条件

maven-surefire-pluginが2.22.0以上であること。

### 依存関係の追加

JUnit Vintageは、pom.xmlで以下2つのアーティファクトを依存関係に追加することで有効にできる：

- `org.junit.jupiter:junit-jupiter`
- `org.junit.vintage:junit-vintage-engine`

```xml
<dependencyManagement>
  <dependencies>
    ...

    <!-- バージョンを揃えるため、JUnitが提供しているbomを読み込む -->
    <dependency>
      <groupId>org.junit</groupId>
      <artifactId>junit-bom</artifactId>
      <version>5.8.2</version>
      <type>pom</type>
      <scope>import</scope>
    </dependency>
  </dependencies>
</dependencyManagement>

<dependencies>
  ...

  <!-- 以下の依存関係を追加する -->
  <dependency>
    <groupId>org.junit.jupiter</groupId>
    <artifactId>junit-jupiter</artifactId>
    <scope>test</scope>
  </dependency>
  <dependency>
    <groupId>org.junit.vintage</groupId>
    <artifactId>junit-vintage-engine</artifactId>
    <scope>test</scope>
  </dependency>
</dependencies>
```
