# バリデーションとFormの生成

## 概要

## 概要

Formは以下2つの責務を持つ業務コンポーネント:
1. 外部入力値の入力チェック
2. 外部入力値の保持

バリデーションは3レベルに分類される:
1. 単項目の入力チェック
2. 複数項目にまたがる入力チェック
3. データベースのデータを使用した入力チェック

本機能は1と2をサポートする。3のDB使用チェックはアプリケーションプログラマが別途実装する。

**前提条件**: 入力値はMap形式、FormはセッタとゲッタをもつJavaクラスであること。バリデーション条件はFormにアノテーションとして設定する。

本機能はAction側で業務ロジック呼び出し前の入力チェックに使用する。実装方法は [validation_and_convert](libraries-08_02_validation_usage.md) を参照。

FormをRDBMSテーブルと1対1対応させ（プロパティとカラムも1対1対応）た実装を **Entity** と呼称する。Entityにより:
- バリデーションとDBへの反映を容易に実装できる
- RDBMSの定義から実装の多くの部分を自動生成できる

## 特徴

## 特徴

**アノテーションによる条件の記述**: バリデーションはFormに設定したアノテーションをもとに実施されるため、煩雑な単項目の入力チェック処理の実装が不要となり、バリデーションの実装を極小化できる。ハードコーディングによるバリデーションを行う場合と比較して生産性を向上できる。また、バリデーション条件をFormに保持するため、画面処理・バッチ処理等の異なる入力元間でバリデーション条件を共有できる。

**メッセージ機能の使用**: バリデーション結果のメッセージ作成に [./07_Message](libraries-07_Message.md) を使用。バリデーションエラーとビジネスロジックが発生させるメッセージを透過的に扱えるため、エラーメッセージ表示処理で両者を同一に取り扱える。

## 要求

## 要求

### 実装済み

- 文字列の特定文字種チェック
- 文字列長チェック
- 必須入力値チェック
- 数値の規定範囲チェック
- 入力値文字列のMapからFormインスタンス生成
- 型変換可否チェック（Formプロパティの型に合わせた変換チェック）
- 入力値文字列をFormプロパティの型に変換
- プロパティ設定前にバリデーション実行
- エラーメッセージのプロパティ表示名指定
- エラーメッセージのプロパティ表示名の国際化
- Form内部のプロパティFormに対するバリデーション
- 日付形式チェック: 西暦(年月日)フォーマット（詳細は :ref:`ExtendedValidation_yyyymmddConvertor` を参照）
- 入力値のトリム
- コード値の有効チェック

### 未実装

- 日付形式チェック: 和暦フォーマット、年月フォーマット、営業日
- 入力禁止文字チェック
- 制御文字チェック
- パスワード強度判定
- 氏名フォーマットチェック
- アノテーションによるトリム要否設定
- 日本語電話番号フォーマットチェック
- メールアドレスフォーマットチェック

### 未検討

- 正規表現マッチチェック
- 複数アノテーションをまとめたアノテーション定義
- カード番号の有効チェック
- 郵便番号の有効チェック
- 金額フォーマットチェック
- 文字列範囲チェック
- トリム後バリデーション
- 外部入力値とDB値をマージしたFormのバリデーション
