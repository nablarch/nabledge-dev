# 自動テストフレームワーク

## 

自動テストフレームワークの特徴:

- **JUnit4ベース**: 各種アノテーション、assertメソッド、MatcherクラスなどJUnit4で提供されている機能を使用する。
- **テストデータの外部化**: データベース準備データや期待するテスト結果などをExcelファイルに記述し、自動テストフレームワークのAPIを通じて読み込める。
- **Nablarch特化API**: トランザクション制御やシステム日付設定など、Nablarchアプリケーション固有のAPIを提供する。

## Excelシート名に関する規約

- 1テストメソッドにつき1シートを用意する（推奨）
- シート名はテストメソッド名と同名にする（推奨）

| テストメソッド | Excelシート名 |
|---|---|
| `@Test public void testInsert()` | `testInsert` |

> **注意**: シートに関する規約は「制約」ではない。テストメソッド名とExcelシート名が同名でなくても正しく動作する。今後の機能追加は上記規約をデフォルトとして開発されるため、命名規約への準拠を推奨する。プロジェクト内で統一すること。

## 自動テストフレームワークの構成

自動テストフレームワークの構成要素:

| 構成物 | 説明 | 作成者 |
|---|---|---|
| テストクラス | テスト処理を記述する | アプリケーションプログラマ |
| テスト対象クラス | テスト対象となるクラス | アプリケーションプログラマ |
| Excelファイル | テストデータを記載する。自動テストフレームワークのAPIを通じて読み込める | アプリケーションプログラマ |
| コンポーネント設定ファイル・環境設定ファイル | テスト実行時の各種設定を記載する | アプリケーションプログラマ（個別のテストに固有の設定が必要な場合） |
| 自動テストフレームワーク | テストに必要な機能を提供する | — |
| Nablarch Application Framework | フレームワーク本体（本機能の対象外） | — |

## シート内の構造

シート内には、データベースに格納するデータやデータベース検索結果など、さまざまな種類のデータを記載できる。テストデータの種類を判別するために「データタイプ」というメタ情報をテストデータに付与する必要がある。

データの種類に依らない共通の書式は以下の通り。

- データ1行目は「データタイプ=値」の形式で、データタイプと値を記載する。
- 2行目以降の書式はデータタイプにより異なる。

### データタイプ一覧

| データタイプ名 | 説明 | 設定する値 |
|---|---|---|
| `SETUP_TABLE` | テスト実行前にDBに登録するデータ | 登録対象のテーブル名 |
| `EXPECTED_TABLE` | テスト実行後の期待するDBデータ。省略したカラムは比較対象外 | 確認対象のテーブル名 |
| `EXPECTED_COMPLETE_TABLE` | テスト実行後の期待するDBデータ。省略したカラムには`default_values_when_column_omitted`が設定されているものとして扱われる | 確認対象のテーブル名 |
| `LIST_MAP` | `List<Map<String,String>>`形式のデータ | シート内で一意になるID（期待値のID） |
| `SETUP_FIXED` | 事前準備用の固定長ファイル | 準備ファイルの配置場所 |
| `EXPECTED_FIXED` | 期待値を示す固定長ファイル | 比較対象ファイルの配置場所 |
| `SETUP_VARIABLE` | 事前準備用の可変長ファイル | 準備ファイルの配置場所 |
| `EXPECTED_VARIABLE` | 期待値を示す可変長ファイル | 比較対象ファイルの配置場所 |
| `MESSAGE` | メッセージング処理テスト用データ | 固定値（`setUpMessages`または`expectedMessages`） |
| `EXPECTED_REQUEST_HEADER_MESSAGES` | 要求電文（ヘッダ）期待値の固定長ファイル | リクエストID |
| `EXPECTED_REQUEST_BODY_MESSAGES` | 要求電文（本文）期待値の固定長ファイル | リクエストID |
| `RESPONSE_HEADER_MESSAGES` | 応答電文（ヘッダ）の固定長ファイル | リクエストID |
| `RESPONSE_BODY_MESSAGES` | 応答電文（本文）の固定長ファイル | リクエストID |

## テストメソッド記述方法

テストメソッドの記述にはJUnit4のアノテーションを使用する。テストメソッドに`@Test`アノテーションを付与する。

## コメント

セル内に`//`から開始する文字列を記載した場合、そのセルから右のセルは全て読み込み対象外となる。テストデータ自体には含めたくないが、可読性を向上させるために付加情報を記載したい場合に使用できる。

例: `EXPECTED_TABLE=PLAYER` のシートで、2行目に`//番号`と記載した場合、その行全体が読み込み対象外となる。また行末のセルに`// このレコードが追加される`と記載した場合、そのセルから右が読み込み対象外となる。

```java
public class SampleTest {

    @Test
    public void testSomething() {
        // テスト処理
    }
}
```

> **注意**: `@Before`や`@After`などのアノテーションも使用できる。テストメソッド前後にリソースの取得・解放などの共通処理を行いたい場合は、[using_junit_annotation](testing-framework-03_Tips.md) を参照。

## マーカーカラム

テストデータを記述する際、実際のデータには含めたくないがExcelシート上には記述しておきたい場合に使用する。コメントは「そのセルから右を全て読み込み対象外にする」性質があるため、左端または中央のセルにはコメントを使用できない。そのような場合に「マーカーカラム」を使用する。

**見出し行のカラム名が半角角括弧で囲まれている場合**（例: `[no]`）、そのカラムは「マーカーカラム」とみなされ、テスト実行時には読み込まれない。全データタイプで使用可能。

例: `LIST_MAP=EXAMPLE_MARKER_COLUMN` のシートで `[no]` カラムを定義すると、テスト実行時には `[no]` カラムが無視され、`id` と `name` カラムのみが有効となる。

## Excelによるテストデータ記述

データベースの準備データやデータベース検索結果などのデータを表すには、Javaソースコードよりスプレッドシートのほうが可読性・編集しやすさで有利。Excelファイルを使用することにより、このようなデータをスプレッドシート形式で扱える。

## セルの書式

セルの書式には、文字列のみを使用する。テストデータを作成する前に、全てのセルの書式を文字列に設定しておくこと。

> **警告**: Excelファイルに文字列以外の書式でデータを記述した場合、正しくデータが読み取れなくなる。

罫線やセルの色付けについては任意に設定可能。罫線やセルの色付けを行うことでデータが見やすくなり、レビュー品質や保守性の向上が期待できる。

## 命名規約

Excelファイル名・ファイルパスには推奨される命名規約がある。この規約に従うことで、テストクラスでディレクトリ名やファイル名を明示的に指定せずに簡潔にテストソースコードを記述できる。また明示的にパスを指定することで任意の場所のExcelファイルを読み込むことも可能。

## セルへの特殊な記述方法

自動テストの利便性を向上させるために、以下の特殊記法を提供する。

| 記述方法 | 自動テスト内での値 | 説明 |
|---|---|---|
| `null` / `Null` | null | 大文字小文字問わずnull値として扱われる。DBにnull値を登録したい場合や期待値にnull値を設定したい場合に使用する |
| `"null"` / `"NULL"` | 文字列のnull | ダブルクォートで囲まれた場合、前後のダブルクォートを除去した文字列として扱われる。文字列中のダブルクォートはエスケープ不要 |
| `"1⊔"` / `"⊔"` / `"１△"` / `"△△"` | スペースを含む値 | ⊔=半角スペース、△=全角スペース。スペースがあることを明示したい場合に使用 |
| `${systemTime}` | システム日時 | `SystemTimeProvider`実装クラスから取得したTimestampの文字列形式（例: `2011-04-11 01:23:45.0`） |
| `${updateTime}` | システム日時 | `${systemTime}`の別名。特にデータベースのタイムスタンプ更新時の期待値として使用する |
| `${setUpTime}` | コンポーネント設定ファイルに記載された固定値 | DBセットアップ時のタイムスタンプに固定値を使用する場合に使用 |
| `${文字種,文字数}` | 指定文字種を指定文字数分増幅した値 | 使用可能文字種: 半角英字、半角数字、半角記号、半角カナ、全角英字、全角数字、全角ひらがな、全角カタカナ、全角漢字、全角記号その他、外字。単独・組み合わせ使用可能（例: `${半角英字,5}` → `geDSfe`、`${全角ひらがな,4}` → `ぱさぇん`） |
| `${binaryFile:ファイルパス}` | BLOB列に格納するバイナリデータ | BLOB列にファイルのデータを格納したい場合に使用する。ファイルパスはExcelファイルからの相対パス |

## パス、ファイル名に関する規約

推奨される規約:

- Excelファイル名は、テストソースコードと同じ名前にする（拡張子のみ異なる）
- Excelファイルを、テストソースコードと同じディレクトリに配置する

| ファイルの種類 | 配置ディレクトリ | ファイル名 |
|---|---|---|
| テストソースファイル | `<PROJECT_ROOT>/test/jp/co/tis/example/db/` | ExampleDbAcessTest.java |
| Excelファイル | `<PROJECT_ROOT>/test/jp/co/tis/example/db/` | ExampleDbAcessTest.xls |

ExcelファイルはExcel2003以前のファイル形式（拡張子`xls`）およびExcel2007以降のファイル形式（拡張子`xlsx`）に対応。

## 注意事項

### テストメソッドの実行順序に依存しないテストを作成する

- テストメソッドの実行順序によってテスト結果が変わらないようにすること
- クラス単体でも複数まとめてテストしても同じ結果になること
- フレームワークはテスト中にコミットを行うため、前後のテストでDBが変更される可能性が高い。自テストクラスで必要な事前条件は全て自テストクラス内で準備すること
- マスタデータのような読み取り専用テーブルの準備は共通Excelファイルに記載し、テスト実行前に1回実行するか事前準備済みの前提でテストを実行すること
- マスタデータの投入には`master_data_setup_tool`を使用する
- `04_MasterDataRestore`によりテスト内で発生したマスタデータの変更をテスト終了時に自動的に元の状態に戻すことができる

### テストデータは全てExcelシートに記述する

テストソースコード中にはテストデータを記載せず、テストデータは全てExcelシートに記述すること。

- Excelシートを見れば、テストケースのバリエーションを把握することができる
- テストデータはExcelシート、テストロジックはテストソースコードと役割分担が明確になる
- Excelシートを読み込む構造にしておくことで、容易にテストケースを追加できる
- テストソースコードの重複を大幅に削減できる（テストソースコード中に単純にリテラル値でデータを記載すると、データのバリエーションが増加すると重複したコードが作られてしまう）
