## 業務コンポーネントの責務配置

業務機能は、業務アクションハンドラ・フォーム/エンティティ・業務共通コンポーネントの各クラスに実装される。

なお、本書に記載される業務アプリケーションの実装例は、ここで述べる責務配置を前提として記述されている。

| 名称 (クラス接尾辞) | 責務 |
|---|---|
| 業務アクションハンドラ (Action) | フレームワークから直接コールバックされる業務処理のエントリーポイント。取引ごとに1つ作成する。単純な業務処理はActionに直接実装してよい。複雑な処理や複数業務・処理形態（バッチと画面オンラインなど）から共用される処理は業務共通コンポーネントに実装する。 |
| 業務共通コンポーネント (Component) | 業務ロジックを実装するクラス。HTTPリクエストオブジェクトや実行コンテキストなどの実行制御基盤オブジェクトに直接依存しないこと。 |
| 業務フォーム (Form) | データの保持と外部入力値の精査を実行。単項目・項目間入力精査を実装する。DBアクセスが必要な精査処理は業務アクションハンドラで実装する。 |
| 業務画面 (View) | 画面オンライン処理のユーザインタフェース（通常はJSPを使用する）。業務処理は実装せず、Actionから渡される処理結果を表示する。 |

> **注意**: 業務アクションハンドラはHTTPリクエストオブジェクトや実行コンテキストに依存するため、自動テストはリクエスト単体テストで行う必要がある。複雑な内部条件をもつ業務ロジックを実装した場合、テストデータのセットアップ作業が煩雑となり、作業効率が低下する。このような場合は、業務処理部分を業務共通コンポーネントとして切り出した上でクラス単体テストを行うこと。

### 処理の流れ（画面オンライン処理）

![画面オンライン処理フロー図](../../../knowledge/about/about-nablarch/assets/about-nablarch-determining_stereotypes/fw_app_flow.png)

1. View（Webクライアント）からリクエストが送られる
2. フレームワークがリクエストを受信し、Actionを呼び出す
3. ActionはバリデーションしてForm（画面入力値格納オブジェクト）を生成し、Componentを呼び出す
4. Componentはビジネスロジックを実行し、結果をActionに返す
5. ActionはComponentの戻り値を処理（リクエストスコープへの値の格納など）し、フレームワークに返す
6. フレームワークはViewを処理（JSPをHTMLに変換など）し、クライアントにレスポンスを返す
