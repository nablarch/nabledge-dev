# 自動テストフレームワーク

## 特徴

**JUnit4ベース**: JUnit4のアノテーション（`@Test`等）、assertメソッド、Matcherクラスなど、JUnit4の機能を使用する。

**テストデータの外部化**: DBの準備データや期待するテスト結果などをExcelファイルに記述し、フレームワークのAPIを通じて使用できる。

**Nablarchに特化したテスト補助機能**: トランザクション制御、システム日付設定など、Nablarchアプリケーションに特化したAPIを提供する。

Excelシートの命名規約（推奨）:

- 1テストメソッドにつき1シートを用意する
- シート名はテストメソッド名と同名にする

例: `@Test public void testInsert()` → Excelシート名: `testInsert`

> **注意**: この規約は「制約」ではない。テストメソッド名とExcelシート名が同名でなくても正しく動作する。今後の機能追加はこの規約をデフォルトとして開発されるため、命名規約への準拠を推奨する。規約を変更する場合もプロジェクト内で統一すること。

## 自動テストフレームワークの構成

![自動テストフレームワークの構成図](../../../knowledge/development-tools/testing-framework/assets/testing-framework-01_Abstract/abstract_structure.png)

シート内にはデータベースデータやファイルなどさまざまな種類のデータを記載できる。テストデータの種類を判別するために「データタイプ」を付与する。データ1行目は「データタイプ=値」の形式で記載する。

| データタイプ名 | 説明 | 設定する値 |
|---|---|---|
| SETUP_TABLE | テスト実行前にDBに登録するデータ | テーブル名 |
| EXPECTED_TABLE | テスト実行後の期待DBデータ。省略カラムは比較対象外 | テーブル名 |
| EXPECTED_COMPLETE_TABLE | テスト実行後の期待DBデータ。省略カラムは`default_values_when_column_omitted`が設定されているものとして扱う | テーブル名 |
| LIST_MAP | List\<Map\<String,String\>\>形式のデータ | シート内で一意なID / 期待値のID（任意の文字列） |
| SETUP_FIXED | 事前準備用の固定長ファイル | 準備ファイルの配置場所 |
| EXPECTED_FIXED | 期待値を示す固定長ファイル | 比較対象ファイルの配置場所 |
| SETUP_VARIABLE | 事前準備用の可変長ファイル | 準備ファイルの配置場所 |
| EXPECTED_VARIABLE | 期待値を示す可変長ファイル | 比較対象ファイルの配置場所 |
| MESSAGE | メッセージング処理のテストデータ（`setUpMessages`または`expectedMessages`） | 固定値 |
| EXPECTED_REQUEST_HEADER_MESSAGES | 要求電文（ヘッダ）の期待値（固定長） | リクエストID |
| EXPECTED_REQUEST_BODY_MESSAGES | 要求電文（本文）の期待値（固定長） | リクエストID |
| RESPONSE_HEADER_MESSAGES | 応答電文（ヘッダ）（固定長） | リクエストID |
| RESPONSE_BODY_MESSAGES | 応答電文（本文）（固定長） | リクエストID |

## 自動テストフレームワーク構成要素

| 構成物 | 説明 | 作成者 |
|---|---|---|
| テストクラス | テスト処理を記述する | アプリケーションプログラマ |
| テスト対象クラス | テスト対象となるクラス | アプリケーションプログラマ |
| Excelファイル | テストデータを記載する。フレームワークのAPIでデータを読み取れる | アプリケーションプログラマ |
| コンポーネント設定ファイル・環境設定ファイル | テスト実行時の各種設定を記載する | アプリケーションプログラマ（個別のテストに固有の設定が必要な場合） |
| 自動テストフレームワーク | テストに必要な機能を提供する | － |
| Nablarch Application Framework | フレームワーク本体（本機能の対象外） | － |

セル内に`//`から開始する文字列を記載した場合、そのセルから右のセルは全て読み込み対象外となる。テストデータ自体には含めたくないが可読性向上のために付加情報を記載したい場合に使用する。

例: 見出し行に論理名をコメントとして記載したり、特定レコードの末尾に注釈を入れたりできる。

## テストメソッド記述方法

JUnit4の`@Test`アノテーションをテストメソッドに付与する。

```java
public class SampleTest {

    @Test
    public void testSomething() {
        // テスト処理
    }
}
```

見出し行のカラム名が半角角括弧で囲まれている場合（例: `[no]`）、そのカラムは「マーカーカラム」とみなされ、テスト実行時に読み込まれない。全データタイプで使用可能。

「コメント」はそのセルから右を全て対象外にするため左端・中央のセルには使用できないが、マーカーカラムであれば任意の位置のカラムを実際のデータに含めずExcelシート上には記述できる。

## JUnit4アノテーション

> **注意**: `@Before`、`@After`などのJUnit4アノテーションも使用できる。テストメソッドの前後にリソースの取得・解放などの共通処理を記述できる（[using_junit_annotation](testing-framework-03_Tips.md)）。

セルの書式は文字列のみを使用すること。テストデータ作成前に全セルの書式を文字列に設定すること。

罫線やセルの色付けは任意に設定可能で、データの可読性・レビュー品質・保守性の向上が期待できる。

> **警告**: Excelファイルに文字列以外の書式でデータを記述した場合、正しくデータが読み取れなくなる。

## Excelによるテストデータ記述

DB準備データや検索結果などのテストデータはJavaソースよりExcelスプレッドシートの方が可読性・編集性で有利。Excelファイルを使用してスプレッドシート形式でデータを扱える。

Excelファイル名とファイルパスには推奨される命名規約がある。この規約に従うと、テストクラスでディレクトリ名・ファイル名を明示指定せずにファイルを読み込める。明示指定により任意の場所のExcelファイルを読み込むことも可能。

自動テストの利便性向上のためにいくつかの特殊記法を提供する。

| 記述方法 | 値 | 説明 |
|---|---|---|
| `null` / `Null`（半角、大小文字区別なし） | null | null値として扱う。DBにnull値を登録したい場合や期待値にnull値を設定したい場合に使用 |
| `"null"` / `"NULL"` | 文字列のnull | ダブルクォートで囲むと前後が除去された文字列として扱われる |
| `"1⊔"` | `1⊔` | スペースがあることを明示するためのダブルクォート記法（⊔は半角スペース） |
| `"⊔"` | `⊔` | 半角スペース1文字をダブルクォート記法で明示 |
| `"１△"` | `１△` | 全角スペース付き文字列をダブルクォート記法で明示（△は全角スペース） |
| `"△△"` | `△△` | 全角スペース2文字をダブルクォート記法で明示 |
| `""` | 空文字列 | 空行を表す場合に使用 |
| `${systemTime}` | システム日時（例: `2011-04-11 01:23:45.0`） | SystemTimeProvider実装クラスから取得したTimestamp |
| `${updateTime}` | システム日時 | `${systemTime}`の別名。特にDBのタイムスタンプ更新時の期待値として使用する |
| `${setUpTime}` | コンポーネント設定ファイルに記載された固定値 | DBセットアップ時のタイムスタンプに固定値を使用したい場合 |
| `${文字種,文字数}` | 指定文字種を指定文字数分増幅した値 | 使用可能: 半角英字、半角数字、半角記号、半角カナ、全角英字、全角数字、全角ひらがな、全角カタカナ、全角漢字、全角記号その他、外字。単独および組み合わせ使用可能 |
| `${binaryFile:ファイルパス}` | BLOB列に格納するバイナリデータ | ファイルパスはExcelファイルからの相対パス |
| `\r` | CR (0x0D) | 改行コードCRを明示的に記述する場合 |
| `\n` | LF (0x0A) | 改行コードLFを明示的に記述する場合。セル内のAlt+EnterもLFとして扱われる |

> **凡例**: ⊔は半角スペース、△は全角スペース、CRは改行コードCR(0x0D)、LFは改行コードLF(0x0A)を意味する。

### ダブルクォート記法の注意事項

ダブルクォートで囲む記法を使用した場合、**文字列中のダブルクォートをエスケープする必要はない**。前後のダブルクォートのみが除去される。

| 記述例 | 値 | 説明 |
|---|---|---|
| `"ab"c"` | `ab"c` | 前後のダブルクォートが除去される |
| `"abc""` | `abc"` | 前後のダブルクォートが除去される |
| `ab"c` | `ab"c` | 前後がダブルクォートでないためそのまま扱われる |
| `abc"` | `abc"` | 前後がダブルクォートでないためそのまま扱われる |

## パス、ファイル名に関する規約

推奨される規約:
- Excelファイル名はテストソースコードと同じ名前にする（拡張子のみ異なる）
- Excelファイルをテストソースコードと同じディレクトリに配置する

| ファイルの種類 | 配置ディレクトリ | ファイル名 |
|---|---|---|
| テストソースファイル | `<PROJECT_ROOT>/test/jp/co/tis/example/db/` | `ExampleDbAcessTest.java` |
| Excelファイル | `<PROJECT_ROOT>/test/jp/co/tis/example/db/` | `ExampleDbAcessTest.xls` |

ExcelファイルはExcel2003以前（拡張子 `.xls`）およびExcel2007以降（拡張子 `.xlsx`）のファイル形式に対応している。

## テストメソッドの実行順序に依存しないテストを作成する

テストメソッドの実行順序によってテスト結果が変わらないようにすること。クラス単体でも複数まとめてテストしても同じ結果になること。

- 本フレームワークではテスト中にコミットが行われるため、前後のテストによってDBの内容が変更される可能性が高い
- 自テストクラスで必要な事前条件は全て自テストクラス内で準備すること
- マスタデータ（読み取り専用テーブル）の準備は共通Excelファイルに記載し、テスト実行前に1回だけ実行するか、事前準備済みを前提にすること（`master_data_setup_tool`を使用）

> **注意**: `04_MasterDataRestore`により、テスト内で発生したマスタデータの変更をテスト終了時に自動的に元の状態に戻すことができる。これにより、マスタデータに変更が必要なテストケースであっても、他のテストケースに影響無く実行できる。

## テストデータは全てExcelシートに記述する

テストソースコード中にはテストデータを記載せず、テストデータは全てExcelシートに記述すること。

- Excelシートを見ればテストケースのバリエーションを把握できる
- テストデータはExcelシート、テストロジックはテストソースコードと役割分担が明確になる
- テストケースの追加が容易になる
- テストソースコードの重複を大幅に削減できる
