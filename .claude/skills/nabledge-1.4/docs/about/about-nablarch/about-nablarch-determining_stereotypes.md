# 業務コンポーネントの責務配置

## 業務コンポーネントの責務配置

各業務クラスの責務配置を以下に示す。

| 名称 (クラス接尾辞) | 責務 |
|---|---|
| 業務アクションハンドラ (Action) | フレームワークから直接コールバックされる業務処理のエントリーポイント。取引ごとに1つ作成する（例: ユーザ一覧照会、ユーザ情報登録）。単純な業務処理はActionに直接実装可。複雑な処理や複数業務・処理形態（バッチと画面オンラインなど）から共用される処理はComponentに分離する。 |
| 業務共通コンポーネント (Component) | 業務ロジックを実装するクラス。HTTPリクエストオブジェクトや実行コンテキストなどの実行制御基盤に属するオブジェクトに直接依存することは避けること。 |
| 業務フォーム (Form) | データ保持と外部入力値の精査を行うモジュール。単項目・項目間入力精査を実装する。DB等へのアクセスが必要な精査処理はActionで実装する。 |
| 業務画面 (View) | 画面オンライン処理のユーザインタフェース（通常はJSP）。業務処理は実装せず、Actionから渡される処理結果を表示する。 |

> **注意**: ActionはHTTPリクエストオブジェクトや実行コンテキストに依存するため、自動テストはリクエスト単体テストで行う必要がある。複雑な内部条件をもつ業務ロジックをActionに実装した場合、テストデータのセットアップが煩雑になる。このような場合は業務処理部分をComponentとして切り出し、クラス単体テストを行うこと。

各構成要素間の処理フロー（画面オンライン処理の例）:

1. View（Webクライアント）からリクエストが送られる
2. アプリケーションフレームワークがリクエストを受信し、Actionを呼び出す
3. Actionはバリデーションを実行し、画面入力値が格納されたForm（オブジェクト）を生成してComponentを呼び出す
4. Componentはビジネスロジックを実行し、結果をActionに返す
5. Actionは必要に応じてComponentの戻り値を処理（リクエストスコープへの値の格納など）し、フレームワークに処理を返す
6. アプリケーションフレームワークはViewを処理（JSPをHTMLに変換など）し、クライアントにレスポンスを返す
