## Nablarch のバージョンアップ方針

Nablarchが提供するコンテンツ(詳細は [Nablarchの製品構成](../../about/about-nablarch/about-nablarch-contents.md#nablarch-contents) をご参照ください)に対するバージョンアップ方針について説明します。
Nablarchコンテンツ種類（Nablarchフレームワーク、 Nablarch標準/ガイド、 Nablarchライブラリ、Nablarchツール）については、 [Nablarchのコンテンツ種類](../../about/about-nablarch/about-nablarch-contents-type.md#contents-type) を参照。

### バージョンアップ種類

Nablarchのバージョンアップ種類は、下記のとおりです。

| バージョンアップ種類 | 説明 | 更新対象コンテンツ | 後方互換性の維持 [1] | リリースサイクル | バージョンアップ時のお客様作業 |
|---|---|---|---|---|---|
| メジャーバージョンアップ | Nablarchアーキテクチャデザイン更新を伴う、 Nablarchの全面改訂です。  技術革新等、大きな外部環境の変化に追随する 場合、ソフトウェアやサービスの新規追加に 伴ってアーキテクチャデザインを見直す場合 等が想定されます。 | Nablarchフレームワーク Nablarch標準/ガイド Nablarchライブラリ Nablarchツール | 対象外 | サイクルなし | お客様のアプリケーションの修正、 およびNablarchオブジェクトコード (jarファイル)の差し替えと 設定ファイルの書き換え。  後方互換性は維持されず、システム再構築と 同等となる場合もあります。 |
| マイナーバージョンアップ | Nablarchフレームワークに対する新規機能追加 および大規模な変更を伴う機能変更を行います。  例）  * 実行制御基盤の追加 * 実行制御基盤の大規模な内部構造変更 |  | 対象 | １年～ | Nablarchのオブジェクトコード(jarファイル) の差し替えと設定ファイルの書き換え。  後方互換性が維持されるため、 原則として [2] 、 お客様のアプリケーション修正は発生しません。 |
| リビジョンアップ [3] | 不具合対応、機能追加・変更 （Nablarchフレームワークについては、 いわゆる「枯れた」状態にするための 改善レベルの機能追加・変更）を行います。  例）  * Java最新版対応 * アプリケーションライブラリの追加/変更 * 標準/ガイドの追加/変更 |  |  | 4半期～半期  不具合については、 影響範囲を見定め リリーススケジュールを 決定し、リリースします。 |  |
| バグフィックス [3] | セキュリティ・運用レベルに 致命的な影響を与える、 緊急性が高いNablarchフレームワークの 不具合に対応します。 | Nablarchフレームワーク |  | 随時  不具合検出後、 直ちに対応します。 |  |

Nablarchの後方互換性ポリシーについては、 [後方互換性ポリシー](../../about/about-nablarch/about-nablarch-versionup-policy.md#backward-compatibility-policy) をご参照ください。

マイナーバージョンアップ、リビジョンアップ、バグフィックスの場合も、例外的に不具合対応のため、やむを得ず後方互換性が保てない場合があります。
この場合、テストコード、テストデータ、お客様のアプリケーションの修正が必要になります(詳細は [後方互換性の例外](../../about/about-nablarch/about-nablarch-versionup-policy.md#backward-compatibility-is-not-maintained) をご参照ください)。

不具合対応は、お客様が使用しているメジャーバージョンおよびマイナーバージョンの組み合わせの最新リビジョンに対して行います。

例えば、お客様がご利用のNablarchバージョンが1.4.1で、Nablarch1.4.3がリリースされている場合

リビジョンアップ　→  1.4.3に対してリビジョンアップを行い、1.4.4としてリリース。
バグフィックス　   →  1.4.3に対してバグフィックスを行い、1.4.3.1としてリリース。

### 後方互換性ポリシー

Nablarchの後方互換性ポリシーについて説明します。

#### 後方互換性を維持する範囲

Nablarchフレームワークとして提供するコンテンツ（アプリケーションフレームワーク(NAF)およびテスティングフレームワーク(NTF)）に対して、
[後方互換性の例外](../../about/about-nablarch/about-nablarch-versionup-policy.md#backward-compatibility-is-not-maintained) に該当する場合を除き、
[後方互換性維持の内容](../../about/about-nablarch/about-nablarch-versionup-policy.md#backward-compatibility-contents) に記載の内容で、後方互換性を維持いたします。

なお、Nablarchフレームワーク以外のコンテンツに対する後方互換性の維持は考慮されません。

#### 後方互換性維持の内容

Nablarch では、下記のポリシーで後方互換性を維持するよう目指しています。

* Nablarchフレームワークのバージョンアップを行った場合に、出来る限り下記を発生させないよう考慮します。

  * 既存のアプリケーションコードの修正。
  * 既存の自動テストコードの修正。
  * 既存の自動テストデータの修正。

この後方互換性維持の方針によりNablarchフレームワークは、基本的にオブジェクトコード(jarファイル)の差し替えと設定ファイルの変更のみでバージョンアップできます。

> **Warning:**
> この後方互換性維持の方針は、NAF/NTFのAPIのうち、Nablarchが定める公開APIを対象にしています。
> Nablarchが定める公開APIとはjavadocにより仕様を公開しているAPIになります。
> 公開APIに対して、javadocにより仕様を公開していないAPIは、非公開APIになります。

> **非公開APIは、事前通知なく後方互換性が維持されないバージョンアップを行う場合がありますので、プロジェクトにて非公開APIを使用しないでください。**
> **プロジェクトにて非公開APIを使用した場合、バージョンアップ時に後方互換が維持されず、思わぬ不具合が発生する可能性があります。**

> Nablarchでは、非公開APIの使用を検知するツールを提供しています。
> プロジェクトにてこのツールを使用して非公開APIが使用されないように運用してください。
> ツールの詳細は、 [使用不許可APIチェックツール](./../nablarch/app_dev_guide/guide/development_guide/08_TestTools/05_JavaStaticAnalysis/UnpublishedApi.html) を参照してください。

#### 後方互換性の例外

下記内容に該当する場合は、事前通知なく後方互換性が維持されないバージョンアップを行う場合があります。

* Nablarchフレームワークが出力するログのレベル、文言に対する変更。
* アーキテクチャデザインの変更を目的とするバージョンアップを行った場合。 [4]

下記内容に該当する場合は、事前通知の上で後方互換性が維持されないバージョンアップを行う場合があります。

* Nablarchの不具合が検出され、その対応が後方互換性を維持したまま実施できない場合。
* Nablarchを動作させる環境である、JDKのバージョンアップに起因する問題が発生し、その対応が後方互換性を維持したまま実施できない場合。

このバージョンアップが実施されるのは、メジャーバージョンアップ時のみです。
バグフィックス、リビジョンアップ、マイナーバージョンアップの際は、アーキテクチャデザインの変更は実施されません。

### EOSLポリシー

Nablarchは、基幹システムのアプリケーション開発・実行基盤として安心してご利用いただくために、EOSL [5] の期間は設けません。
ご利用中のお客様がいるバージョンに対しては、基本サービスを提供いたします。

基本サービスの内容については、 [基本サービスの内容](../../about/about-nablarch/about-nablarch-support-service.md#basic-support-service) をご覧ください。

EOSL：End Of Service Lifeの略で、製品に対するサポートやサービスの提供を終了することです。

> **Note:**
> * >   新規に導入いただく場合は、原則として、契約時点での最新版をご導入いただきます。
> * >   不具合対応などのバージョンアップはお客様がご利用中のマイナーバージョンの最新のリビジョンに対してのみ実施されます。
> * >   マイナーバージョンアップでは、既にご利用中のお客様がいなくなったバージョンはお選びいただけません。

> バージョン番号については、 [バージョン番号体系](../../about/about-nablarch/about-nablarch-versionup-policy.md#version-number) をご参照ください。

### バージョン番号体系

#### バージョン番号体系

Nablarchのバージョン番号体系は、下記のとおりです。

```html
(メジャーバージョン番号).(マイナーバージョン番号).(リビジョン番号).(バグフィックス番号)(安定度)
```

例）
1.4.0   ： 正式リリースバージョン（GA版）
1.4.0.1 ： 正式リリースバージョン（GA版）に対して緊急不具合リリースを１回実施したバージョン
1.4.0b1 ： βリリース(1回目）のバージョン

##### メジャーバージョン番号

メジャーバージョンアップを行う際にインクリメントする番号です。
開始番号は1です。
全てのバージョン表示にメジャーバージョン番号を表記します。
他のバージョンアップが行われても、メジャーバージョン番号は変化しません。

##### マイナーバージョン番号

マイナーバージョンアップを行う際にインクリメントする番号です。
開始番号は0です。
全てのバージョン表示にマイナーバージョン番号を表記します。
メジャーバージョンアップを行う際に、マイナーバージョン番号を0にリセットします。

##### リビジョン番号

リビジョンアップを行う際にインクリメントする番号です。
開始番号は0です。
全てのバージョン表示にリビジョン番号を表記します。
メジャーバージョンアップ、マイナーバージョンアップを行う際に、リビジョン番号を0にリセットします。

##### バグフィックス番号

バグフィックスを行う際にインクリメントする番号です。
開始番号は1です。
緊急リリースを実施した場合のみ、バージョン表示にバグフィックス番号を表記します。

##### 安定度

Nablarchコンテンツの安定度を示すバージョン種類は、下記のとおりです。
GA版以外のリリースを行う際に、バージョン表示に安定度を表記します。

| 安定度 | リリース後のコンテンツ変更 |  |  |
|---|---|---|---|
| α | あり | あり | あり |
| β | なし | あり | あり |
| RC | なし | なし | あり |
| GA | － | － | － |

#### コンテンツとバージョン番号の関係

Nablarchのコンテンツにはさまざまな物が含まれます(詳細は [Nablarchの製品構成](../../about/about-nablarch/about-nablarch-contents.md#nablarch-contents) をご参照ください)。
これら一式に対して1つのバージョン番号が割り振られます。

#### 差し替え用コンテンツ提供

Nablarchフレームワーク以外のコンテンツについては、迅速に不具合修正版をご提供する方法として、バージョンアップではなく差し替え用コンテンツをご提供することがあります。差し替え用コンテンツとしてご提供したコンテンツは、後日のバージョンアップでNablarchに取り込まれます。
差替えコンテンツのバージョン体系は、下記のとおりです。

```html
patch(メジャーバージョン番号).(マイナーバージョン番号).(リビジョン番号).(バグフィックス番号)(安定度)_yyyyMMdd_連番(1開始)
```

例）
patch1.4.2_20140201_01
