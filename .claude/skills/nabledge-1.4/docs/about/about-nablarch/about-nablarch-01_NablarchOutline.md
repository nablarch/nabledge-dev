# Nablarch Application Framework 概要

## NAFのアプリケーション処理モデル

NAFのアプリケーション処理モデルは、入力データに対して処理を行い結果を返す最小単位の繰り返しで構成される。

- 画面オンライン処理: 入力=HTTPリクエスト、出力=HTTPレスポンス
- バッチ処理: 入力=ファイルまたはDBレコード、出力=処理結果ステータス

**アプリケーションの基本構造**

NAFの処理は「トランザクション制御」「応答電文送信」などのステップに分割され、各ステップを **ハンドラ** が実装する。

- **ハンドラキュー**: 各ハンドラを実行順に並べたキュー。先頭から順に実行される。
- 各ハンドラは入力データを処理した後、同じ入力データを引数として次のハンドラを呼び出す。
- 最終的に **業務アクションハンドラ** が業務処理を実行し結果をリターンする。
- 処理結果オブジェクトはハンドラキューを逆方向に遡り、先頭のハンドラまで戻った時点で処理が完了する。

## 標準ハンドラ構成

フレームワークが提供するハンドラの種類:

**汎用的に利用できるハンドラ**
- トランザクション制御ハンドラ
- リクエストディスパッチハンドラ
- 認可制御ハンドラ
- 開閉局制御ハンドラ

**画面オンライン処理に固有のハンドラ**
- HTTPレスポンスハンドラ
- HTTPアクセスログハンドラ
- Nablarchカスタムタグ制御ハンドラ

**バッチ処理に固有のハンドラ**
- プロセス常駐化ハンドラ
- マルチスレッド実行制御ハンドラ
- プロセス多重起動防止ハンドラ

**メッセージング処理に固有のハンドラ**
- 電文応答制御ハンドラ
- 再送電文制御ハンドラ

NAFでは標準的なハンドラキューの構成を用途ごとに定義しており、これを **標準ハンドラ構成** と呼ぶ。

## 業務アプリケーションの実装

業務アプリケーション開発者が実装するハンドラは **業務アクションハンドラ** のみ。その他のハンドラはフレームワーク提供のものをそのまま使用できる。

業務アクションハンドラの処理フロー:

1. **入力データの取得** (図中①): 入力データはMap形式で取得できる。入力精査やDBアクセスライブラリもMapを引数・戻り値として使用する。

2. **入力精査** (図中②): フォーム定義に沿って精査処理を行う。
   - **フォーム**: システムへの入力項目の精査と値の保持を行うクラス。画面項目定義や電文フォーマット定義に沿って作成。フォームクラスの各項目には宣言的に精査仕様を設定できる。
   - **エンティティ**: フォームのうち、RDBMSのテーブルと1対1に紐づくクラス（フォームクラスのプロパティとテーブルのカラムも1対1）。テーブル定義に沿って作成される。
   - DB内容との比較などの処理を含む精査はビジネスロジックの一部として実装する。

3. **ビジネスロジックの実行** (図中③): 精査済み入力データをフォームまたはエンティティオブジェクトとして取得し、ビジネスロジックを実行する。フレームワーク提供のDB/データファイル/メッセージキューの入出力ライブラリも利用可能（Mapインターフェース経由）。
   - **業務共通コンポーネント** (図中④): 複数の業務アクションから共用しうるロジックは業務アクションハンドラに直接実装せず、個別クラス（業務共通コンポーネント）として実装することを推奨。既存の業務共通コンポーネントが適用可能であれば積極的に利用すること。

4. **処理結果の返却** (図中⑤): ビジネスロジックの処理結果を表すオブジェクトを返却する。バッチ処理など応答を伴わない場合は正常終了を表すマーカオブジェクトをリターンする。業務アクションハンドラから実行時例外が送出された場合は、ハンドラキュー上のハンドラによって自動的にトランザクションがロールバックされる。

**レスポンス処理**: 画面オンライン処理や同期応答メッセージング処理では、応答の内容を指定するオブジェクトを処理結果として返却する必要がある。
- 画面オンライン処理: 遷移先JSPパスを指定した `HttpResponse` オブジェクトを返す
- メッセージング処理: 応答電文の内容・宛先キュー名を指定した `ResponseMessage` オブジェクトを返す
- これらのオブジェクトはハンドラキュー上の **レスポンスハンドラ** によって処理され、送信元にレスポンスとして返される。

## 業務コンポーネントの責務配置

業務機能は以下のコンポーネントに実装される: 業務アクションハンドラ、フォーム/エンティティ、業務共通コンポーネント。

各クラスへの責務配置については **様々な考え方があり、特定の方法に絞ることはできない**。開発者はプロジェクトの状況に応じて責務配置を判断する必要がある。

典型的なプロジェクトで採用される責務配置の例については [業務コンポーネント責務配置](../../../knowledge/about/about-nablarch/assets/about-nablarch-01_NablarchOutline/業務コンポーネント責務配置.xlsx) を参照。

## 実行コンテキストとスコープ

業務アクションハンドラからデータを業務画面に渡したり、ハンドラ間でデータを保持する際は、**実行コンテキスト**と呼ばれるオブジェクトにデータを保持できる。

実行コンテキストのデータは以下のスコープに分けられる:

| 名称 | 説明 |
|---|---|
| リクエストスコープ | 1リクエストの処理の間データを保持する。画面オンライン/メッセージング処理では1リクエスト処理中、バッチ処理では1データ処理中に同じデータを保持できる。 |
| セッションスコープ | 複数リクエストの処理の間データを保持する。画面オンライン処理ではログアウトまで、メッセージング処理では1リクエスト処理中、バッチ処理では処理開始〜終了まで。 |
| ウィンドウスコープ | 画面オンライン処理のみ使用可能。セッションスコープと同様に複数リクエスト間でデータを保持するが、複数画面を同時使用した際に画面ごとに別々のデータを保持できる。使用方法は他の2種類と異なる。詳細は :ref:`guide_appendix_windowScope` を参照。 |
