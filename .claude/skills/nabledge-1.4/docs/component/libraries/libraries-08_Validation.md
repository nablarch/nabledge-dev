# バリデーションとFormの生成

## 概要

入力値のバリデーション機能とFormの生成機能を提供する。

**Form**の責務:
1. 外部入力値の入力チェック
2. 外部入力値の保持

バリデーションのレベル:
1. 単項目の入力チェック
2. 複数項目にまたがる入力チェック
3. データベースのデータを使用した入力チェック

本機能では1, 2のチェックをサポートする。3はアプリケーションプログラマが別途実装すること。

使用条件:
- 入力値はMap形式であること
- Formはセッタとゲッタを持つクラスであること
- バリデーション情報はFormにアノテーションとして設定すること

本機能はActionで業務ロジックを呼び出す前の入力チェックに使用する。[validation_and_convert](libraries-08_02_validation_usage.md) 以降のバリデーション使用方法を参考にしてActionを実装すること。

**Entity**: FormをRDBMSのテーブルと1対1に作成し、Formのプロパティとテーブルのカラムを1対1に対応づけた形式のFormを特にEntityと呼ぶ。この条件を満たすと以下の利点がある:
- バリデーションとデータベースへの反映を容易に実装できる
- RDBMSの定義から実装の多くの部分を自動生成できる

## 特徴

**アノテーションによる条件の記述**

バリデーションはFormに設定したアノテーションをもとに実施されるため、煩雑な単項目の入力チェック処理の実装が不要となり、バリデーションの実装が極小化できる。ハードコーディングによるバリデーションを行う場合と比較して生産性を向上できる。また、Formにバリデーション条件を保持するため、画面処理とバッチ処理等、異なる入力元に対してバリデーション条件を共有できる。

**メッセージ機能の使用**

バリデーション結果のメッセージには [./07_Message](libraries-07_Message.md) の機能を使用する。バリデーションメッセージとビジネスロジックのメッセージを透過的に扱えるため、エラーメッセージ表示処理でどちらのメッセージも同じように取り扱える。

## 要求

**実装済み**

- 文字列が特定の文字種のみで構成されているかチェック
- 文字列長チェック
- 必須入力値チェック
- 数値が規定の範囲に入っているかチェック
- 入力値のMapからFormインスタンスを生成
- 入力値をFormのプロパティの型に変換可能かチェック
- 入力値をFormのプロパティの型に合わせて変換
- Formのプロパティに設定する前に値に対してバリデーション実行
- エラーメッセージに出力するプロパティの表示名称を任意指定
- エラーメッセージに出力するプロパティの表示名称の国際化
- Form内部のプロパティのFormに対するバリデーション
- 西暦(年月日)フォーマットの日付チェック（詳細は :ref:`ExtendedValidation_yyyymmddConvertor` 参照）
- 入力値の文字列に対してトリム
- コード値として有効かのチェック

**未実装**

- 和暦フォーマット、年月フォーマット、営業日の日付チェック
- 入力禁止文字チェック（システム毎に規定された禁止文字が含まれていないかのチェック）
- 制御文字チェック
- パスワード強度判定
- 氏名フォーマットチェック
- アノテーションによるトリム要否設定
- 日本語電話番号フォーマットチェック
- メールアドレスフォーマットチェック

**未検討**

- 正規表現マッチチェック
- 複数アノテーションをまとめたアノテーション定義
- カード番号チェック
- 郵便番号チェック
- 金額チェック
- 文字列の範囲チェック
- トリム後バリデーション
- 外部入力値とDBの値をマージしたFormへのバリデーション
