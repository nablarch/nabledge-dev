## 常駐バッチ実行制御基盤

本節で解説する [常駐バッチ実行制御基盤](../../processing-pattern/nablarch-batch/nablarch-batch-batch-resident.md) では、一定間隔 [1] ごとに要求データを検索し、
随時処理を行う常駐型プロセスを作成するための制御基盤を提供する。

本実行制御基盤では、マルチスレッドで実行した場合、各スレッドが一定間隔で要求データの有無をチェックする。
このため、幾つかのスレッドで処理遅延が発生した場合でも、残りのスレッドで随時要求データを処理することができる。

> **Note:**
> [常駐バッチ実行制御基盤](../../processing-pattern/nablarch-batch/nablarch-batch-batch-resident.md) のハンドラ構成は、Nablarchバージョン1.4.3で大きく変更している。
> 1.4.2以前のハンドラ構成については、 [常駐バッチ実行制御基盤（スレッド同期型）](../../processing-pattern/nablarch-batch/nablarch-batch-batch-resident-thread-sync.md) を参照。

> [常駐バッチ実行制御基盤（スレッド同期型）](../../processing-pattern/nablarch-batch/nablarch-batch-batch-resident-thread-sync.md) では、処理が遅いスレッドの終了を他のスレッドが待つことにより、 要求データの取り込み遅延が発生する。
> このため、新規プロジェクトでは本ドキュメントに従い、新しいハンドラ構成にて常駐バッチを作成することを推奨する。

要求データが存在しない場合に一定時間待機する。

### 基本構造

[常駐バッチ実行制御基盤](../../processing-pattern/nablarch-batch/nablarch-batch-batch-resident.md) の構造は、前半の **プロセス制御部分(メインスレッド)** と
後半の **リクエスト処理部分(サブスレッド)** の2つに大きく別れる。

1. プロセス制御部分(メインスレッド)

Javaコマンドから [共通起動ランチャ](../../component/handlers/handlers-Main.md) を実行することによって開始され、 [リポジトリ](../02_FunctionDemandSpecifications/01_Core/02_Repository.html) の初期化処理を実行する。

初期化処理完了後、ハンドラキュー上のハンドラを順次実行し、 [業務アクションハンドラ](../../component/handlers/handlers-BatchAction.md) の初期化処理とデータリーダの生成を行う。

[マルチスレッド実行制御ハンドラ](../../component/handlers/handlers-MultiThreadExecutionHandler.md) でサブスレッドを作成した後、サブスレッドが終了するまで待機する。

1. リクエスト処理部分(サブスレッド)

プロセス制御部分で生成したデータリーダを使用し、要求管理テーブル(常駐バッチ処理のインプットとなるテーブル)の要求データを検索する。

要求管理テーブルに要求データ(未処理のデータ)が存在する場合には、
そのデータを読み込み [業務アクションハンドラ](../../component/handlers/handlers-BatchAction.md) を実行する。
業務アクションハンドラでは、要求データを元に業務処理を行う。

業務アクションハンドラの処理が終了すると、 [リクエストスレッド内ループ制御ハンドラ](../../component/handlers/handlers-RequestThreadLoopHandler.md) により、要求管理テーブルの検索処理に戻る。

以降、同様の処理を繰り返す。

> **Note:**
> 実行する業務アクションハンドラの指定は、バッチプロセス起動時の引数で指定する。
> 詳細は、 [都度起動バッチ実行制御基盤](../../processing-pattern/nablarch-batch/nablarch-batch-batch-single-shot.md) を参照すること。

### 業務アクションハンドラの実装

業務アクションを実装するには、FW側で提供されるテンプレートクラスを継承して作成する。
詳細は、 [バッチ処理用業務アクションハンドラのテンプレートクラス](../../component/handlers/handlers-BatchAction.md) を参照すること。

|---|---|---|---|---|

|  | 異常フロー | 重複起動エラー | プロセス起動時、すでに同一プロセスが 起動していた場合は異常終了する。 | フローを表示 |
| リクエストスレッド内制御 | 正常フロー | 要求データあり | 要求管理テーブル上に未処理レコード(要求データ)が 存在する場合、要求データを取得し業務処理を実行する。  > **Note:** > 要求の2重取り込みを防止するために、現在処理中のデータをヒープ上で管理する。 > これにより、処理の遅いスレッドが処理中のデータが未処理として要求管理テーブル > 上に残り続けた場合でも、2重にデータが取り込まれることを防止できる。 | フローを表示 |
|  | 代替フロー | 要求データなし | 要求管理テーブル上に未処理レコード(要求データ)が 存在しない場合、一定時間待機後スレッドループの先頭に処理を戻す。  スレッドループの先頭から処理を再実行することで、 プロセス管理テーブルの状態が確認され処理の停止や閉局状態への変更が可能となる。 | フローを表示 |
|  | 代替フロー | 閉局中処理待機 | バッチ処理のリクエストIDに対する業務機能が閉局中であった場合は、 処理を行わず次の実行タイミングまで待機する。 | フローを表示 |
|  | 異常フロー | 業務エラー | 処理中に実行時例外が送出された場合は、障害通知ログを出力した上で、 リトライ可能例外を送出しリトライハンドラに処理を移譲する。  エラーが発生した要求データは、エラーステータスとなり データメンテを行わない限り要求データとして処理されることはない。 | フローを表示 |

| プロセス停止制御 | 正常フロー | プロセス正常停止 | プロセス管理テーブルのフラグを変更する ことで、リクエストスレッドでの新規処理 受付を停止する。その後、全スレッドの 終了を待ってプロセスを正常終了させる。 | フローを表示 |
|  | 異常フロー | プロセス異常停止 | プロセスの継続に影響する致命的なエラーが 発生した場合は、リクエストスレッドでの 新規処理の受付を停止した上で、全スレッド の終了を待ってプロセスを異常終了させる。 | フローを表示 |
|  | 異常フロー | DB接続リトライ失敗 | DBの接続リトライが上限回数に達しても 成功しなかった場合は、プロセスを異常停止 させる。 | フローを表示 |
