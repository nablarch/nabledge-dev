# 意味検索 設計書

**Status**: Draft
**Date**: 2026-05-12

## 目的

ユーザーの自然言語質問から、知識ファイルの関連セクションを正確に特定する。

## 方式

### 概要

```
質問 → Stage 1: index.mdで候補ページ選定 → Stage 2: 本文を読みセクション選定 → セクションIDリスト
```

2段階方式: インデックスで絞り込み、本文で精密選定。

### プロンプト構成

スキルワークフロー: `workflows/semantic-search.md`（Stage 1・Stage 2プロンプトをインライン化済み）

```
入力: 質問文, ヒアリング回答（あれば）

Stage 1: 候補ページ選定
  1. Read knowledge/index.md
  2. 質問とヒアリング回答からユーザーの意図を特定する
  3. index.mdのセクションタイトル一覧から、回答に必要な情報を含みそうなページ（ファイル）を選定する
  4. 最大10ページ。確信度が高いものから順に選定する
  5. 中間出力: 選定したファイルパスと選定理由（1行）のリストを出力する

Stage 2: セクション選定
  1. Stage 1で選定した各ページの知識ファイル（JSON）を、確信度の高い順に読み込む
  2. 各セクションの本文を読み、質問への関連度を判定する
  3. 選定基準:
     - high: このセクションの情報がないと質問に回答できない
     - partial: 回答の補足や前提知識として有用（highで不足する情報を補完するもの）
  4. 選定順序: まずhighセクションを全て選定し、残り枠（30 − high数）でpartialを追加する
  5. 最大30セクション

出力: pointer JSON（下記スキーマ）。results はrelevance降順（high → partial）でソートする
```

### 出力スキーマ

下流のqa.md、read-sections.sh、ベンチマークランナーと共有するpointer JSONスキーマ。

```json
{
  "results": [
    {
      "file": "component/libraries/libraries-universal-dao.json",
      "section_id": "s9",
      "relevance": "high"
    },
    {
      "file": "component/libraries/libraries-database.json",
      "section_id": "s29",
      "relevance": "partial"
    }
  ]
}
```

| Field | Type | Description |
|---|---|---|
| file | string | knowledge/からの相対パス |
| section_id | string | セクションID（s1, s2, ... 形式） |
| relevance | "high" \| "partial" | high: 直接回答可能 / partial: 前提知識・関連情報 |

results はrelevance降順（high → partial）。空配列は一致なし。

### Stage 1 ガイドライン（ページ選定）

**選定すべきページ:**
- 質問の操作対象に直接対応するページ（例: "ページングしたい" → ユニバーサルDAO）
- 質問の技術的課題を解決する機能のページ（例: "メモリが足りない" → ユニバーサルDAO）
- 質問のコンテキストに必要な処理方式のページ（例: "REST APIで登録" → RESTfulウェブサービスの登録）
- ヒアリング回答で特定された処理方式に関するページ

**選定しないページ:**
- 質問と関連はあるが直接的でないもの
- 質問の処理方式と異なるページ（例: ヒアリングでREST明示時にWeb画面のページ）

**不確実な場合:**
- 確信度が高いファイルが3件未満の場合、中間出力でその旨を記載する（無理に10件まで埋めない）
- 10件を超える候補がある場合、超過原因を分析し、適切な絞り込み観点を提示して中間出力に記載する

### Stage 2 ガイドライン（セクション選定）

**high（必須）:**
- このセクションの情報がないと質問に回答できない
- 質問の操作対象に直接対応する（例: "遅延ロードする" セクション）
- 質問の技術的課題を解決する実装方法を含む

**partial（補足）:**
- highセクションでは不足する情報を補完するもの
- 前提知識や関連する設定方法で、highセクションの内容を実装する際に必要なもの

**partialに含めないもの:**
- highセクションの内容から既に推測可能な情報
- 概念定義のみで実装詳細がないもの
- highセクションと同じ情報を別の切り口で説明しているだけのもの

**選定しないセクション:**
- 一般的な概要のみで具体的な情報がないもの（"機能概要" で内容が空）
- モジュール一覧、改訂履歴など定型情報
- 質問と関連はあるが直接的でないもの

## ヒアリング回答の構造

ヒアリング回答は2軸の構造化データとして表現する。

| 軸 | フィールド | 内容 | 例 |
|---|---|---|---|
| 処理方式 | processing_type | Nablarchの処理方式 | Web / REST API / Nablarchバッチ / 都度起動バッチ |
| やりたいこと | goal | ユーザーの具体的な目的 | 入力画面のフォームでバリデーションを使う |

### 処理方式の制約

- 処理方式は**1つ**に絞る。「WebとREST APIの両方」のような複合回答は受け付けない
- 複数の処理方式に跨る質問は、処理方式ごとに分けて回答する
- 処理方式が該当しない質問（テスティングフレームワーク等の横断的機能）では、processing_typeをnullとする

### 検索への影響

- Stage 1: processing_typeに該当する処理パターンのページを優先選定
- Stage 2: goalに基づいて関連セクションの絞り込みを精緻化

### プロンプトでの表示形式

ヒアリング回答がある場合:
```
処理方式: Web
やりたいこと: 入力画面のフォームでバリデーションを使う
```

processing_typeがnullの場合は「処理方式」行を省略する。ヒアリング回答がない場合は「なし」。

## index.md 設計

### フォーマット

```markdown
# Knowledge Index

## component/libraries

### ユニバーサルDAO
path: component/libraries/libraries-universal-dao.json
- s2: SQLを書かなくても単純なCRUDができる
- s3: 検索結果をBeanにマッピングできる
- s5: 使用方法
  - s6: 設定
  - s7: SQLファイルで検索する
  - s8: テーブルをJOINした検索結果を取得する
  - s9: 検索結果を遅延ロードする
  - s11: ページングを行う
  - s14: バッチ実行(一括登録、更新、削除)を行う
  - s15: 楽観的ロックを行う
  - s20: 現在のトランザクションとは異なるトランザクションで実行する

### データベースアクセス(JDBCラッパー)
path: component/libraries/libraries-database.json
- s1: 機能概要
  - s2: データベースの方言を意識することなく使用できる
  - s3: SQLはロジックではなくSQLファイルに記述する
  - s6: 実行時のBeanオブジェクトの状態を元にSQL文を動的に構築できる
- s9: 使用方法
  - s12: SQLをファイルで管理する
  - s13: SQLIDを指定してSQLを実行する
  - s21: $if構文を使用して条件に合ったSQL文を動的に構築する
  - s29: 別のトランザクションで実行する
```

### 構成ルール

1. **カテゴリごとにH2** — `## component/libraries`, `## processing-pattern/nablarch-batch` 等
2. **ファイルごとにH3** — ファイルタイトルをH3、直下にpath行
3. **L2セクションをリストアイテム** — `- sN: タイトル`
4. **L3セクションをインデント** — `  - sN: タイトル`（L2の下にネスト）
5. **L4以下は省略** — L3までで十分
6. **セクション0件のファイル** — H3 + path行のみ
7. **汎用タイトルはそのまま** — ファイルのH3が文脈を提供

### 省略ルール

以下のタイトルのセクションは省略する（検索対象にならない定型情報）：
- "モジュール一覧"
- "アプリケーションフレームワーク"（リリースノートのカテゴリヘッダ）
- "制約"（ハンドラの定型セクション、多くが空）
- "ハンドラクラス名"（定型情報）

releases カテゴリもセクションタイトルは具体的な変更内容を表すため、他と同じルールで出力する。

### 実測サイズ（v6プロトタイプ）

| 指標 | 値 |
|------|-----|
| 総文字数 | 74,506 |
| 総行数 | 2,916 |
| ファイル数 | 353 |
| セクションエントリ数 | 1,914 |
| 推定トークン数 | ~19K |

全5バージョン推定: v5 ~25K, v1.4 ~21K, v1.3 ~14K, v1.2 ~14K。
全バージョンでコンテキストウィンドウの10%以下。

### Excel由来ファイル（check, releases）

Excel由来セクションはlevelフィールドなし。
- check: 全セクションをフラットリスト（セキュリティチェック項目は検索対象）
- releases: 全セクションをフラットリスト（バージョンごとの変更内容は検索対象）

## RBKC実装

index.md生成は新規RBKCステップとして実装。

**入力**: knowledge/*.json（全知識ファイル）
**出力**: knowledge/index.md

**生成ロジック**:
1. knowledge/配下のJSONを再帰的に走査
2. ディレクトリ構造からカテゴリ（H2）を生成
3. 各JSONファイルからH3 + path + セクションリストを生成
4. 省略ルールを適用
5. ソート: カテゴリ内はファイル名の辞書順

**クロスバージョン**: 同一ロジックで全5バージョンに適用。バージョン固有の違い（カテゴリ構成）はディレクトリ構造から自動反映。

## 検証ポイント

1. **index.md のサイズ**: 実際にRBKCで生成して全5バージョンのトークン数を確認
2. **AI選定の精度**: シミュレーション — AIにindex.mdを渡してセクション選定させ、must-sectionとの一致率を測定
3. **レイテンシ**: index.md読み込み + AI選定の所要時間

## 根拠

### 2段階方式の理由

タイトルのみの単発方式で実験した結果、以下の問題が判明した:

1. **汎用タイトルの問題**: "使用方法", "アクション" 等のタイトルは内容を反映しておらず、タイトルだけでは正しいセクションを選定できない
2. **セクション数が多すぎる**: v6で1,914セクション。AIがタイトルだけで正しい数件を選ぶのは困難
3. **本文を読まないとわからない**: 同一ファイル内の類似タイトルの区別にはコンテンツ読解が必要

2段階方式の利点:
- Stage 1でファイル数を絞る（353→5件）ことでStage 2の精度が向上
- Stage 2で本文を読むことで、タイトルでは区別できないセクション間の選定が可能
- 各ステージのタスクが単純（ファイル選定 vs セクション選定）で、AIの判断品質が向上

### 調査結果からの制約

| 制約 | 根拠 |
|------|------|
| 検索単位はセクション | ベンチマーク24件のmust-sectionは全て特定セクション。ファイル単位では粗すぎる |
| index.mdにセクション情報が必要 | セクション選定にはタイトル+文脈が必要 |
| セクションタイトルにファイルタイトルが文脈を提供 | 14.5%のセクションが汎用タイトル("概要", "使用方法")。ファイル文脈なしでは特定不可 |

### Stage 1: 最大10ページの根拠

ベンチマーク24件のmust-sectionが参照するファイル数: 最大4ファイル（qa-12）。
当初5ページ上限で運用したが、qa-12（必要7件 > 上限5件）でファイル枠超過による未選定が発生。
10ページ上限は最大実績の2.5倍のマージンで、枠超過リスクを排除する。

10件を超える候補がある場合は、超過原因を分析し適切な絞り込み観点を提示する（ヒアリングコンポーネントで処理パターンの絞り込みを必須化して対応）。

### Stage 2: 最大30セクションの根拠

ベンチマーク24件のmust-section数: 平均1.67件/シナリオ、最大4件。acceptable含めても最大6件。
当初10セクション上限で運用したが、qa-02でセクション枠超過（10枠を他ファイルのセクションが占有し、必要なs8を選定できない）が発生。
Stage 1の上限が10ページに拡大したことで、読み込むセクション総数も増加する。30セクション上限は十分なマージンを確保する。

30件で不足するケースへの対応:
- semantic-searchの出力が空（results: []）の場合、qa.mdが「知識ファイルに含まれていません」レスポンスを返す
- 回答生成後のverify.md（根拠検証）でmust factが欠けていた場合、再検索を指示可能（将来実装）
- ベンチマーク §検証ポイント2 で30件上限がmust-section missを起こさないことを全シナリオで確認する（必須ゲート）
