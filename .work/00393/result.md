# ベンチマーク精度検証レポート

## 論点

DeepEval の自動評価は nabledge-6 の品質管理に信頼できる判断基準として使えるか？

## 結論

**DeepEval による自動評価は品質担保の手段として有効である。ただし誤検知が多く運用負荷がかかるため、改善が必要。**

### 検証結果

| 指標 | 誤検知率 | 見逃し率 |
|------|----------|----------|
| 正確性（answer_correctness） | 88.9%（8/9 件） | 0.0%（0/93 件） |
| 関連性（answer_relevancy） | 60.9%（14/23 件） | 0.0%（0/79 件） |
| 忠実性（faithfulness） | 94.3%（33/35 件） | 1.5%（1/67 件） |

- **誤検知**: ベンチマークが NG と判定したが、再検証（AI＋人間）では問題なかった件
- **見逃し**: ベンチマークが OK と判定したが、再検証（AI＋人間）では問題があった件

### 考察

DeepEval によるベンチマーク結果（102 件）を人間が全量確認した。見逃し率はほぼ 0 であり、問題のある回答を見逃していないことが確認できた。品質の安全網として機能している。

一方、誤検知率は全指標で 60〜94% と高く、NG アラートの大半は実際には問題のない回答だった。DeepEval は LLM as a Judge であり最終的には人間の判断が必要だが、今回の結果を見る限り **OK は信用に値する。NG は要確認** という運用が適切である。

誤検知の傾向を分析すると、関連性・忠実性とも実害はなく、回答品質の問題ではないことが分かった。関連性は回答末尾の出典メタデータ行が無関係コンテンツと判定されることが主因（誤検知 14 件中 13 件）。忠実性は nabledge-6 が意図的に要点のみ説明する（省略・要約）スタイルを DeepEval が矛盾と解釈することが主因である。

### 実績値からの提案

誤検知を減らすため、以下の閾値調整を提案する。

| 指標 | 現在の閾値 | 提案閾値 | 根拠 |
|------|-----------|---------|------|
| 正確性（answer_correctness） | 0.99 | 0.99 | Nablarch 開発での利用においてミスリードは許容できないため閾値は下げられない。誤検知が少ない（8件）ため、NG が出た場合は手動確認する運用でカバーする |
| 関連性（answer_relevancy） | 0.95 | 0.70 | 質問に直接関係しない同機能の補足説明や出典メタデータなど、実害のない情報を一定割合含むことを許容する。誤検知の最小スコアが 0.77 のため 0.70 に下げる |
| 忠実性（faithfulness） | 0.99 | 0.90 | 省略・要約など実害のない情報を一定割合含むことを許容する。0.90 に下げることで誤検知の大半（スコア 0.83〜0.89）を解消する |

### 今後の方針

今回は 34 シナリオ × 3 run = 102 件のサンプルによる検証であり、まだデータ量は限られている。実利用者からのフィードバックと合わせて誤検知・見逃しの傾向を継続的に観測しながら、閾値・評価ロジックを調整していく。

## 背景と検証方法

### 対象システム

**nabledge-6** は、Java フレームワーク Nablarch 6 に関する質問に回答する AI アシスタント（Claude Code スキル）である。ナレッジファイルを参照して回答を生成する。

### 現行ベンチマーク設定

**DeepEval** の RAG 評価メトリクスを使って回答品質を自動評価している。3 指標を測定し、スコアが閾値を下回ると NG（失敗）と判定する。本レポートで検証するのは、この指標・閾値・入力構成の組み合わせによる判定精度である。

| 指標 | DeepEval メトリクス | 測定内容 | スコアの意味 | 合格閾値 |
|------|---------------------|----------|-------------|----------|
| 正確性（answer_correctness） | `GEval` | 回答が期待事実をすべて含んでいるか | 期待事実をカバーしている度合い（1.0 = 完全カバー） | スコア ≥ 0.99 |
| 関連性（answer_relevancy） | `AnswerRelevancyMetric` | 回答が質問に関連しているか | 回答中の文のうち質問に関連する文の割合（1.0 = 全文が関連） | スコア ≥ 0.95 |
| 忠実性（faithfulness） | `FaithfulnessMetric` | 回答が参照コンテキストと矛盾しないか | 回答中の主張のうちナレッジで裏付けられる主張の割合（1.0 = 矛盾なし） | スコア ≥ 0.99 |

各シナリオで以下の 4 種類の情報を評価器に渡す。

| 評価器への入力 | 内容 |
|---|---|
| 質問文 | ユーザーの質問 |
| 回答 | nabledge-6 が生成した回答 |
| 期待事実 | 回答が必ず含むべき事実（数件） |
| 参照コンテキスト | nabledge-6 が参照したナレッジセクションの本文（ページ全文ではなく、選択されたセクションのみ） |

#### 入出力の例（代表 5 件）

| 質問 | 期待事実 |
|---|---|
| 大量データを検索するとメモリが足りなくなる。1件ずつ読み込む方法はないか？ | UniversalDao.deferメソッドで遅延ロードを使用する / DeferredEntityListのcloseメソッドを呼び出す必要がある |
| 業務トランザクションとは別のトランザクションでSQLを実行する方法はあるか？ロールバックされても別トランザクションの更新は残したい | SimpleDbTransactionManagerを使って個別トランザクションを定義する |
| 入力画面と確認画面のJSPを共通化して実装を減らす方法はあるか？ | confirmationPageタグで確認画面のJSPに入力画面のJSPへのパスを指定して共通化する |
| バッチ処理の進捗状況をWebSocketでリアルタイムにブラウザへ通知したい。NablarchでWebSocketを使う方法はあるか？ | NablarchにはWebSocketのサポートがないと回答している |
| REST APIでURLパスの一部を受け取ったり、検索条件をURL末尾のパラメータで渡す実装はどう書く？ルーティングの設定も含めて確認したい | パスパラメータはルーティング設定で名前を定義しリソースクラスで受け取る / クエリーパラメータはJaxRsHttpRequestから取得する |

### 検証方法

検証は 2 ステップで実施した。

**ステップ 1: AI 照合**  
別の AI エージェント（以下「**AI 照合**」）が 102 件を 1 件ずつ独立に読んで合否を判定した。AI 照合はベンチマークのスコアを参照せず、回答とナレッジのみを根拠とした。指標ごとの評価観点と参照した情報は以下の通り。

| 指標 | 観点 | 参照した情報 |
|------|--------------|-------------------|
| 正確性（answer_correctness） | 期待事実が回答に含まれているか | 回答・期待事実 |
| 関連性（answer_relevancy） | 回答に質問と無関係な内容が含まれているか | 回答・質問文・参照ナレッジページ全文 |
| 忠実性（faithfulness） | 回答がナレッジの記述と矛盾しているか | 回答・参照ナレッジページ全文 |

**ステップ 2: 人間による最終判断**（実施済み: 2026-07-08）  
102 件を 1 件ずつ読み、AI 照合結果を PR レビューコメントで確認・修正した。回答文のコンテキストを踏まえた再判断により、文字列比較による過剰な NG 判定を解消した。本レポートの数値はステップ 2 完了後の確定値である。

**対象データ**: `tools/benchmark/results/20260616-1214-fullbench-classes-v6/` の run-1〜run-3（34 シナリオ × 3 回 = 102 件）  
**照合日**: 2026-07-07

## 指標ごとの詳細

### 正確性（answer_correctness）— 回答が期待事実を含んでいるか

| | ベンチマーク OK | ベンチマーク NG |
|---|---|---|
| **再検証 OK** | 93 件（両者合意・問題なし） | **8 件（誤検知）** |
| **再検証 NG** | 0 件（見逃しなし） | 1 件（要・実害確認） |

**誤検知の原因**: ベンチマークは「回答に補足情報が多い」「参照事実と表現が異なる」といった理由で NG と判定するが、参照すべき事実は実際に回答に含まれている。

**誤検知の例:**
- [run-1-qa-12.md](checks/run-1-qa-12.md) — 期待事実「エラー表示タグでリクエストスコープのメッセージを表示する」は回答のStep 2・Step 3に含まれているが、包括的な説明すぎるとして NG（スコア 0.60）
- [run-3-review-09.md](checks/run-3-review-09.md) — 事実は含まれているが表現が異なるとして NG

**要・実害確認の例:**
- [run-3-pre-03.md](checks/run-3-pre-03.md) — 「検索結果がBeanにマッピングされる」という事実がコード例（`List<Project>` 等）では示されているが、文章として明示されておらず期待事実が欠落していると判定。コードで自明なため実害は限定的

### 関連性（answer_relevancy）— 回答が質問に関連しているか

| | ベンチマーク OK | ベンチマーク NG |
|---|---|---|
| **再検証 OK** | 79 件（両者合意・問題なし） | **14 件（誤検知）** |
| **再検証 NG** | 0 件（見逃しなし） | 9 件（要・実害確認） |

**誤検知の原因**: nabledge-6 の回答末尾には `参照: libraries-xxx.json:s9` という形式の出典メタデータが付与される。ベンチマークはこれを「質問と無関係なコンテンツ」と解釈してスコアを下げる。再検証では回答の本文が質問に直接答えており問題なしと判断した。誤検知 14 件中 13 件がこのパターンに該当する。

**誤検知の例:**
- [run-1-impact-01.md](checks/run-1-impact-01.md) — 本文は質問に直接関連しているが、末尾の `参照: libraries-database.json:s29, libraries-universal-dao.json:s20` を「無関係コンテンツ」として減点し NG（スコア 0.92）
- [run-1-qa-03.md](checks/run-1-qa-03.md) — 同上

**要・実害確認の例（9件）:**
- [run-1-pre-01.md](checks/run-1-pre-01.md) — 「起動コマンドと -requestPath の書き方」を問う質問に対し、DataReader や DispatchHandler の内部動作フローを含めており、質問の範囲を超えた内容が混入。質問と同じ機能領域の補足情報であり実害は限定的
- [run-2-pre-03.md](checks/run-2-pre-03.md) — 「検索方法」の質問に対し、注意点として「主キー以外での更新・削除はJDBCラッパーを使う」という更新・削除の制限を記載。同機能の関連情報であり実害は限定的

### 忠実性（faithfulness）— 回答がナレッジと矛盾しないか

| | ベンチマーク OK | ベンチマーク NG |
|---|---|---|
| **再検証 OK** | 66 件（両者合意・問題なし） | **33 件（誤検知）** |
| **再検証 NG** | 1 件（**見逃し**） | 2 件（要・実害確認） |

**誤検知の原因**: ベンチマークは「ナレッジに記載された情報を回答が省略している」ことを矛盾と解釈する。しかし省略は矛盾ではない。再検証では「回答に含まれる記述がナレッジと矛盾しているか」のみを基準としたため、判定が分かれた。

**誤検知の例:**
- [run-1-impact-01.md](checks/run-1-impact-01.md) — `UniversalDao.Transaction` コンストラクタが `SimpleDbTransactionManager` オブジェクトも受け付けることをナレッジが記述しているが、回答はコンポーネント名のみ説明。省略としてNG（スコア 0.83）だが、記述の矛盾はない

**見逃しの例:**
- [run-2-qa-11.md](checks/run-2-qa-11.md) — 回答の「上記以外の例外・エラー → FATAL, 500」が `ThreadDeath` / `VirtualMachineError` を包含しているが、ナレッジではこれらは「何もせず上位ハンドラに委譲」と明記されており矛盾。ベンチマークはスコア 1.0 で見逃し

**要・実害確認の例（2件）:**
- [run-2-impact-06.md](checks/run-2-impact-06.md) — 「認証情報の保持」推奨ストアをナレッジ（DBストア/HTTPセッションストア）と異なる「DBストア/Redisストア」と回答。ナレッジに存在しない記述。補足情報の誤記であり実害は限定的
- [run-3-qa-11.md](checks/run-3-qa-11.md) — `ThreadDeath` と `VirtualMachineError` のログレベルをナレッジでは INFO と FATAL と区別しているが、回答は同一扱いで記述。通常アプリで遭遇しないエッジケースであり実害は限定的
