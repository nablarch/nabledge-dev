## 🔬 仮説と改善提案

### 仮説 1: `full-text-search.sh` 修正で QA 時間短縮

- 証拠: 複数エージェントが route 1 で 0 hits、route 2 フォールバックに追加 60-80 秒消費。
- 検証方法: `scripts/full-text-search.sh` の jq 式を `(.sections[] | .content // "") | tostring | test(...)` に修正し、再計測。
- 予測: QA 平均時間 -10-20% (66 秒 → 55-60 秒)、qa-002 / qa-004 の欠落キーワードが関連 section の再ヒットで +1-2 項目改善する可能性。

### 仮説 2 (訂正): qa-004 欠落は知識ファイル側ではなく参照ファイル選択ミス

- 証拠: `libraries-tag.json#s21` に `<n:form useToken="true">` + `<n:submit allowDoubleSubmission="false">` が両方存在。前回エージェントは参照していた。今回は `handlers-use-token.json#s3` のみ参照。
- 検証方法: 仮説 1 の `full-text-search.sh` 修正が最優先。修正後は `useToken` キーワード検索で `libraries-tag.json` もヒット対象になる。
- 予測: 仮説 1 実施で qa-004 4/8 → 8/8、QA 検出率 85.0% → 95.0%+。

### 仮説 3: ca-001 Overview は code-analysis workflow の英名併記ルールで解消

- 証拠: Overview は意図的に日本語で要約。Class Summary / Nablarch Usage には英名が正しく出ている。
- 検証方法: `.claude/skills/nabledge-6/workflows/code-analysis.md` の Overview テンプレートに「主要 Nablarch コンポーネントは英名を併記する」制約を追加、再計測。
- 予測: ca-001 35/37 → 37/37、CA 検出率 98.1% → 99.1%。

### 仮説 4: qa-001 trial 間揺らぎは検出期待値を OR 化すれば安定

- 証拠: trial 1/3 は 100%、trial 2 のみ 62.5% で `withNoneOption` などが欠落。
- 検証方法: `scenarios.json` の qa-001 expectations を `withNoneOption` OR `noneOption` OR「選択なし」のように緩和し、再計測。
- 予測: qa-001 benchmark mean 87.5% → 95%+、SD 10% 前後に収束。
