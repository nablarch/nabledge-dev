## 🔬 仮説と改善提案

### 提案 1: grader の case-insensitive マッチ化 (qa-002 `pageNumber` 類の救済)
- 根拠: qa-002 で AI は `searchCondition.getPageNumber()` と記述、期待値は `pageNumber`。case sensitive substring マッチで落ちるが意味的には正答。
- 対応: scripts/grade.py の QA マッチを case-insensitive にする (一般化しすぎる場合はフィールド単位のオプション)。
- 期待効果: qa-002 は 100% 再現、他シナリオへの副作用は要確認。

### 提案 2: qa-001 シナリオの問い/期待値整合性見直し
- 根拠: 問い「コード値のプルダウン」に対し期待値に汎用 `n:select` 系リテラル (`listName`/`element*Property`) が混ざっており、AI が codeSelect に集中すると落とす構造。
- 対応: qa-001 を「コード値専用プルダウン (codeSelect)」と「汎用プルダウン (n:select)」の 2 シナリオに分割、期待値もそれぞれ再編。
- 期待効果: codeSelect 側は安定して 100%、select 側は独立指標として評価可能。

### 提案 3: code-analysis workflow の Overview / class-diagram の粒度強化
- 根拠: ca-001 Overview で `SessionUtil` 欠落、class-diagram / Component Summary で `ProjectDto` 欠落。ca-003 は 3 試行固定で Overview `BusinessDateUtil` 欠落。いずれも本文には存在するが要約層で抽象化されて落ちる。
- 対応: v6 skill の code-analysis workflow テンプレートに「Overview では Nablarch コンポーネント名を抽象化せず具体名を含める」「class-diagram は本文に登場する全 DTO/Entity を描画」を追加。
- 期待効果: ca-001 37/37、ca-003 37/37。

### 提案 4: 以降の nabledge-test は skill workflow 変更時のみ再取得
- 根拠: 本 baseline で RBKC 側変更は v6 knowledge/docs に影響しないこと (22-B-12 前後で v6 は byte identical、C6) と、精度差はすべて生成ゆらぎに帰着することを確認済。
- 対応: RBKC 変更は 5-version verify + 生成物 diff で副作用ゼロ確認をもって合格、nabledge-test は skill workflow/template を触ったときのみ再取得。
- 期待効果: RBKC 修正と skill 精度改善の評価軸を分離し、cat-and-mouse を避ける。

### 重要な前提: AI 生成ゆらぎを含めた厳密判定は不可能
- 3 試行ベンチマーク + CI 判定で統計的に有意な劣化がないことは示せるが、1 試行広域の小数点項目差異を 100% 再現性で管理する方法は存在しない。
- したがって本 baseline (20260424-172654) は合格とみなし、以降はこれを固定点とした相対比較で運用する。
