## 観察 3: 列構成はバージョンごとに異なる

- v1.2/v1.3 detail.xls: 10 列固定
  `№ / 分類 / タイトル / 変更概要 / 変更区分 / 変更実施バージョン / 業務アプリへの影響 / 影響ありの場合の内容と対処 / 参照先 / 備考`
- v1.4 以降 xlsx 本体: 14 列前後
  `No. / 分類 / リリース区分 / タイトル / 概要 / 修正後のバージョン / 不具合の起因バージョン / (モジュール副列) / システムへの影響の可能性 / システムへの影響の可能性の内容と対処 / 参照先 / JIRA issue`
- v5/v6 はほぼ同じ列。v1.4 もほぼ同じ列。v1.2/v1.3 detail.xls だけ 1 列少ない古い列名。

## 観察 4: ヘッダ行位置はファイルごとに異なる

- v6 releasenote 本体: row 4 がヘッダ + row 5-6 副列
- v1.2/v1.3 detail.xls: row 3 がヘッダ、副列なし
- セキュリティ対応表 `2.チェックリスト`: row 7 がヘッダ + row 8 副列

→ ヘッダ位置を固定値にはできない。「連続 3 セル以上非空」かつ「直下にデータ行が続く」行を動的に検出する必要がある。

## 観察 5: 「分類」シートと「バージョンアップ手順」シートの扱い

- `分類` シート (28 出現): 71 行 × 2 列、用語 → 分類の対応表。これ自体は小さな表で、**P1 として扱って 1 行 = 1 section** で良い。
- `バージョンアップ手順` シート (49 出現): 短い段落 + 数行の手順。**P2 として扱って 1 section = シート全体** で良い。

## 観察 6: ドキュメント内「移行対応方法」系シートは独立したページにしたい

`NumberRangeの対応方法`, `リポジトリを...と同じ動作にする方法`, `汎用データフォーマットXXE脆弱性`, `HIDDENストア脆弱性` 等、**独立した解説ページ**として扱いたいシートが複数ある。1 シート = 1 ファイルで出せば独立 MD として検索可能。

## 観察 7: セキュリティ対応表 v5/v6 は同一構造

v5 と v6 の `Nablarch機能のセキュリティ対応表.xlsx` は sheet 名と構造がほぼ同じ (`改訂履歴` / `1.概要` / `2.チェックリスト` / `3.PCIDSS対応表`)。同じ converter ロジックで対応可。

## 最終設計案

### シート分類 (2 種)

- **P1 (データ表)**: ヘッダ行 + データ行がある。1 行 = 1 section、docs MD は MD テーブル
- **P2 (段落)**: ヘッダ行なし、または単純な前書き。1 section = シート全体、docs MD は段落のまま

判定ロジック: **row 1-10 の範囲で「非空セル 3 個以上の行 + 直下にデータ行 (同構造 2 行以上)」があれば P1、なければ P2**。

### ファイル分割

1 `.xlsx` → 各シートごとに 1 `.json` + 1 `.md`。

ファイル命名: `{基底名}-{sheet_slug}.json`。シート名は日本語のまま (既に知識ファイルで多数使用、問題なし)。

例:
- `nablarch6-releasenote.xlsx` → `releases/releases/releases-nablarch6-releasenote-6.json`, `-バージョンアップ手順.json`, `-モジュールバージョン一覧.json`
- `Nablarch機能のセキュリティ対応表.xlsx` → `check/security-check/security-check-改訂履歴.json`, `-1.概要.json`, `-2.チェックリスト.json`, `-3.PCIDSS対応表.json`

### JSON スキーマ (P1)

- `title`: row 1 の `■...` (なければシート名)
- `content`: 空文字、またはタイトル直下の前書き段落
- `sections`: 1 データ行 = 1 section
  - `id`: 自動付番 (`row-1`, `row-2`, ...)
  - `title`: 「タイトル」列の値。列がなければ先頭非 No. 列の値。shift fallback あり
  - `content`: 全列を `{列名}: {値}` で改行区切り列挙

### JSON スキーマ (P2)

- `title`: row 1 の `■...` (なければシート名)
- `content`: シート全体のテキスト (現行 xlsx_releasenote の動作に近い)
- `sections`: 空

### docs MD (P1)

```markdown
# {title}

{前書き段落があれば}

| 列1 | 列2 | ... |
|---|---|---|
| 行1値1 | 行1値2 | ... |
| 行2値1 | 行2値2 | ... |
```

### docs MD (P2)

```markdown
# {title}

{全体テキスト}
```

### verify

- QC1-4 は既存の Excel 処理 (sequential-delete) を新 JSON 粒度で再適用
- QO2 は MD 側に「テーブル罫線 `|---|`」「列ヘッダ行」が入るため、**JSON の section content の全テキストが MD に含まれる方向のみチェック**に緩和 (Excel 系のみ)
- spec §3-3 QO2 に Excel 例外条項を追加

### index.toon エントリ増加

v5/v6 セキュリティ: 1 → 4 (約 +3)
リリースノート全部: 約 76 → 約 212 (約 +136)
→ index.toon は検索時にメモリに載るだけなので問題なし

### シート別 mapping は最小限

「P1/P2 の自動判定」+ 「タイトル列の自動検出 (列名が『タイトル』 or 先頭非 No. 列)」で大半が対応可能。例外が出たら converter 設定ファイルで override (初期版では mapping ファイル不要、必要になったら追加)。
