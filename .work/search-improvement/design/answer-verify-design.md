# 回答生成＋根拠検証 設計書

**Status**: Draft
**Date**: 2026-05-14

## 目的

意味検索で取得したセクション内容に基づき、正確な回答を生成し、ハルシネーションがないことを検証する。

## 方式

### 概要

```
セクション内容 → answer.md: 回答生成 → verify.md: 根拠検証 → 最終回答
```

回答生成と根拠検証を分離し、2ステップで品質を担保する。

### qa.md オーケストレーション

```
入力: 質問文, ヒアリング回答（あれば）

Step 1: semantic-search.md → pointer JSON
Step 2: read-sections.sh → セクション内容（high優先、最大10件）。以降 sections_content として保持
Step 3: qa/answer.md → 回答テキスト
Step 4: qa/verify.md → 検証結果（verify_result として保持）。PASS/FAILに関わらず常に結果を記録
Step 5: 分岐
  - PASS → Step 3の回答をそのまま出力
  - FAIL → verify_resultのissuesを除外指示に追加し、Step 2のsections_contentを再利用してStep 3を再実行（最大1回）→ 再生成回答を出力
Step 6: 最終回答を出力
```

### ベンチマーク統合

ベンチマークランナー（run.py）がqa.mdの各ステップの出力をキャプチャする。answer.md/verify.mdの追加に伴い、以下を追加キャプチャする。

| ファイル | 内容 | タイミング |
|----------|------|-----------|
| answer.md | 最終回答テキスト | Step 3完了後（FAIL再生成時は再生成後の回答） |
| verify.json | verify.md出力（result + claims + issues） | Step 4完了後。PASS/FAILに関わらず常に記録 |

verify.jsonは診断情報として、改善サイクルの原因切り分けに使う。FAIL時のissues（除外された主張リスト）は「answer.mdのプロンプトが知識外の情報をどの程度補完するか」の傾向分析に使える。

## answer.md（回答生成）

部品: `tools/benchmark/prompts/answer.md`（トレース付きJSON出力）
スキルワークフロー: `workflows/qa/answer.md`（fork元で作成）

### 入出力

```
入力:
  - question: 質問文
  - hearing_answer: ヒアリング回答（あれば）
  - sections_content: read-sections.shの出力（セクション本文）

出力: JSON
  {
    "answer": "日本語の回答テキスト（下記フォーマット）",
    "trace": {
      "intent": "ユーザーの意図（1文）",
      "key_sections": ["使用したセクションID"],
      "approach": "回答の構成方針"
    }
  }
```

### 処理手順

```
1. セクション内容を全て読む
2. 質問とヒアリング回答からユーザーの意図を特定する
3. セクション内容の中から、質問に直接回答する情報を特定する
4. 以下のフォーマットで日本語回答を生成する
```

### 出力フォーマット

```
**結論**: [質問への直接的な回答]

**根拠**: [コード例・設定例・仕様情報]

**注意点**: [制約事項・制限・よくある落とし穴]

参照: file.json:sN, file.json:sN
```

### 各セクションのガイドライン

**結論:**
- 質問への直接的な回答を1〜2文で述べる
- 質問をオウム返ししない
- 具体的なメソッド名・クラス名・アプローチを含める
- 例: "UniversalDao.deferメソッドを使用して遅延ロードします。検索結果を1件ずつストリーミング処理できます。"

**根拠:**
- 結論の裏付けとなるコード例・設定例・仕様情報をセクション内容から抽出する
- コード例・設定例はコードブロックで表示する
- 優先順位: 実装例 > 設定例 > API仕様 > 概念説明
- 複数セクションの情報を使う場合、実装の流れに沿って整理する
- セクション内容のコード例はそのまま引用する（改変しない）

**注意点:**
- 制約事項、リソース管理、よくある間違いがある場合に記載する
- 該当がなければ省略する（無理に書かない）
- 例: "DeferredEntityListは必ずcloseしてください。closeしないとDBコネクションがリークします。"

**参照:**
- 回答で実際に引用したセクションのみをリストする
- `file.json:sN` 形式（例: `libraries-universal-dao.json:s9`）
- カテゴリパスは省略する（`component/libraries/` は不要）

### 回答ルール

- 知識セクションの情報のみに基づく
- 知識セクションにない情報を推測で補わない
- 一般的なJava/プログラミング知識（try-catch、Bean、getter/setter等）は使用可
- 500トークン以内（複雑な質問は800トークンまで）

### エッジケース

**複数のアプローチがセクション内に存在する場合:**
- ヒアリング回答の処理方式（processing_type）に合うアプローチを選ぶ
- 処理方式で絞れない場合、最も一般的なアプローチを結論で述べ、他を注意点で言及する

**セクション内容だけでは質問に完全に回答できない場合:**
- セクション内容から回答できる範囲で回答する
- 不足部分は「この情報は知識ファイルの対象範囲外です」と明示する
- 推測で補完しない

**検索結果が空（pointer JSON results: []）の場合:**
- qa.mdがanswer.mdを呼ばずに直接「この情報は知識ファイルに含まれていません」を返す
- answer.mdは検索結果がある場合のみ呼ばれる

## verify.md（根拠検証）

部品: `tools/benchmark/prompts/verify.md`
スキルワークフロー: `workflows/qa/verify.md`（fork元で作成）

### 入出力

```
入力:
  - answer: answer.mdの出力テキスト
  - sections_content: read-sections.shの出力（Step 2と同一データ）

出力: JSON
  {
    "result": "PASS" | "FAIL",
    "claims": [
      {
        "claim": "抽出した主張テキスト",
        "supported": true | false,
        "evidence": "裏付けのセクション参照 or 'general_knowledge'"
      }
    ],
    "issues": [
      {
        "claim": "裏付けのない主張",
        "quote": "回答中の該当箇所の引用"
      }
    ]
  }
```

### 処理手順

```
1. 回答テキストからNablarch固有の主張を全て抽出する
2. 各主張について、セクション内容で裏付けられるか検証する
3. 裏付けのない主張が1つでもあれば FAIL、なければ PASS
4. 結果をJSON形式で出力する
```

### 主張の抽出基準

**抽出対象（Nablarch固有の主張）:**

| カテゴリ | 例 |
|----------|-----|
| API名 | "UniversalDao.deferメソッド", "@InjectForm アノテーション" |
| クラス名 | "DatabaseRecordReader", "BatchAction" |
| 設定方法 | "web-component-configuration.xmlに設定", "コンポーネント定義ファイルに記述" |
| 動作仕様 | "遅延ロードはDB接続をストリーミングする", "バリデーションエラー時にステータスコード400を返す" |
| 制約事項 | "closeしないとリソースリーク", "Formのプロパティは全てString型" |
| パラメータ | "-requestPathで指定", "SQLID" |

**抽出対象外（一般知識）:**

| カテゴリ | 例 |
|----------|-----|
| Java一般 | "Beanクラスを作成する", "try-with-resourcesを使う" |
| プログラミング一般 | "バリデーションを実行する", "エラーメッセージを表示する" |
| フロー記述 | "まず〜して、次に〜する" |
| 一般的なWeb概念 | "HTTPリクエスト", "JSONレスポンス" |

### 検証基準

各主張について、以下の順で判定する:

```
1. セクション内容に直接記載がある → OK
   例: セクションに "UniversalDao#defer" → 回答に "UniversalDao.deferメソッド" → OK

2. セクション内容の直接的な言い換え（パラフレーズ・略称・同義表現）→ OK
   例: セクションに "ページング機能" → 回答に "ページング検索が可能" → OK
   例: セクションに "DatabaseRecordReaderでデータを読み込む" → 回答に "DatabaseRecordReaderを使用する" → OK

3. セクション内容に明示されていない属性・動作・制約 → NG（推論の妥当性に関わらず）
   例: セクションに DatabaseRecordReader の機能説明 → 回答に "スレッドセーフ" → NG（明示なし）
   例: セクションにキャッシュの記述なし → 回答に "キャッシュ機能で高速化" → NG
   例: セクションに該当メソッドなし → 回答に "setTimeoutメソッドでタイムアウトを設定" → NG
```

**境界ルール**: 推論が有効なのは、セクション内容の直接的な言い換え（同じ情報の別表現）のみ。セクションに明示されていない属性・動作・制約は、たとえ技術的に妥当であっても、セクション内容に根拠がないためNGとする。

### FAIL時の再生成フロー

```
verify.md → FAIL (issues: [{claim: "xxx", quote: "yyy"}, ...])
  ↓
qa.md: answer.mdを再実行
  入力:
    - question: 同一
    - hearing_answer: 同一
    - sections_content: Step 2で取得済みの同一データ（read-sections.shを再実行しない）
  プロンプトに追記:
    「以下の主張は知識セクションに裏付けがないため含めないこと: {issues}」
  ↓
再生成された回答を最終出力とする（再検証はしない）
```

再検証しない理由: 再生成時に問題箇所を明示的に除外指示しているため、同じ問題は発生しない。検証ループは無限ループリスクがありコストも増加する。

## ベンチマークとの関係

| 観点 | プロダクション (verify.md) | ベンチマーク (hallucination-judge.md) |
|------|--------------------------|--------------------------------------|
| 検証対象 | 実際に読んだセクション | シナリオ定義のゴールドスタンダード（must+acceptable） |
| 目的 | 回答品質の即時改善 | システム全体の品質測定 |
| 独立性 | 検索結果に依存（実用上問題なし） | 検索結果から独立（測定の正確性重視） |

verify.mdとhallucination-judge.mdは目的が異なる。verify.mdは「読んだセクションに基づいているか」、hallucination-judge.mdは「正解セクションに基づいているか」を判定する。

## 根拠

### 回答フォーマットの維持

現行の結論/根拠/注意点/参照フォーマットは、ユーザーが必要な情報に素早くアクセスできる構造。変更する理由がないため維持する。

### verify.mdの必要性

現行は構造的防止（知識のみルール + 引用必須）のみでハルシネーション対策としている。しかしLLMは指示に反して学習データから情報を補完する傾向がある。事後検証により、構造的防止をすり抜けたケースを捕捉する。

### 再生成を最大1回に制限する理由

- 1回の再生成で問題箇所を除外指示すれば解消する
- 複数回ループはコスト・レイテンシ増大のリスクがある
- ベンチマークで再生成後もFAILするケースが頻発する場合、answer.mdのプロンプト改善で対応する（verify.mdの緩和ではなく）
