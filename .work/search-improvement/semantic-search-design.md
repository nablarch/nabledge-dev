# 意味検索 設計書

**Status**: Draft
**Date**: 2026-05-11

## 目的

ユーザーの自然言語質問から、知識ファイルの関連セクションを正確に特定する。

## 設計根拠

### 調査結果からの制約

| 制約 | 根拠 |
|------|------|
| 検索単位はセクション | ベンチマーク24件のmust-sectionは全て特定セクション。ファイル単位では粗すぎる |
| index.mdにセクション情報が必要 | 現index.toonはファイルレベルのみ。セクション選定にはタイトル+文脈が必要 |
| セクションタイトルにファイルタイトルが文脈を提供 | 14.5%のセクションが汎用タイトル("概要", "使用方法")。ファイル文脈なしでは特定不可 |

### 調査結果: タイトルの十分性

ベンチマーク24件（must-section 35件）に対してシミュレーション実施:

- セクションタイトルと質問の文字レベル重複: **97.1%** (34/35)
- セクション本文を含めた文字レベル重複: **100%** (35/35)
- 唯一のタイトル不一致: "DBからデータを読み込んで集計し…定期処理" vs "Nablarchバッチアプリケーションで使用するアクション" → AIの意味理解で十分カバー可能

結論: **index.mdのタイトル情報だけでAIは十分な精度でセクション選定できる**。

### 現行方式の問題

1. index.toonにセクション情報がない → ファイル選定後に全セクションを読む必要あり
2. section judgmentは最大20セクションまで内容を読む → コスト高
3. full-text-searchフォールバック時にindex.toonからのファイル選定が不正確

### 棄却した案

**annotate-index.sh（キーワード事前フィルタ）**

当初、AIがキーワード抽出 → スクリプトが★マーク → AIが★を優先的に確認、の3ステップを検討。
棄却理由:
- index.md全体が~19Kトークン（v6）で、AIが直接スキャン可能なサイズ
- キーワード抽出の品質がボトルネック（日本語の分かち書き問題）
- 実装複雑性に対する精度向上が見合わない（タイトルだけで97%マッチ）
- ツール呼び出し1回追加 = レイテンシ増加

ベンチマーク結果でタイトルマッチの不足が判明した場合に再検討。

## 方式

### 概要

```
質問 → AIがindex.mdを読みセクション選定 → セクションIDリスト
```

1ステップ: AIによる意味的選定のみ。

### ワークフロー (semantic-search.md)

```
入力: 質問文

Step 1: index.mdを読む
  - Read knowledge/index.md

Step 2: セクション選定（AI）
  - index.mdと質問を照合
  - 質問の意図を理解し、関連するセクションを選定
  - 選定基準:
    1. 質問に直接回答できるセクション (must)
    2. 回答に必要な前提知識セクション (nice-to-have)
  - 最大10件

出力: pointer JSON（下記スキーマ）
```

### 出力スキーマ

既存の`_knowledge-search.md`のpointer JSONスキーマと同一。下流のqa.md、read-sections.sh、ベンチマークランナーとの互換性を維持する。

```json
{
  "results": [
    {
      "file": "component/libraries/libraries-universal-dao.json",
      "section_id": "s9",
      "relevance": "high"
    },
    {
      "file": "component/libraries/libraries-database.json",
      "section_id": "s29",
      "relevance": "partial"
    }
  ]
}
```

| Field | Type | Description |
|---|---|---|
| file | string | knowledge/からの相対パス |
| section_id | string | セクションID（s1, s2, ... 形式） |
| relevance | "high" \| "partial" | high: 直接回答可能 / partial: 前提知識・関連情報 |

results はrelevance降順（high → partial）。空配列は一致なし。

### 最大10件の根拠

ベンチマーク24件のmust-section数: 平均1.46件/シナリオ、最大3件。acceptable含めても最大5件。
10件上限はベンチマーク実績の2倍以上のマージン。

10件で不足するケースへの対応:
- semantic-searchの出力が空（results: []）の場合、qa.mdが「知識ファイルに含まれていません」レスポンスを返す（既存仕様）
- 回答生成後のverify.md（根拠検証）でmust factが欠けていた場合、再検索を指示可能（将来実装）
- ベンチマーク §検証ポイント2 で10件上限がmust-section missを起こさないことを全シナリオで確認する（必須ゲート）

### 選定ガイドライン（semantic-search.mdに記載）

**選定すべきセクション:**
- 質問の操作対象に直接対応するセクション（例: "ページングしたい" → "ページングを行う"）
- 質問の技術的課題を解決するセクション（例: "メモリが足りない" → "遅延ロードする"）
- 質問のコンテキストに必要な前提知識（例: "REST APIで登録" → RESTfulウェブサービスの登録機能）

**選定しないセクション:**
- 一般的な概要のみで具体的な情報がないもの（"機能概要" で内容が空）
- モジュール一覧、改訂履歴など定型情報
- 質問と関連はあるが直接的でないもの（例: "ページング" の質問に対して "トランザクション管理"）

**カテゴリの活用:**
- component/libraries: 個別機能の詳細 → "〜の使い方" 系の質問
- component/handlers: ハンドラの設定と動作 → "〜ハンドラの設定" 系の質問
- processing-pattern: アプリケーション種別ごとの構成 → "〜アプリの作り方" 系の質問
- development-tools: テスト・ツール → "テストの書き方" 系の質問
- setup: 初期構築 → "プロジェクト作成" 系の質問

## index.md 設計

### フォーマット

```markdown
# Knowledge Index

## component/libraries

### ユニバーサルDAO
path: component/libraries/libraries-universal-dao.json
- s2: SQLを書かなくても単純なCRUDができる
- s3: 検索結果をBeanにマッピングできる
- s5: 使用方法
  - s6: 設定
  - s7: SQLファイルで検索する
  - s8: テーブルをJOINした検索結果を取得する
  - s9: 検索結果を遅延ロードする
  - s11: ページングを行う
  - s14: バッチ実行(一括登録、更新、削除)を行う
  - s15: 楽観的ロックを行う
  - s20: 現在のトランザクションとは異なるトランザクションで実行する

### データベースアクセス(JDBCラッパー)
path: component/libraries/libraries-database.json
- s1: 機能概要
  - s2: データベースの方言を意識することなく使用できる
  - s3: SQLはロジックではなくSQLファイルに記述する
  - s6: 実行時のBeanオブジェクトの状態を元にSQL文を動的に構築できる
- s9: 使用方法
  - s12: SQLをファイルで管理する
  - s13: SQLIDを指定してSQLを実行する
  - s21: $if構文を使用して条件に合ったSQL文を動的に構築する
  - s29: 別のトランザクションで実行する
```

### 構成ルール

1. **カテゴリごとにH2** — `## component/libraries`, `## processing-pattern/nablarch-batch` 等
2. **ファイルごとにH3** — ファイルタイトルをH3、直下にpath行
3. **L2セクションをリストアイテム** — `- sN: タイトル`
4. **L3セクションをインデント** — `  - sN: タイトル`（L2の下にネスト）
5. **L4以下は省略** — L3までで十分
6. **セクション0件のファイル** — H3 + path行のみ
7. **汎用タイトルはそのまま** — ファイルのH3が文脈を提供

### 省略ルール

以下のタイトルのセクションは省略する（検索対象にならない定型情報）：
- "モジュール一覧"
- "アプリケーションフレームワーク"（リリースノートのカテゴリヘッダ）
- "制約"（ハンドラの定型セクション、多くが空）
- "ハンドラクラス名"（定型情報）

releases カテゴリは検索対象としての価値が低いため、ファイルエントリのみ（セクション省略）。

### 実測サイズ（v6プロトタイプ）

| 指標 | 値 |
|------|-----|
| 総文字数 | 74,506 |
| 総行数 | 2,916 |
| ファイル数 | 353 |
| セクションエントリ数 | 1,914 |
| 推定トークン数 | ~19K |

全5バージョン推定: v5 ~25K, v1.4 ~21K, v1.3 ~14K, v1.2 ~14K。
全バージョンでコンテキストウィンドウの10%以下。

### Excel由来ファイル（check, releases）

Excel由来セクションはlevelフィールドなし。
- check: 全セクションをフラットリスト（セキュリティチェック項目は検索対象）
- releases: ファイルエントリのみ（セクション省略）

## RBKC実装

index.md生成は新規RBKCステップとして実装。

**入力**: knowledge/*.json（全知識ファイル）
**出力**: knowledge/index.md

**生成ロジック**:
1. knowledge/配下のJSONを再帰的に走査
2. ディレクトリ構造からカテゴリ（H2）を生成
3. 各JSONファイルからH3 + path + セクションリストを生成
4. 省略ルールを適用
5. ソート: カテゴリ内はファイル名の辞書順

**クロスバージョン**: 同一ロジックで全5バージョンに適用。バージョン固有の違い（カテゴリ構成）はディレクトリ構造から自動反映。

## 検証ポイント

1. **index.md のサイズ**: 実際にRBKCで生成して全5バージョンのトークン数を確認
2. **AI選定の精度**: シミュレーション — AIにindex.mdを渡してセクション選定させ、must-sectionとの一致率を測定
3. **レイテンシ**: index.md読み込み + AI選定の所要時間
