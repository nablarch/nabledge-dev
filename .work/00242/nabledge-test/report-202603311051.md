## 🔬 仮説と改善提案

### 仮説1: qa-001 `general_select_tag` は検出率を下げる期待値として不適切

**根拠**: 3試行中2試行で未検出。`n:codeSelect` に特化した質問に対し、スキルが `n:select` 比較セクションを含めるかどうかはコンテキスト依存。

**検証方法**: 質問文を「コード値プルダウン以外の一般的なプルダウン実装も含めて教えて」に変更して再実行し、安定して検出されるか確認。

**期待される結論**: `general_select_tag` の期待値をオプション（OR条件の弱い方）として扱うか、qa-001の質問自体を「n:codeSelectとn:selectの使い分け」に変更することで安定検出が可能になる。

---

### 仮説2: CA Overview の期待値はクラス名ではなく「概念」で設計するべき

**根拠**: ca-001 で SessionUtil（"セッションストア" として概念的に言及）、ca-002 で ZipCodeData（データクラスとして省略）が Overview から漏れた。Overviewセクションは高レベル要約を目的とし、全クラス名の列挙は行わない。

**検証方法**: ca-001/ca-002 の Overview 期待値を「主要な処理概念・パターンキーワード」に絞り込んで再評価。例: SessionUtil → "PRGパターン" または "セッション管理"。

**期待される結論**: Overview 期待値をアーキテクチャ概念（パターン名、主要フレームワーク機能名）に限定することで、より安定した検出率が得られる。

---

### 仮説3: CA Nablarch Usage 見出しは「主要クラスのみ」のパターンが安定

**根拠**: ca-001のBeanUtil、ca-002のDataReaderは本文言及あるが独立見出しなし。一方、ca-003はObjectMapper/BatchAction/FilePathSetting/BusinessDateUtil/DatabaseRecordReader全て独立見出しあり（100%達成）。差異は「そのクラスがbatch処理で主要な役割を果たすか」にある。

**検証方法**: 次回ベースラインでca-001 BeanUtil見出しの有無を3試行追跡し、パターンを確認。

**期待される結論**: BeanUtil/DataReader等のサポート役ユーティリティを Nablarch Usage 期待値から除外するか、検出ロジックを「見出し内に含む」から「セクション内テキストに含む」に緩和することで偽陰性を減らせる。
