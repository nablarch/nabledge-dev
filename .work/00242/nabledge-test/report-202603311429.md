## 💡 主要な発見

### QA: 90.9%の高検出率、CAに比べてパターンが安定

QAシナリオは5本中3本が100%を達成（qa-002, qa-003, qa-005）。qa-001（75%）とqa-004（80%）に残存ギャップがある。qa-001の`n:select`未検出はv1.3固有の問題ではなく、v1.3の知識ベースにn:selectが独立した説明として登場しないことによる構造的制約。qa-004の`@UseToken`/`UseToken`は、Nablarch 1.3の二重サブミット防止でn:formタグの`useToken`属性は存在するがアノテーション版(`@UseToken`)が少ない環境であることが影響していると推測される。

### CA: Overviewセクションの抽象化によるクラス名欠落が主要ギャップ

ca-001/ca-002ともに、Overviewセクションで具体的なNablarchクラス名（`OnDoubleSubmission`/`MailRequester`/`MessageSender` for ca-001, `ValidatableFileDataReader`/`BusinessDateUtil`/`UserInfoTempEntity` for ca-002）が言及されないパターンが一貫している。Overviewは高レベル要約を優先するため、補助クラスや機能クラスが省略される。また、`getValidatorAction`メソッドはProcessing FlowとSequence Diagramで一貫して未出力（フレームワーク設定メソッドとして処理フロー記述対象外とみなされる）。

### Nablarch Framework Usageの見出しレベル検出ギャップ

`DbAccessSupport`（ca-001）と`BusinessDateUtil`/`TransactionAbnormalEnd`（ca-002）はNablarch Framework Usageセクションに独立した`###`見出しを持たない。これらは他のクラスの説明の中で言及されるにとどまり、スキルがNablarch Usageの主役として扱うクラスの選択基準が検出期待値と一致していない。

## 🔬 仮説と改善提案

### 仮説1: CAのOverviewプロンプトに「主要Nablarchクラスを列挙する」指示を追加することで検出率が改善する

**根拠**: ca-001/ca-002で一貫してOverviewでのNablarchクラス未言及が発生。現状のOverviewは機能の概要説明を優先し、使用クラスの網羅より「何をするか」の説明に注力している。

**検証方法**: Overviewセクションの生成プロンプトに「主要なNablarchクラス（継承親・使用するフレームワーク機能）をOverviewに明示する」という指示を追加し、再実行して検出率を比較する。

**期待効果**: ca-001 Overview検出 2/5 → 5/5、ca-002 Overview検出 2/5 → 4-5/5 程度の改善が期待できる。

### 仮説2: Nablarch Framework Usageの対象クラス選定基準をスキルプロンプトで明示すると、DbAccessSupport/BusinessDateUtilの見出しが追加される

**根拠**: `DbAccessSupport`（ca-001）と`BusinessDateUtil`（ca-002）は本コードの重要依存クラスだが、現在は他クラスの説明に埋め込まれている。スキルがNablarch Usageの対象クラスを判断するロジックが「アクションメソッドで直接呼ばれるか」に依存している可能性がある。

**検証方法**: Nablarch Framework Usageの対象クラスの選定基準（継承元クラスも対象とする、dependency graphに現れるNablarchクラスを全て対象とする等）をプロンプトに明記して再実行。

**期待効果**: ca-001 Nablarch Usage 4/5 → 5/5、ca-002 Nablarch Usage 3/5 → 4-5/5 程度の改善が期待できる。
