## 経緯と今後の方針

### 何が起きたか

Stage 1 の facet 抽出プロンプトを「カテゴリ名 + 軽い説明だけ」で AI に渡し、質問から 24 カテゴリを選ばせる設計だった。**カテゴリの中身（どのファイルが含まれるか）を AI に見せていなかった**ため、「セッション = libraries」「エラー応答 = handlers」のような連想が原理的に難しい状態で測定していた。30件ベースラインは出たが、情報条件の欠陥ベースで数字は信頼できない。

エキスパートレビューも AI だったため、レビュアー自身がカテゴリの中身を知っている状態で「モデルも分かるはず」と投影し、情報条件の欠陥を見逃した。

### 方針

1. **使えるものは使う**: 捨てない、再利用の観点で仕分ける
2. **比較のやり方**: 新旧フローを **同一 30件** で計測し、**回答精度 / 実行時間 / コスト（モデル×トークン量）** で比較。更新するかしないかを数値で判断
3. **設計思考はもうできている**: 「質問から絞込条件をどう出すか、出すために機械的に用意できるものは何か」を **計測せずシミュレーション**（紙上＋ちょい試し）で詰めてから走らせる。ここが実質唯一ちゃんと考える部分
4. **残りは実装と計測**: 設計が決まれば作業は速い

## 資産の仕分け

| 資産 | 判断 |
|---|---|
| 模範回答 30件 | **残す**。ただし粒度は「質問のユーザー意図で必須か」視点で見直し |
| 30件ベースライン結果 | 参考記録として保存、数字は**不採用** |
| Stage 1 プロンプト（現行） | 破棄、情報設計からやり直し |
| Stage 2 filter スクリプト | 残す。アーキテクチャ次第で接続先を変えるだけ |
| Stage 2 スクリプト採点 | 残す。citation 抽出は再利用可能 |
| Stage 3 prompts (select / answer / judge) | 条件付きで残す。judge は模範回答ベースに改訂、select/answer はアーキテクチャ次第 |
| エキスパートレビュー記録 | 反省材料として残す |
