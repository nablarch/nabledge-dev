# RBKC verify 品質保証設計

## 1. 目的とスコープ

### 1-1. 目的

RBKC（RST/Markdown/Excel → JSON 知識ファイル変換）の出力品質を自動検証し、nabledge が正確な Nablarch 知識を提供することを保証する。

### 1-2. 検証対象

| ソースフォーマット | 資料 | 出力形式 |
|-----------------|------|---------|
| RST | Nablarch 解説書 | JSON + docs MD |
| Markdown | Nablarch システム開発ガイド | JSON + docs MD |
| Excel | セキュリティ対応表、リリースノート | JSON + docs MD |

### 1-3. スコープ外

RBKC の内部実装（converters、resolver、hints 等）の正しさ。verify はソースと出力だけを見る。

---

## 2. 品質基準

### 2-1. 品質水準

Nablarch は大規模金融システム等の重要インフラで使用されるミッションクリティカルなエンタープライズフレームワークである。nabledge はその知識を提供するため、同等の品質基準が求められる。

| 原則 | 内容 |
|------|------|
| ゼロトレランス | 1% のリスクも許容しない |
| 品質は二値 | 正しいか正しくないか。「十分に良い」は存在しない |
| verify は品質ゲート | verify が FAIL を報告したら修正は RBKC 側で行う。verify を弱めて出力をパスさせてはならない |
| 100% カバレッジ目標 | JSON にコンテンツトークンが欠落していれば RBKC のバグ |

### 2-2. verify の独立性原則

verify は RBKC 実装から完全に独立していなければならない。

- RBKC 実装モジュール（converters、resolver、hints、run 等）をインポートしない
- ロジックはソースフォーマット仕様（RST/Markdown/Excel）のみから導出する
- ソースと出力だけを見て正しさを判断する

## 3. 品質観点（MECE）

RBKC 変換で起こりうる全障害モードを、カテゴリ別に網羅的かつ重複なく整理する。

### カテゴリ 1: コンテンツ変換品質（ソース → JSON `content`）

| ID | 観点 | 障害モード |
|----|------|-----------|
| Q1 | **完全性** | ソースのコンテンツが JSON に欠落している |
| Q2 | **正確性** | ソースにないコンテンツが JSON に混入している |
| Q3 | **非重複性** | 同一コンテンツが JSON に重複して含まれている |
| Q4 | **配置正確性** | ソースのセクション A のコンテンツが JSON の異なるセクションに配置されている |
| Q5 | **形式純粋性** | RST/Markdown の構文記法（`:role:`、`.. directive::` 等）が JSON に残留している |

### カテゴリ 2: リンク・参照品質（ソース → JSON リンクフィールド）

| ID | 観点 | 障害モード |
|----|------|-----------|
| Q6 | **リンク完全性** | ソースのリンク・参照（`:ref:`、`.. figure::`、外部 URL 等）が JSON に欠落している |
| Q7 | **リンク先実在性** | リンク先のラベル・ファイルパスがリポジトリ内に実在しない |

### カテゴリ 3: 出力整合性品質（JSON ↔ docs MD）

| ID | 観点 | 障害モード |
|----|------|-----------|
| Q8 | **docs MD 整合性** | JSON を変換した docs MD が JSON の内容を正確に反映していない |

### カテゴリ 4: ファイル分類・構造品質

| ID | 観点 | 障害モード |
|----|------|-----------|
| Q9 | **ファイル分類正確性** | toctree のみで実コンテンツを持たないファイル（no_knowledge_content）の誤分類 |
| Q10 | **index.toon 網羅性** | 変換済み JSON が index.toon に未登録 |
| Q11 | **docs MD 存在確認** | JSON に対応する docs MD が存在しない |

## 4. 検証アプローチ

### 4-1. アプローチ A: 削除ベース検証（ソース ↔ JSON コンテンツ）

**設計原理**: JSON のテキストをソースから上から順に「削除」することで、コンテンツの完全性・正確性・配置を一度に検証する。

```
[ソースファイル] → 構文除去 → [コンテンツテキスト]
                                      ↑ 順次削除
[JSON content] → テキスト抽出 → [テキストリスト（上から順）]
```

**手順**:

| ステップ | 処理 | 検証観点 |
|---------|------|---------|
| Step 1 | JSON の各セクション `content` と `title` を上から順のテキストリストとして抽出する | — |
| Step 2 | ソースファイルをフォーマット固有の構文要素（ディレクティブ、ラベル、アンダーライン等）を除去してコンテンツテキストとして準備する | — |
| Step 3 | Step 1 のリストを上から順にソースから削除する。削除はソース上で上から下へ進む必要がある。期待位置より後ろにしか見つからなければセクション間の配置ミス → **FAIL** | **Q4** |
| Step 4 | 削除後のソース残渣をチェックする。残りがあれば変換されなかったコンテンツが存在する → **FAIL** | **Q1** |
| Step 5 | Step 3 でソースから削除できなかった JSON テキストをチェックする。存在すれば誤追加または重複 → **FAIL** | **Q2, Q3** |

### 4-2. アプローチ B: リンク・参照検証（ソース → JSON）

**設計原理**: ソースに出現する全リンク・参照を抽出し、(a) JSON への変換漏れ、(b) リンク先の実在性を確認する。

| リンク種別 | (a) リンク先実在性（Q7） | (b) JSON への包含（Q6） |
|-----------|----------------------|----------------------|
| `:ref:\`label\`` | 全 RST から収集したラベルマップに label が存在するか | 対応テキストが JSON に含まれるか |
| `.. figure::`、`.. image::`、`.. literalinclude::` | パスが実ファイルとして存在するか | ファイル名/テキストが JSON に含まれるか |
| `[text](path)`（Markdown） | パスが実ファイルとして存在するか | — |
| 外部 URL | — | URL が JSON に含まれるか |

### 4-3. アプローチ C: docs MD 整合性検証（JSON ↔ docs MD）

**設計原理**: docs MD は JSON の人間可読レンダリングであるため、両者は完全に一致しなければならない。JSON が FAIL の場合は docs MD も自動的に FAIL とする。

| チェック項目 | 検証内容 | 観点 |
|------------|---------|------|
| タイトル | JSON `title` == docs MD の `# ` 見出し | Q8 |
| セクションタイトル | JSON 各セクションのタイトルが docs MD の `##`/`###` に存在するか | Q8 |
| hints | JSON `hints` が docs MD の keywords ブロックに全て存在するか | Q8 |
| content 完全一致 | JSON 各セクションの `content` が docs MD に完全に含まれているか | Q8 |

## 7. verify の変更ルール

verify は品質ゲートであり、その変更は厳密に管理する。

### 変更プロセス

1. 変更内容と理由をユーザーに説明する
2. ユーザーの明示的な承認を取得する
3. TDD で実装する（RED → GREEN）

### 許容される変更 / 許容されない変更

| 許容される変更 | 許容されない変更 |
|--------------|--------------|
| verify 自体のロジックバグの修正（例: 正規表現の誤り、範囲の間違い） | RBKC の現在の出力に合わせて verify 基準を緩める |
| 仕様上必要なチェックの追加 | チェックを無効化・スキップする |
| 誤検知（false positive）の修正 | RBKC 実装の詳細に合わせた回避策の追加 |
