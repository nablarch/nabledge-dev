# RBKC verify 要件定義とアプローチ

## 1. 要件

### 1-1. 求められる品質レベル

Nablarchはミッションクリティカルなエンタープライズフレームワークであり、大規模金融システム等の重要インフラで使用されています。nabledgeはNablarchに関する知識を提供するため、同等の品質基準が求められます。

- 1%でもリスクがあれば排除する — 許容しない
- 「十分に良い」は存在しない。品質は二値: 正しいか正しくないか
- verifyはRBKC出力の品質ゲート。verifyがFAILを報告したら、修正はRBKC側で行う。verifyを弱めてRBKC出力をパスさせてはならない
- コンテンツカバレッジ100%が目標。JSONにコンテンツトークンが欠落していればRBKCのバグ

### 1-2. RBKCとの完全分離

- verifyはRBKC実装モジュール（converters、resolver、hints、run等）をインポートしてはならない
- verifyのロジックはソースフォーマット仕様（RST、Markdown、Excel）のみから導出できなければならない（RBKCの実装方法には依存しない）
- verifyはソースと出力だけを見て正しさを判断する。RBKCが内部でどう動くかを知る必要はない

## 2. アプローチ

### 2-1. 担保する品質観点

verifyが担保しようとしている品質観点を以下に整理する。各観点に対してアプローチがカバーできているかを示す。

| # | 観点 | 説明 | アプローチでのカバー |
|---|------|------|---------------------|
| Q1 | **コンテンツ完全変換** | ソースファイルのコンテンツがJSONに漏れなく含まれている | ✅ アプローチA Step 5（残渣チェック）でカバー |
| Q2 | **コンテンツ誤追加なし** | ソースファイルに存在しない情報をJSONに含んでいない | ✅ アプローチA Step 6（JSON残存チェック）でカバー |
| Q3 | **コンテンツ重複なし** | ソースファイルのコンテンツがJSONに重複して含まれていない | ✅ アプローチA Step 4（順次削除）でカバー |
| Q4 | **セクション配置正確性** | ソースのセクションAのコンテンツがJSONのセクションAに対応している（セクション間の配置ミスがない） | ✅ アプローチA Step 4（順序チェック）でカバー |
| Q5 | **タイトルと内容の整合** | セクションタイトルがコンテンツ内容と整合している | 部分的（見出し存在確認のみ、内容整合は未実装） |
| Q6 | **リンク漏れなし** | 内部リンク（:ref:）・外部URL・アセット（figure等）がJSONに漏れなく含まれている | ✅ アプローチB Check C・Check D でカバー |
| Q7 | **リンク先正確性** | リンク先が実ファイル・実ラベルとして存在することを確認できる | ✅ アプローチB Check C でカバー |
| Q8 | **docs MD整合性** | docs MDの内容がJSONと完全一致している | ✅ アプローチC でカバー |

### 2-2. アプローチA: 削除ベース検証（ソース ↔ JSON）

ソースとJSONの間のコンテンツ整合性を検証するアルゴリズム:

1. **JSONからテキストを抽出する**: JSONの各セクションの `content` フィールドとトップレベルの `title` フィールドの文字列値を連結し、検証用テキストリスト（上から順に並んだテキスト断片のリスト）を作成する
2. **ソーステキストを用意する**: 対応するソースファイル（RST / Markdown / Excel）のテキストを上から順に読み込む
3. **ソースから構文要素を除去する**: RST/Markdown/Excelのフォーマット固有の構文要素（ディレクティブ、ラベル、アンダーライン等）を除去し、コンテンツテキストのみを抽出する
4. **JSONテキストを順番にソースから削除する**: Step 1で抽出したJSONテキストを上から順に取り出し、対応するテキストをソースから削除する。削除は**ソース上で上から順番に**行う。もし削除しようとしたテキストがソースの期待位置より後ろにしか見つからなければ、セクション間の配置ミスを示す → FAIL
5. **ソース残渣をチェックする（コンテンツ完全変換）**: 削除後にソースに残ったテキストがゼロかチェックする。残りがあれば、RBKCが変換しなかったコンテンツが存在する → FAIL（Q1）
6. **JSON残存テキストをチェックする（誤追加・重複）**: Step 4でソースから削除できなかったJSONテキストが存在する場合、そのテキストはソースに存在しない（誤追加）または重複している → FAIL（Q2, Q3）

このアプローチにより:
- Q1（コンテンツ完全変換）: ソースに残渣がなければ全コンテンツが変換されている
- Q2（誤追加なし）: ソースから削除できなかったJSONテキストは誤追加
- Q3（重複なし）: 同じテキストを2回削除しようとしても2回目は削除できない
- Q4（配置正確性）: 削除の順序がソース上で上から順になっていなければ配置ミス

### 2-3. アプローチB: リンク・参照の検証（ソース → JSON）

ソースに出現するリンクや参照が、JSONに存在し、かつリンク先が実在することを確認する:

- **内部リンク（:ref:\`label\`）**: ソース中の `:ref:` 参照を全て抽出し、(a) ラベルがグローバルラベルマップ（全RSTから収集）に存在するか確認（リンク先の実在確認）、(b) 対応するテキストがJSONに含まれているか確認（Q6, Q7）
- **アセット参照（.. figure::、.. image::、.. literalinclude::）**: ソース中のアセットパスを全て抽出し、(a) パスが実ファイルとして存在するか確認（リンク先の実在確認）、(b) 対応するテキストまたはファイル名がJSONに含まれているか確認（Q6, Q7）
- **Markdownリンク（[text](path)）**: ソース中のリンクパスを全て抽出し、パスが実ファイルとして存在するか確認（Q7）
- **外部URL**: ソースに出現する外部URLが全てJSONに含まれているか確認（Q6）

### 2-4. アプローチC: docs MD整合性検証（JSON ↔ docs MD）

docs MDはJSONを人間が読める形式に変換したものであるため、JSONの内容とdocs MDの内容が完全に一致することを確認する。JSONが正しい場合のみdocs MDの検証を行う（JSONがFAILの場合はdocs MDもFAILとする）:

- **タイトル一致**: JSONの `title` == docs MDの `# ` 見出し（Q8）
- **セクションタイトル一致**: JSONの各セクションタイトルがdocs MDの `## ` / `### ` に存在するか（Q8）
- **hints一致**: JSONの `hints` がdocs MDのkeywordsブロックに存在するか（Q8）
- **content完全一致**: JSONの各セクションの `content` がdocs MDに完全に含まれているか（Q8）

### 2-5. 検証の方向性

verifyはソースを起点に走査する（ソース駆動）:

- 全ソースファイルを走査し、対応するJSONが存在するか確認する
- JSONが存在しない（変換クラッシュ等）→ FAIL
- JSONが存在しても内容が不正 → FAIL

## 5. verifyの変更ルール

- verifyの変更には事前にユーザーの明示的な承認が必要
- 許容される変更: verifyのロジックバグ修正、仕様上必要なチェックの追加、誤検知（false positive）の修正
- 許容されない変更: RBKCの現在の出力に合わせてverify基準を緩める、チェックを無効化・スキップする、RBKC実装の詳細に合わせた回避策を追加する
