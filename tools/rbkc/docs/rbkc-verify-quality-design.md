# RBKC verify 品質保証設計

## 1. 目的とスコープ

### 1-1. 目的

RBKC（RST/Markdown/Excel → JSON 知識ファイル変換）の出力品質を自動検証し、nabledge が正確な Nablarch 知識を提供することを保証する。

### 1-2. 検証対象

| ソースフォーマット | 公式ドキュメント | 出力形式 |
|-----------------|----------------|---------|
| RST | Nablarch 解説書 | 知識ファイル（JSON）、閲覧用知識（docs MD） |
| Markdown | Nablarch システム開発ガイド | 知識ファイル（JSON）、閲覧用知識（docs MD） |
| Excel | セキュリティ対応表、リリースノート | 知識ファイル（JSON）、閲覧用知識（docs MD） |

### 1-3. スコープ外

RBKC の内部実装（converters、resolver、hints 等）の正しさ。verify はソースと出力だけを見る。独立性の詳細は 2-2 参照。

---

## 2. 品質基準

### 2-1. 品質水準

Nablarch は大規模金融システム等の重要インフラで使用されるミッションクリティカルなエンタープライズフレームワークである。nabledge はその知識を提供するため、同等の品質基準が求められる。

- **ゼロトレランス**: 1% のリスクも許容しない
- **品質は二値**: 正しいか正しくないか。「十分に良い」は存在しない
- **verify は品質ゲート**: verify が FAIL を報告したら修正は RBKC 側で行う。verify を弱めて出力をパスさせてはならない
- **100% カバレッジ必達**: JSON にコンテンツトークンが欠落していれば RBKC のバグ

### 2-2. verify の独立性原則

verify は RBKC 実装から完全に独立していなければならない。

- RBKC 実装モジュール（converters、resolver、hints、run 等）をインポートしない
- ロジックはソースフォーマット仕様（RST/Markdown/Excel）のみから導出する
- ソースと出力だけを見て正しさを判断する

## 3. 品質チェック

品質観点 ID は QC（コンテンツ）、QL（リンク）、QO（出力）+ 連番。各サブセクションは「品質観点と検証状況」→「検証方法」の順で記述する。

### 3-1. コンテンツ検証（QC1–QC5）

**品質観点と検証状況**:

| ID | 観点 | 定義 | 手順 | RST | MD | Excel |
|----|------|------|------|:---:|:---:|:----:|
| **QC1** | 完全性 | ソースのコンテンツが JSON に欠落している | 手順3 | ❌ | ❌ | ❌ |
| **QC2** | 正確性 | ソースにないコンテンツが JSON に混入している | 手順4 | ❌ | ❌ | ❌ |
| **QC3** | 非重複性 | 同一コンテンツが JSON に重複して含まれている | 手順4 | ❌ | ❌ | ❌ |
| **QC4** | 配置正確性 | ソースのセクション A のコンテンツが JSON の異なるセクションに配置されている | 手順2 | ❌ | ❌ | — |
| **QC5** | 形式純粋性 | フォーマット固有の構文記法が JSON に残留している（RST: `:role:`・`.. directive::` 等、MD: raw HTML・エスケープ文字等） | 独立チェック | ❌ | ❌ | — |

**検証方法（RST / MD）**: JSON のテキストをオリジナルのソースから上から順に「削除」し、削除後に残ったテキストがソース構文要素のみかどうかを確認することで、コンテンツの完全性・正確性・配置を一度に検証する。

```
[ソースファイル（原文のまま）]
        ↑ 順次削除（上から下）
[JSON content] → テキスト抽出 → [テキストリスト（上から順）]
        │
        ├─ 削除できなかった JSON テキスト → 誤追加/重複 → FAIL（QC2, QC3）
        │
        └─ 削除後のソース残存テキスト → 構文要素のみか？ → PASS / FAIL（QC1, QC4）
```

**手順**:

1. JSON の各セクション `content` と `title` を上から順のテキストリストとして抽出する。削除の単位はセクションの `content` フィールドおよび `title` フィールドの文字列全体（1フィールド = 1単位）。一致判定は空白・改行を正規化（連続する空白→単一スペース、行末空白除去）した上で行う。`hints` フィールドは QC1 スコープ外（QL1/QO1 で別途検証）
2. オリジナルのソースファイルを変更せず、手順1のリストを上から順にソースから削除する。削除はソース上で上から下へ進む必要がある。期待位置と異なる位置（前後いずれも）にしか見つからなければセクション間の配置ミス
3. 削除後の残りをチェックする。フォーマット固有の構文要素（ディレクティブ、ラベル、アンダーライン等）以外のコンテンツが残っていれば、変換されなかったコンテンツが存在する
4. 手順2でソースから削除できなかった JSON テキストをチェックする（JSON にあるがソースに存在しない = 誤追加、または削除済み箇所と重複）

**検証方法（QC5 形式純粋性）**: JSON の全 `content` フィールドと `title` フィールドを対象に、フォーマット固有の構文パターンが残留していないかを正規表現で検査する。

- **RST**: `:role:\`text\`` パターン、`.. directive::` パターン、`_label:` ラベル定義、見出しアンダーライン（`====`、`----` 等）
- **MD**: `<[a-zA-Z]` で始まる raw HTML タグ（`<details>`、`<summary>`、`<br>` 等）、`\*`・`\_` 等のバックスラッシュエスケープ

いずれかのパターンが検出された場合 → **FAIL**（QC5）

**検証方法（Excel）**: Excel はシート/行/セル構造であり上から下の順序保証がないため、削除ベース手順は適用しない。代わりに集合比較で検証する。ソースの全シートから全セル値を収集し、JSON の全コンテンツトークンと集合として突き合わせる。

- ソースにあって JSON にないセル値 → **FAIL**（QC1）
- JSON にあってソースにないトークン → **FAIL**（QC2）
- JSON 内に同一トークンが複数存在 → **FAIL**（QC3）

### 3-2. リンク検証（QL1–QL2）

**品質観点と検証状況**:

| ID | 観点 | 定義 | RST | MD | Excel |
|----|------|------|:---:|:---:|:----:|
| **QL1** | 内部リンクの正確性 | ソース内の内部リンク（`:ref:`、figure/image/literalinclude、`[text](path)` 等）が、意図したページ・セクションに正しく機能するよう JSON および docs MD に反映されていない | ❌ | ❌ | — |
| **QL2** | 外部リンクの一致 | 外部 URL が JSON および docs MD にソースと完全同一で含まれていない | ❌ | ❌ | — |

**検証方法**:

QL1（内部リンク）: ソースの内部リンクを抽出し、JSON と docs MD の両方でリンク先のページ・セクションまで正しく機能するかを確認する。リンク種別ごとの期待形式:

| リンク種別 | JSON での期待形式 | docs MD での期待形式 |
|-----------|----------------|-------------------|
| `:ref:\`label\`` | ラベルが指すセクションのタイトルが JSON に含まれているか | セクションへのアンカーリンクが正しく解決されているか |
| `.. figure::`、`.. image::` | キャプションまたはファイル名が JSON に含まれているか | 画像パスまたはキャプションが正しく含まれているか |
| `.. literalinclude::` | 参照先コードの内容が JSON に含まれているか（`:lines:`・`:start-after:`・`:end-before:` 等のオプションがある場合はその範囲のみを対象とする） | コードブロックとして正しく含まれているか |
| `[text](path)`（Markdown） | リンクテキストが JSON に含まれているか | href がリンク先ページ・セクションと一致しているか |

QL2（外部リンク）: URL の文字列がソースと完全一致しているかを確認する。

### 3-3. 出力検証（QO1–QO5）

**品質観点と検証状況**:

| ID | 観点 | 定義 | RST | MD | Excel |
|----|------|------|:---:|:---:|:----:|
| **QO1** | docs MD 整合性（タイトル・セクション・hints） | JSON を変換した docs MD が JSON の内容を正確に反映していない | ✅ | ✅ | ✅ |
| **QO2** | docs MD 存在確認 | JSON に対応する docs MD が存在しない | ✅ | ✅ | ✅ |
| **QO3** | 目次ページの除外 | 実コンテンツを持たない目次ページが知識ファイルとして誤って変換されている | ❌ | — | — |
| **QO4** | index.toon 網羅性 | 変換済み JSON が index.toon に未登録 | ✅ | ✅ | ✅ |
| **QO5** | docs MD 整合性（content 完全一致） | JSON 各セクションの content が docs MD に完全一致で含まれていない | ❌ | ❌ | ❌ |

**検証方法（docs MD）**: docs MD は JSON の人間可読レンダリングであるため、両者は完全に一致しなければならない。JSON が FAIL の場合は docs MD も自動的に FAIL とする。以下の要素を個別に検証する:

- **タイトル**: JSON `title` == docs MD の `# ` 見出し（QO1）
- **セクションタイトル**: JSON 各セクションのタイトルが docs MD の `##`/`###` に存在し、かつ JSON と同じ順序で並んでいるか（QO1）
- **hints**: JSON `hints` が docs MD の keywords ブロックに全て存在するか（QO1）
- **content 完全一致**: JSON 各セクションの `content` が docs MD に完全に含まれているか（QO5）

**検証方法（index.toon）**: 変換済みの全 JSON ファイルが index.toon に登録されているかを確認する。登録漏れがあると nabledge の検索から除外される。index.toon に存在する全エントリと出力ディレクトリの JSON ファイル一覧を突き合わせ、未登録ファイルを検出する。

## 5. verify の変更ルール

verify は品質ゲートであり、その変更は厳密に管理する。

### 変更プロセス

1. 変更内容と理由をユーザーに説明する
2. ユーザーの明示的な承認を取得する
3. TDD で実装する（RED → GREEN）

### 許容される変更

- verify 自体のロジックバグの修正（例: 正規表現の誤り、範囲の間違い）
- 仕様上必要なチェックの追加
- 誤検知（false positive）の修正

### 許容されない変更

- RBKC の現在の出力に合わせて verify 基準を緩める
- チェックを無効化・スキップする
- RBKC 実装の詳細に合わせた回避策を追加する
