# RBKC verify 品質保証設計

## 1. 目的とスコープ

### 1-1. 目的

RBKC（RST/Markdown/Excel → JSON 知識ファイル変換）の出力品質を自動検証し、nabledge が正確な Nablarch 知識を提供することを保証する。

### 1-2. 検証対象

| ソースフォーマット | 公式ドキュメント | 出力形式 |
|-----------------|----------------|---------|
| RST | Nablarch 解説書 | 知識ファイル（JSON）、閲覧用知識（docs MD） |
| Markdown | Nablarch システム開発ガイド | 知識ファイル（JSON）、閲覧用知識（docs MD） |
| Excel | セキュリティ対応表、リリースノート | 知識ファイル（JSON）、閲覧用知識（docs MD） |

### 1-3. スコープ外

RBKC の内部実装（converters、resolver、hints 等）の正しさ。verify はソースと出力だけを見る。独立性の詳細は 2-2 参照。

---

## 2. 品質基準

### 2-1. 品質水準

Nablarch は大規模金融システム等の重要インフラで使用されるミッションクリティカルなエンタープライズフレームワークである。nabledge はその知識を提供するため、同等の品質基準が求められる。

- **ゼロトレランス**: 1% のリスクも許容しない
- **品質は二値**: 正しいか正しくないか。「十分に良い」は存在しない
- **verify は品質ゲート**: verify が FAIL を報告したら修正は RBKC 側で行う。verify を弱めて出力をパスさせてはならない
- **100% カバレッジ目標**: JSON にコンテンツトークンが欠落していれば RBKC のバグ

### 2-2. verify の独立性原則

verify は RBKC 実装から完全に独立していなければならない。

- RBKC 実装モジュール（converters、resolver、hints、run 等）をインポートしない
- ロジックはソースフォーマット仕様（RST/Markdown/Excel）のみから導出する
- ソースと出力だけを見て正しさを判断する

## 3. 品質チェック

品質観点 ID は QC（コンテンツ）、QL（リンク）、QO（出力）+ 連番。各サブセクションは「品質観点と検証状況」→「検証方法の詳細」の順で記述する。

### 3-1. コンテンツ検証（QC1–QC5）

**品質観点と検証状況**:

| ID | 観点 | 定義 | 検証手順 | RST | MD | Excel | 影響度 |
|----|------|------|---------|:---:|:---:|:----:|:------:|
| **QC1** | 完全性 | ソースのコンテンツが JSON に欠落している | Step 3 | ❌ | ❌ | ❌ | **高** |
| **QC2** | 正確性 | ソースにないコンテンツが JSON に混入している | Step 4 | ❌ | ❌ | ❌ | **高** |
| **QC3** | 非重複性 | 同一コンテンツが JSON に重複して含まれている | Step 4 | ❌ | ❌ | ❌ | **中** |
| **QC4** | 配置正確性 | ソースのセクション A のコンテンツが JSON の異なるセクションに配置されている | Step 2 | ❌ | ❌ | ❌ | **高** |
| **QC5** | 形式純粋性 | RST 固有の構文記法（`:role:`、`.. directive::` 等）が JSON に残留している（RST のみ対象） | 独立チェック | ❌ | — | — | **中** |

**検証方法**: JSON のテキストをオリジナルのソースから上から順に「削除」し、削除後に残ったテキストがソース構文要素のみかどうかを確認することで、コンテンツの完全性・正確性・配置を一度に検証する。

```
[ソースファイル（原文のまま）]
        ↑ 順次削除（上から下）
[JSON content] → テキスト抽出 → [テキストリスト（上から順）]
        │
        ├─ 削除できなかった JSON テキスト → 誤追加/重複 → FAIL（QC2, QC3）
        │
        └─ 削除後のソース残存テキスト → 構文要素のみか？ → PASS / FAIL（QC1, QC4）
```

**手順**:

1. JSON の各セクション `content` と `title` を上から順のテキストリストとして抽出する。削除の単位はセクションの `content` フィールドおよび `title` フィールドの文字列全体（1フィールド = 1単位）
2. オリジナルのソースファイルを変更せず、Step 1 のリストを上から順にソースから削除する。削除はソース上で上から下へ進む必要がある。期待位置より後ろにしか見つからなければセクション間の配置ミス
3. 削除後の残りをチェックする。フォーマット固有の構文要素（ディレクティブ、ラベル、アンダーライン等）以外のコンテンツが残っていれば、変換されなかったコンテンツが存在する
4. Step 2 でソースから削除できなかった JSON テキストをチェックする（JSON にあるがソースに存在しない = 誤追加、または削除済み箇所と重複）

### 3-2. リンク検証（QL1–QL3）

**品質観点と検証状況**:

| ID | 観点 | 定義 | RST | MD | Excel | 影響度 |
|----|------|------|:---:|:---:|:----:|:------:|
| **QL1** | リンク完全性 | ソースのリンク・参照（`:ref:`、`.. figure::`、外部 URL 等）が JSON に欠落している | ✅ | ✅ | — | **高** |
| **QL2** | リンク先実在性 | リンク先のラベル・ファイルパスがリポジトリ内に実在しない | ✅ | ✅ | — | **中** |
| **QL3** | リンク先内容正確性 | リンク先のコンテンツ（キャプション、参照先テキスト等）が JSON に正しく反映されていない | ❌ | ❌ | — | **高** |

**検証方法**: ソースに出現する全リンク・参照を抽出し、(a) リンク先の実在性、(b) JSON への変換漏れ、(c) リンク先内容の正確性を確認する。

リンク種別ごとの検証範囲:

| リンク種別 | (a) リンク先実在性（QL2） | (b) JSON への包含（QL1） | (c) リンク先内容正確性（QL3） |
|-----------|----------------------|----------------------|--------------------------|
| `:ref:\`label\`` | 全 RST から収集したラベルマップに label が存在するか | 参照テキストが JSON に含まれるか | label が指すセクションのコンテンツが JSON に正しく反映されているか |
| `.. figure::`、`.. image::`、`.. literalinclude::` | パスが実ファイルとして存在するか | ファイル名/テキストが JSON に含まれるか | キャプション等の内容が JSON に正しく含まれているか |
| `[text](path)`（Markdown） | パスが実ファイルとして存在するか | — | — ※1 |
| 外部 URL | — | URL が JSON に含まれるか | — ※2 |

※1 Markdown リンクのリンク先コンテンツは verify スコープ外（外部ファイル内容まで追跡しない）  
※2 外部 URL のリンク先コンテンツは verify スコープ外（外部依存のため実在性・内容保証不可）

### 3-3. 出力検証（QO1–QO4）

**品質観点と検証状況**:

| ID | 観点 | 定義 | RST | MD | Excel | 影響度 |
|----|------|------|:---:|:---:|:----:|:------:|
| **QO1** | docs MD 整合性（タイトル・セクション・hints） | JSON を変換した docs MD が JSON の内容を正確に反映していない | ✅ | ✅ | ✅ | **高** |
| **QO1** | docs MD 整合性（content 完全一致） | 〃（content の全文一致） | ⚠️ 先頭50文字のみ | ⚠️ 先頭50文字のみ | ⚠️ 先頭50文字のみ | **高** |
| **QO2** | docs MD 存在確認 | JSON に対応する docs MD が存在しない | ✅ | ✅ | ✅ | **高** |
| **QO3** | ファイル分類正確性 | toctree のみで実コンテンツを持たないファイル（no_knowledge_content）の誤分類 | ❌ | — | — | **中** |
| **QO4** | index.toon 網羅性 | 変換済み JSON が index.toon に未登録 | ✅ | ✅ | ✅ | **高** |

**検証方法（docs MD）**: docs MD は JSON の人間可読レンダリングであるため、両者は完全に一致しなければならない。JSON が FAIL の場合は docs MD も自動的に FAIL とする。以下の要素を個別に検証する:

- **タイトル**: JSON `title` == docs MD の `# ` 見出し
- **セクションタイトル**: JSON 各セクションのタイトルが docs MD の `##`/`###` に存在し、かつ JSON と同じ順序で並んでいるか
- **hints**: JSON `hints` が docs MD の keywords ブロックに全て存在するか
- **content 完全一致**: JSON 各セクションの `content` が docs MD に完全に含まれているか

**検証方法（index.toon）**: 変換済みの全 JSON ファイルが index.toon に登録されているかを確認する。登録漏れがあると nabledge の検索から除外される。index.toon に存在する全エントリと出力ディレクトリの JSON ファイル一覧を突き合わせ、未登録ファイルを検出する。

## 6. verify の変更ルール

verify は品質ゲートであり、その変更は厳密に管理する。

### 変更プロセス

1. 変更内容と理由をユーザーに説明する
2. ユーザーの明示的な承認を取得する
3. TDD で実装する（RED → GREEN）

### 許容される変更

- verify 自体のロジックバグの修正（例: 正規表現の誤り、範囲の間違い）
- 仕様上必要なチェックの追加
- 誤検知（false positive）の修正

### 許容されない変更

- RBKC の現在の出力に合わせて verify 基準を緩める
- チェックを無効化・スキップする
- RBKC 実装の詳細に合わせた回避策を追加する
