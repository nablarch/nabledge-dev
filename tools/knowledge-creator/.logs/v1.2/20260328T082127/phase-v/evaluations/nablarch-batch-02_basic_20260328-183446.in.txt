================================
バッチ共通のアプリケーション構造
================================

本項では、バッチ処理に共通の基本的なクラス構造について説明する。

概要
====

Nablarch Application Frameworkでは、\
巨大で複雑になりがちなバッチ処理を簡潔かつ整然とした状態に保つことができるよう、\
以下のような機能を備えている。

* `作業単位`_\ 毎のループ処理を代行

 煩雑なループ制御をフレームワークが代行する為、
 業務アプリケーションは作業単位毎の業務処理に集中できる。

* 開始時、エラー発生時、終了時の各イベント時のコールバック

 各イベント発生時に、フレームワーク側から各イベントに対応するメソッドが起動される。
 これにより、業務アプリケーションでのエラー処理漏れやリソース解放漏れを抑止できる。

* ファイル入出力の簡易化

 設計書から自動生成されたフォーマット定義ファイルを使用することにより、
 物理的なファイルレイアウトを意識したプログラミングをしなくてよい。
 
 例えば、「レコード先頭xバイト目からxバイト分を半角数字として取得」するというような処理を
 プログラム上でする必要はなく、単に引数で与えられた入力データからそのフィールド名を
 指定して取得するだけでよい。
 

バッチ処理を実装する場合、mainメソッドから全処理を実装するのではなく、\
フレームワークのクラスを継承し、業務に必要な処理を実装する。


作業単位
--------

作業単位とは、同一のトランザクションで処理しなければならないデータのまとまりを指す。\
典型的なバッチ処理では、データベース入力の場合には検索結果1行、\
ファイル入力の場合は1レコードが作業単位となる。
 
コントロールブレイクを行う場合は複数件のまとまりが作業単位となることがある。\
例えば「顧客毎に当月の売上明細を集計して翌月請求額を求める」という場合には、\
顧客の当月請求明細のまとまりが作業単位となる。

DBを入力とするバッチを実装する場合
----------------------------------

BatchActionまたはそのサブクラスを継承し以下のメソッドを実装する。


================================================================== ============================= ================================= ======
メソッド                                                           概要                          起動タイミング                     要否
================================================================== ============================= ================================= ======
``void initialize(CommandLine command, ExecutionContext context)``  初期化処理を行う。            メイン処理開始前に1回             任意
``DataReader<D> createReader(ExecutionContext context)``            リーダを作成する。            初回データ読み込み時に1回         必須
``Result handle(D inputData, ExecutionContext context)``            メイン処理を行う。            1\ `作業単位`_ の入力毎           必須
``void transactionSuccess(D inputData, ExecutionContext context)``  正常終了時の処理を行う。      handleメソッド正常終了時毎        任意
``void transactionFailure(D inputData, ExecutionContext context)``  異常終了時の処理を行う。      handleメソッド異常終了時毎 [#]_   任意
``void error(Throwable error, ExecutionContext context)``           エラー発生時の処理を行う。    メイン処理でエラー発生後に1回     任意
``void terminate(Result result, ExecutionContext context)``         終了処理を行う。　            メイン処理終了後に1回             任意
================================================================== ============================= ================================= ======


.. [#] 
 この処理は、メイン処理とは別のトランザクションで実行される。
 これは、メイン処理でエラーが発生した場合、そのトランザクションはロールバックされて終了するからである。

.. note::
 上記表のメソッドシグネチャにある\ **D**\ は、レコードの型を表す総称型である。
 例えば、DB入力バッチの場合は、通常この型は\ **SqlRow**\ となる。
 
.. note::
 BatchActionBaseはDbAccessSupportを継承しているため、サブクラスでも
 DbAccessSupportの機能を用いてデータベースアクセスを行うことができる。

フレームワーク動作イメージ
--------------------------

フレームワークの処理制御を擬似コードで示すと以下のようになる。

.. code-block:: java

  initialize()                //-- 本処理開始前に一度だけ呼ばれる。
  createReader()              //-- 初回データ読み込み時に1度だけ呼ばれる。
  try {
    while(reader.hasNext()) {
      try {
        handle()              //-- 入力データ1件毎に繰り返し呼ばれる。
        transactionSuccess()  //-- handleが正常に終了した場合に呼ばれる。
      } catch(e) {
        transactionFailure()  //-- handleで例外が発生した場合に呼ばれる。
      }
    }
  } catch(e) {
    error()                 //-- 本処理がエラー終了した場合に、一度だけ呼ばれる。
  } finally {
    postProcess()             //-- 本処理が全て終了した後、一度だけ呼ばれる(エラー終了時でも呼ばれる)。
  }

コマンドライン引数
------------------

Nablarch Application Frameworkでは、Java起動時に渡されるコマンドライン引数を
以下の２通りに解釈する。

============ ===================================================== ==============================
種別         説明                                                  アプリケーションからの取得方法 
============ ===================================================== ==============================
 オプション  -キー名=値 という形式で表現されるkey-valueのペア。    CommandLine#getParamMap()      
 引数        オプション以外の引数。単一の値を持つ。                CommandLine#getArgs()          
============ ===================================================== ==============================


バッチ処理共通のコマンドラインオプション引数は以下のである。

============= ================================== =============================================================
キー名        設定値                             説明                                              
============= ================================== =============================================================

requestPath   リクエストを特定するためのパス     通常「"ss" + (機能ID) + "." + (アクションクラス名) + "/" + (リクエストID)」とする

メソッド詳細
============

初期化処理
----------

``initialize``\ メソッドを実装する（任意）。

メイン処理実行前にフレームワークにより起動される。\
本メソッドでは、入力データを処理する前に行うべき前処理を実装する。\
特に処理が必要ない場合はこのメソッドをオーバーライドする必要はない。


実装すべき処理の例を以下に示す。

* `コマンドライン引数`_ を処理する。
* リソースの取得を行う。
* 開始ログの出力を行う。
* インスタンス変数の初期化が必要な場合

    
.. warning:: 
  バッチアクションでインスタンス変数を使用する場合、以下の制約が生じる。
  
  * インスタンス変数を使用するバッチアクションは\ **マルチスレッド実行できない**\ 。
  * インスタンス変数は、初期化処理(initializeメソッド)で\ **明示的に初期化しなければならない**\ 。
  
  ただし、インスタンス変数を使用する場合であっても、以下の条件を満たす場合はマルチスレッド実行が可能である。
  
  * インスタンス変数の初期化（代入）が初期化処理で行われ、それ以降は値が更新されないこと（読取専用）。



リーダ作成
----------

``createReader``\ メソッドを実装する（必須）。


このメソッドでは、バッチ処理の入力となるデータを読み込むリーダを返却する。\
通常、入力データはファイルデータベースいずれかとなる。

フレームワークは、このメソッドを呼び出してリーダを取得する。
そのリーダを使用して入力データの読み込みを行う。


.. _`about_unit_of_work`:

1作業単位毎の処理
-----------------

``handle``\ メソッドを実装する（必須）。

フレームワークは、createReaderメソッドを呼び出して取得したリーダからデータを読み込む\
`作業単位`_\ 毎にバッチアクションの\ ``handle``\ メソッドを起動する。\
handleメソッドでは1作業単位毎のバッチ処理を行う。


戻り値には、nablarch.fw.Resultインタフェース実装クラスのインスタンスを返却する必要がある。\
通常は、正常終了（\ ``nablarch.fw.Result.Success``\ ）を返却する。\



エラー発生時の処理
------------------

エラーが発生した作業単位に対するエラー処理
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

``transactionFailuer``\ メソッドを実装する（任意）。

本メソッドはhandleメソッドでエラーが発生する度に呼び出される。
引数には、エラー発生時に処理していたデータが渡される。

.. note::
 このメソッドは、handleメソッドとは別のトランザクションで実行される。


本メソッドで実装すべき処理の例を以下に示す。
 
 * エラーが発生したレコードのステータスを異常終了に更新する。
 * 専用のエラーファイルに、エラーが発生したレコードを出力する（エラーリスト作成）。

 
エラーが発生したリクエストに対するエラー処理
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~


\ ``error``\ メソッドを実装する（任意）。

本メソッドは、本処理がエラー終了した場合に、一度だけ呼び出される。
（マルチスレッドで実行している場合でも、1回だけ呼び出される）。
通常、引数のThrowableには\ ``java.util.concurrent.ExecutionException``\ が渡される。
メイン処理で発生した例外またはエラーは、\ ``ExecutionException#getCause()``\ で取得できる。


終了処理
--------

``terminate``\ メソッドを実装する（任意）。

リーダから全てのデータを読み取った後、フレームワークから起動される。\
本メソッドでは、全データ処理後に行うべき終了処理を実装する。\

.. note::
 実装すべき処理の例を以下に示す。
  
 * リソースの解放を行う。
 * 処理件数をログ出力する。
