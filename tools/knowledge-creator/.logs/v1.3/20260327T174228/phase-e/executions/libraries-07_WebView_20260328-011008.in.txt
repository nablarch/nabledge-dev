.. _`custom_tag_index`:

====================================
JSPカスタムタグライブラリ
====================================

------------
概要
------------
Webアプリケーションの画面作成を助けるカスタムタグを提供する。

本機能を使用するには、下記の制約がある。

* JSP2.1以降をサポートしているWebコンテナで動作する。
* カスタムタグはXHTML 1.0 Transitionalに対応した属性をサポートする。
* クライアントのJavaScriptが必須である。

本機能のカスタムタグは、以下のような画面処理方式において最も効果的に機能するように設計されている。

* 画面遷移は全てPOST(リンク含む)を使用する。
* アクションはPOSTを受けてJSPにフォワードする。JSPへはアクション経由でのみアクセス可とする。
* アクションから別アクションを直接呼び出す場合は、リダイレクトではなく、フォワードを使用する。
* JSPはカスタムタグを使用して作成する。
* 入力データは、バリデーション機能を使用してエンティティ(又は独自クラス)に設定して使用する。
* 入力データのリクエストパラメータ名には、オブジェクトとMap型はドット区切り、List型は角括弧と括弧内にインデックスを使用する。
* 画面を跨いで使用するデータは、クライアントで保持する。(セッションを使用するのではなく、hiddenを使用する)
* データベースにコミットを伴う処理を要求する画面(確認画面など)は、二重サブミットを防止する。

本機能は、入力画面→確認画面→完了画面といった画面遷移により機能を提供するWebアプリケーションを対象にしており、\
画面遷移せずに機能を提供するリッチクライアント(Ajax、Flashなど)には対応していない。

アプリケーションプログラマは、JSP、JSTL及び本機能を使用してWebアプリケーションの画面を作成する。

.. note::

 HTML5で追加されたタグと属性については、次の属性のみ対応する。\
 属性が追加されたHTMLのタグ名を ( ) に記載する。
 
 * autocomplete(input、password、form)
 * autofocus(input、textarea、select、button)
 * placeholder(text、password、textarea)
 * maxlength(textarea)
 * multiple(input)
 
 HTML5の属性のうち、頻繁に使用されるものを先行して取り込んでいる。

------------
特徴
------------

生産性の向上
====================================
本機能のカスタムタグでは、入力画面と確認画面が1対1に対応するような画面を作成する場合に、JSPを共通化する機能を提供する。\
JSPを共通化することで、入力画面と確認画面で重複する記述を最小化できるので、画面作成の生産性が向上する。\

画面設計の柔軟性
================================================
本機能では、入力データを画面間で持ち回る際はクライアント側(ブラウザ)に保持する機構を採用しているため、\
ブラウザに予め備わっている「戻るボタン」や「複数ウィンドウ」といった機能をアプリケーションで使用できる。\
また、ユーザの誤操作による「二重サブミット」については、JavaScriptを使用したクライアント側と、\
一意なトークンを使用したサーバ側のそれぞれで二重サブミットを防止する機能を提供する。\
このため、個別アプリケーションでは、ブラウザの操作性やユーザの誤操作について予め考慮している本機能を前提にすることで、\
画面設計を柔軟に行うことができる。

------------
要求
------------

実装済み
========================
* フォームに含まれる複数のボタン又はリンクから、それぞれ異なるURIにサブミットできる。
* フォームに含まれる複数のボタン又はリンク毎にパラメータを変更できる。
* ボタンの二度押しによるエラー発生やデータ不整合を防止できる。
* 入力画面の再表示時(精査エラーや確認画面から戻る場合など)に、入力データを復元した状態で表示できる。
* ブラウザの戻るボタンが使用された場合に、キャッシュの使用を抑制できる。
* HTMLエスケープ処理ができる。
* エラーが発生した際に、全てのエラー内容を伝えるメッセージを画面に表示できる。
* 入力値のチェックでエラーが発生した際に、エラーの原因となった入力項目をハイライト表示できる。
* エラーが発生した際に、エラーメッセージをレベル毎に色分け(Error:赤、Warning:青、Info:緑など)して表示できる。
* リストボックス表示を作りこみせずに実現できる。
* コード値をリストボックス表示できる。(コード値については、 :doc:`../03_Common/02_CodeManager` を参照)
* 入力画面と確認画面のJSPを共通化することで、重複する記述を抑制できる。
* クライアント側(hidden)で保持するデータの改ざんを防止できる。
* 認可判定の結果に応じて画面項目（メニューやボタンなど）の表示・非表示を切り替えることができる。
* 開閉局判定の結果に応じて画面項目（メニューやボタンなど）の表示・非表示を切り替えることができる。

未実装
==================
* 確認画面で、変更箇所を色付け表示できる。
* JavaScriptに埋め込む動的な値に対してエスケープ処理ができる。

未検討
==================
* システム利用ユーザ間で共有したい情報を表示できる。例えば、「今日は早帰り日です」のようなもの。
* 実行環境がわかる状態で表示できる。例えば、単体テスト環境なのかST環境なのかが画面表示できる。
* 表示項目の内容に応じて表示形式の制御ができる。例えば、金額項目であれば、右寄せ折り返し表示など。
* ヘッダーやフッターなどの全画面で共通的に表示する部分のレイアウトを定義できる。
* 精査エラーが発生した際に先頭のエラー項目にフォーカスできる。
* エンターキーで入力項目間を移動できる。
* エンターキーによるサブミットの許可・不許可を制御できる。
* キーバインドをカスタマイズできる。例えば、F1(ファンクションキー)でサブミットするなど。
* クライアントサイドのバリデーションができる。
* ラジオボタンをクリア(未選択な状態)できる。
* 画面全体の背景色の指定ができる。
* 入力画面の場合、各業務で指定がない場合は初期フォーカスをデフォルト(例えば、一番上の入力ボックス)で指定する。
* 項目名や見出し、入力ボックスなどの背景色、文字(サイズ、色、パディング)の指定ができる。
* ボタンのサイズ、色などの指定ができる(標準ボタンの提供)。

取り下げ
========================
* 複数のエラーが発生した場合、レイアウトを崩さずに全てのエラー内容を伝えるメッセージを画面に表示できる。

  全てのエラー内容をリスト表示する機能は提供している。ただし、「レイアウトを崩さずに」という部分については、\
  アプリケーション開発時にレイアウトが決定するため、フレームワークとしてレイアウト崩れを防ぐ方法が提供できない。
