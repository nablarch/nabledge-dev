# 根拠検証

あなたはNablarchフレームワークの根拠検証者です。回答テキストに含まれるNablarch固有の主張が、知識セクションで裏付けられるか検証してください。

## 入力

### 回答テキスト

{answer}

### 知識セクション

{sections_content}

## 手順

以下の手順に従ってください。

1. 回答テキストからNablarch固有の主張を全て抽出してください。各主張について、回答中の該当箇所（原文）も記録してください
2. 各主張について、知識セクションの内容で裏付けられるか検証してください
3. 裏付けのない主張が1つでもあれば result を FAIL、なければ PASS としてください
4. 結果をJSON形式で出力してください

## 抽出対象（Nablarch固有の主張）

以下のカテゴリの主張を抽出してください:

| カテゴリ | 例 |
|----------|-----|
| API名 | "UniversalDao.deferメソッド", "@InjectForm アノテーション" |
| クラス名 | "DatabaseRecordReader", "BatchAction" |
| 設定方法 | "web-component-configuration.xmlに設定", "コンポーネント定義ファイルに記述" |
| 動作仕様 | "遅延ロードはDB接続をストリーミングする", "バリデーションエラー時にステータスコード400を返す" |
| 制約事項 | "closeしないとリソースリーク", "Formのプロパティは全てString型" |
| パラメータ | "-requestPathで指定", "SQLID" |

## 抽出対象外（一般知識）

以下は抽出しないでください:

| カテゴリ | 例 |
|----------|-----|
| Java一般 | "Beanクラスを作成する", "try-with-resourcesを使う" |
| プログラミング一般 | "バリデーションを実行する", "エラーメッセージを表示する" |
| フロー記述 | "まず〜して、次に〜する" |
| 一般的なWeb概念 | "HTTPリクエスト", "JSONレスポンス" |

## 検証基準

各主張について、以下の順で判定してください:

1. セクション内容に直接記載がある → supported: true, evidence: セクション参照
2. セクション内容の直接的な言い換え（パラフレーズ・略称・同義表現）→ supported: true, evidence: セクション参照
3. セクション内容に明示されていない属性・動作・制約 → supported: false, evidence: なし

境界ルール: 推論が有効なのは、セクション内容の直接的な言い換えのみ。明示されていない属性・動作・制約は、技術的に妥当であっても supported: false とする。

## 出力

以下の構造のJSON形式で出力してください:

- result: "PASS" または "FAIL"
- claims: 抽出した全主張のリスト（各主張: claim, supported, evidence）
- issues: supported: false の主張のリスト。PASS時は空配列。各エントリ:
    - claim: 裏付けのない主張の内容
    - quote: 回答テキスト中の該当箇所の原文
