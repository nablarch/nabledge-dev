### Answer

DBから読み込んで別テーブルに書き込む定期集計処理は、**都度起動バッチ × DB to DB パターン**で実装するのが適切です。

---

**構成の選択根拠**

- **起動方式：都度起動バッチ**
  - 日次・月次などの定期実行には都度起動バッチを使います。プロセスをその都度起動して処理を完結させます。
  - 常駐バッチはDBを監視して非同期処理する用途向けであり、定期集計には不要です。

- **入出力パターン：DB to DB**
  - 入力はSELECT文の結果セットの各レコードです（`createReader` で発行）。
  - `handleメソッド` で1レコード分のデータを受け取り、集計結果テーブルをINSERT/UPDATEします。
  - 1レコードの処理中に行われる更新はすべて同一トランザクション下で実行されるため、障害発生時も不整合が発生しません。

---

**実装上の注意点**

**N+1問題を避ける（重要）**

集計に必要な関連データを `handleメソッド` 内で都度SELECTすると、処理件数Nに対してN+1回のSQLが発行され深刻な性能劣化が起きます。`createReader` で発行するSQLにJOINを含め、1回で必要なデータをすべて取得するようにしてください。

**処理の流れ（アーキテクチャ）**

1. 共通起動ランチャ（Main）がハンドラキューを実行
2. DataReader が集計対象レコードを1件ずつ提供（SELECT）
3. DispatchHandler がリクエストパスからアクションクラスを特定
4. アクションクラスの `handleメソッド` で集計ロジックを実行し、結果テーブルへINSERT/UPDATE
5. 処理対象データがなくなるまで2〜4を繰り返す

**ファイルを扱う場合**

今回はDB to DBなので不要ですが、もし将来ファイル出力が加わる場合（DB to FILE）は、ファイル移動・コピー処理をバッチ本体に含めず別出しにしてください。再実行しやすさとテスタビリティが向上します。

---

**まとめ**

| 項目 | 選択 |
|------|------|
| 起動方式 | 都度起動バッチ |
| 入出力パターン | DB to DB |
| 入力 | `createReader` でSELECT（JOINで必要データを一括取得） |
| 出力 | `handleメソッド` で集計結果テーブルへINSERT/UPDATE |
| トランザクション | 1レコード処理ごとに同一トランザクション内で完結 |
