**結論**: APサーバをスケールアウトする場合、セッション変数の保存先は **DBストア** または **Redisストア** を選択するべきです。HTTPセッションストアはスケールアウト時に工夫が必要で、原則として避けます。

**根拠**:

Nablarch が標準提供するセッションストアは以下の3種類＋アダプタ1種です。

### DBストア
- 保存先: データベース上のテーブル
- **スケールアウト対応**: 全APサーバがDBを共有するため、スケールアウトに対して自然に対応できる
- ローリングメンテナンスなどでAPサーバが停止してもセッション変数を復元可能
- APサーバのヒープ領域を圧迫しない
- 同一セッションの処理が複数スレッドで同時実行された場合は「後勝ち」（先に保存されたデータは消失）

### HIDDENストア
- 保存先: クライアントサイド（hidden タグを使って画面間でセッション変数を引き回す）
- **スケールアウト対応**: 保存先がクライアントのため、APサーバ台数に影響されない
- 複数タブでの画面操作を許容できる
- APサーバのヒープ領域を圧迫しない
- 複数スレッド同時実行では、スレッドごとに独立したデータとして保存される

### HTTPセッションストア
- 保存先: APサーバのヒープ領域（サーバ設定によってはDBやファイルに保存されることもある）
- **スケールアウト非推奨**: 各APサーバが独立してセッション情報を持つため、スケールアウト時に工夫が必要
- 認証情報など全体で頻繁に使う情報の保持には適しているが、APサーバ単体での利用を前提とする
- 大量データを保存するとヒープを圧迫する恐れがある

### Redisストア（Lettuce アダプタ）
- 保存先: Redis（外部KVS）
- **スケールアウト対応**: 特徴はDBストアと同じ（保存先が異なるだけ）
- セッション情報は `nablarch.session.<セッションID>` というキーで保存される
- Cluster / Sentinel / Standalone 等 Redis の構成に合わせた設定が可能

### 用途別選択基準

| 用途 | 推奨ストア |
|------|-----------|
| 入力〜確認〜完了間の入力情報保持（複数タブ不可） | DBストア |
| 入力〜確認〜完了間の入力情報保持（複数タブ許容） | HIDDENストア |
| 認証情報の保持 | DBストア または HTTPセッションストア |
| 検索条件・一覧・画面表示項目・エラーメッセージ | 使用しない（セッションストアは不適切） |

**注意点**:
- スケールアウト構成では HTTPセッションストアを認証情報保持に使うには追加の工夫（スティッキーセッション、セッションレプリケーション等）が必要になります。スケールアウトを前提にするなら DBストア への移行が推奨されます
- セッションストアは複数機能にまたがるデータ保持を想定していません。検索条件・一覧・表示用マスタ等はセッションに入れず、URLやリクエストスコープで渡す設計にしてください
- 大量データのセッション保存は DBストアでも負荷の原因になりえます
- セッションの有効期間はデフォルトで HTTPセッションに保存されますが、DBストアや Redisストアに変更することで完全にステートレスな構成にできます
