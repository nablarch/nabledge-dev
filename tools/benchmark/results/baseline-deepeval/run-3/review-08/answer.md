**結論**: 入力→確認→完了画面間のセッション変数保持では、「複数タブでの操作を許容するか否か」が使い分けの唯一の基準です。許容しないならDBストア、許容するならHIDDENストアを選びます。

**根拠**:

セッションストアの特長と選択基準（`libraries-session-store.json:s16` より）：

| 用途 | セッションストア |
|---|---|
| 入力～確認～完了画面間で入力情報の保持（複数タブを**許容しない**） | **DBストア** |
| 入力～確認～完了画面間で入力情報の保持（複数タブを**許容する**） | **HIDDENストア** |

各ストアの特長（`libraries-session-store.json:s16` より）：

**DBストア**（データベース上のテーブルに保存）
- ローリングメンテナンス等でAPサーバが停止した場合でもセッション変数の復元が可能
- APサーバのヒープ領域を圧迫しない
- 同一セッションの処理が複数スレッドで実行された場合、後勝ちとなる（先に保存されたデータは消失する）

**HIDDENストア**（クライアントサイドの `hidden` タグで画面間を引き回す）
- 複数タブでの画面操作を許容できる
- APサーバのヒープ領域を圧迫しない
- 複数スレッドで実行された場合、セッションのデータはそれぞれのスレッドに紐付けて保存される

HIDDENストアを使用する場合は、入力・確認画面のJSPに `hiddenStore` タグを追加する必要があります（`libraries-session-store.json:s9` より）：

```jsp
<n:form>
  <!--
    name属性にはコンポーネント設定ファイルに定義した、
    HiddenStoreのparameterNameプロパティの値を設定
  -->
  <n:hiddenStore name="nablarch_hiddenStore" />
  <!-- その他のタグは省略 -->>
</n:form>
```

**注意点**:
- DBストアを使用する場合、データベース上に `USER_SESSION` テーブルを作成する必要があります
- DBストアを使用した場合、ブラウザが閉じられるなどしてテーブル上にセッション情報が残ることがあるため、期限切れのセッション情報を定期的に削除する必要があります（`libraries-session-store.json:s8`）
- HIDDENストアはデータがクライアントサイドに保存されるため、アプリケーションサーバが冗長化されている場合は暗号化キーを明示的に設定する必要があります（`libraries-session-store.json:s12`）
- セッションストアに格納するオブジェクトはFormではなく、業務ロジックを実行するためのEntity（バリデーション済みのオブジェクト）を推奨します（`libraries-session-store.json:s9`）

参照: `libraries-session-store.json:s9`, `libraries-session-store.json:s16`
