**結論**: スケールアウト環境では、**DBストア**（またはRedisストア）を選択するのが基本方針です。HTTPセッションストアはAPサーバ毎に状態を持つため、そのままではスケールアウトに対応できません。

**根拠**:

Nablarchは標準で以下の3種類のセッションストアを提供しています（Redisは別途アダプタで追加可能）。

| ストア | 保存先 | スケールアウト適性 |
|--------|--------|------------------|
| DBストア | データベース上のテーブル | ◎ APサーバ間で共有可能 |
| HIDDENストア | クライアントサイド（hiddenタグ経由） | ◎ APサーバ間の共有不要 |
| HTTPセッションストア | APサーバのヒープ領域 | △ 工夫が必要 |
| Redisストア | Redis（アダプタ使用時） | ◎ APサーバ間で共有可能 |

**各ストアの特徴**:

**DBストア**
- ローリングメンテナンス等でAPサーバが停止した場合でもセッション変数の復元が可能
- APサーバのヒープ領域を圧迫しない
- 同一セッションの処理が複数スレッドで実行された場合、後勝ちとなる（先に保存されたデータは消失する）
- 期限切れセッション情報は定期的に削除するバッチが必要

**HIDDENストア**
- 複数タブでの画面操作を許容できる
- APサーバのヒープ領域を圧迫しない
- 同一セッションの処理が複数スレッドで実行された場合、セッションのデータはそれぞれのスレッドに紐付けて保存される
- 保存先はクライアント側のため、APサーバをまたいだ共有という概念がそもそも不要

**HTTPセッションストア**
- 認証情報のようなアプリケーション全体で頻繁に使用する情報の保持に適している
- **APサーバ毎に情報を保持するため、スケールアウトを行う際に工夫が必要となる**
- 大量データを保存するとヒープ領域を圧迫する恐れがある

**Redisストア（Lettuceアダプタ）**
- 保存先が異なるだけで特徴はDBストアと同じ
- 有効期限が切れたキーはRedisが自動的に削除するため、ゴミ掃除バッチが不要

**用途別の選択基準**（s16より）:

| 用途 | 推奨ストア |
|------|-----------|
| 入力〜確認〜完了画面の入力情報保持（複数タブ不可） | DBストア |
| 入力〜確認〜完了画面の入力情報保持（複数タブ許容） | HIDDENストア |
| 認証情報の保持 | DBストア または HTTPセッションストア |

**注意点**:

- セッションの有効期間はデフォルトではHTTPセッションに保存されます。スケールアウト環境でDBストアを使用する際は、`DbManagedExpiration` に差し替えて有効期間もデータベースに保存するよう設定変更が必要です（`SessionStoreHandler` の `expiration` プロパティで設定）。
- HTTPセッションストアでスケールアウトに対応するには、スティッキーセッション有効化、セッションレプリケーション、NoSQL化などの対処が必要です。
