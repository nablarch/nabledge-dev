**結論**: NablarchにはWebSocketのサポートは提供されていません（知識ファイルに記載なし）。バッチ進捗をブラウザへリアルタイム通知するには、Nablarchが推奨する非同期パターン（テーブルをキューとして使ったメッセージング＋ブラウザポーリング）で代替実装するアプローチが現実的です。

**根拠**:

Nablarchのウェブアプリケーションが提供する機能の一覧（初期化・バリデーション・DBアクセス・ファイル操作・URI/アクションマッピング・CSRF対策等）にWebSocketは含まれておらず、アダプタとしても提供されていません。

NablarchのOSSを使用しない方針（知識ファイル：基本方針）により、WebSocket関連ライブラリのアダプタも提供されません：

> Nablarchのプロダクションコードは、致命的な不具合や脆弱性が見つかった際に、迅速に対応かつリリースすることを目的としてOSSを使用していない。

代わりに、Nablarchが推奨する非同期処理パターンを活用した構成が考えられます：

```
[ブラウザ]                    [Webアプリ]              [バッチ]
  |-- バッチ起動リクエスト --> |                          |
  |                           |-- 依頼データをDB登録 --> |
  |<-- レスポンス返却 -------  |                          |
  |                           |              バッチ処理中 |
  |-- 進捗確認ポーリング ----> |                          |
  |  (定期的にREST APIを呼ぶ) |-- 進捗状況をDBから取得   |
  |<-- 進捗データ返却 -------  |                          |
  |                           |        バッチ終了、進捗を |
  |-- 最終確認 -------------> |        DBに記録           |
  |<-- 完了 ----------------   |                          |
```

具体的な実装の流れ：
1. バッチが処理件数・処理済み件数などの進捗状況をDBテーブルに書き込む
2. ウェブアプリ側にRESTful APIエンドポイント（進捗照会）を設ける
3. ブラウザからJavaScriptで定期的（例：1〜3秒間隔）にポーリングしてUIを更新する

**注意点**:

- WebSocketが要件として必須の場合、NablarchのフレームワークではカバーできないためJakarta WebSocket APIをNablarchフレームワーク外で直接使用する必要がありますが、この場合はNablarchのハンドラキューやDIコンテナの管理外となります。
- ポーリング間隔が短すぎるとサーバ負荷が上がるため、進捗の変化頻度に応じて適切な間隔を設定してください。
