**結論**: 入力→確認→完了画面間のセッション変数保持では、**複数タブでの操作を許容しない場合はDBストア**、**複数タブでの操作を許容する場合はHIDDENストア**を使い分けます。

**根拠**:

選択基準は「複数タブでの画面操作を許容するか否か」が唯一の判断軸です。

| 用途 | セッションストア |
|---|---|
| 入力～確認～完了画面間での入力情報保持（複数タブ不許可） | DBストア |
| 入力～確認～完了画面間での入力情報保持（複数タブ許可） | HIDDENストア |

各ストアの特徴:

**DBストア**（データベース上のテーブルに保存）
- APサーバが停止してもセッション変数の復元が可能（ローリングメンテナンス等に対応）
- APサーバのヒープ領域を圧迫しない
- 同一セッションが複数スレッドで並行実行された場合は後勝ち（先のデータは消失）

**HIDDENストア**（クライアントサイド＝hiddenタグで画面間を引き回す）
- 複数タブでの画面操作を許容できる
- APサーバのヒープ領域を圧迫しない
- 同一セッションが複数スレッドで実行された場合でも、各スレッドに紐付けて保存される

HIDDENストアを使用する場合は、入力・確認画面のJSPに `hiddenStore` タグが必要です。

```jsp
<n:form>
  <!--
    name属性にはコンポーネント設定ファイルに定義した、
    HiddenStoreのparameterNameプロパティの値を設定
  -->
  <n:hiddenStore name="nablarch_hiddenStore" />
  <!-- その他のタグは省略 -->
</n:form>
```

**注意点**:
- セッションストアに格納するのは **FormではなくEntity（業務ロジック実行用オブジェクト）** にすること。Formを格納するとバリデーション前の信頼できない値が混入するリスクがあります。
- 複数機能にまたがるデータ（検索条件・一覧データ等）の保持にはセッションストアを使わないこと。
- DBストアの有効期間はデフォルトでHTTPセッションに保存されるが、設定変更によりデータベースに保存することも可能です。
