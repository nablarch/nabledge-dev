**結論**: NablarchのJSPカスタムタグを使えば、HTML出力時のエスケープ（サニタイジング）が自動で適用され、XSSの根本的解決（IPA分類の5-(i)）が可能です。ただし対応できる範囲に限界があり、JavaScriptへのエスケープ・URL制限・スタイルシート制御などはプロジェクトで個別対応が必要です。

---

**根拠**:

### カスタムタグで対応できる範囲

**5-(i): ウェブページに出力する全ての要素に対してエスケープ処理を施す → 〇**

カスタムタグは出力時に全ての HTML 属性に対して以下のエスケープを自動適用します：

| 元の文字 | 変換後 |
|--------|------|
| `&` | `&amp;` |
| `<` | `&lt;` |
| `>` | `&gt;` |
| `"` | `&#034;` |
| `'` | `&#039;` |

カスタムタグを使って実装している限り、HTMLエスケープ漏れを防げます。

**5-(viii): Content-Type に charset 指定 → 〇**

HTTP文字エンコード制御ハンドラが Content-Type に MIME Type・文字コードを設定し、特定ブラウザで発生し得る XSS を防ぎます。

**5-(v): 入力値の内容チェック → 〇**（保険的対策）

Bean Validation / Nablarch Validation による入力値チェックで対応可能です。

**5-(x): セキュリティヘッダの返却 → 〇**（保険的対策）

セキュアハンドラによりブラウザの XSS 対策機能を有効にするレスポンスヘッダが出力されます。

---

### カスタムタグでも対応できない範囲（プロジェクト対応が必要）

| IPA 分類 | 内容 | Nablarch 対応状況 |
|---------|------|----------------|
| 5-(ii) | URL を出力するときは http:// または https:// のみ許可 | × |
| 5-(iii) | `<script>...</script>` 要素の内容を動的に生成しない | × |
| 5-(iv) | スタイルシートを任意のサイトから取り込めるようにしない | × |
| 5-(ix) | Cookie に HttpOnly 属性を加える | × |

---

### 注意点

**JavaScript へのエスケープは提供していません。**  
`scriptタグ` のボディや `onclick` 属性など、JavaScriptを記述する部分に動的な値（入力データなど）を埋め込まないこと。埋め込む場合はプロジェクトの責任でエスケープ処理を実施してください。

**EL式（`${...}`）を直接出力に使用しない。**  
JSPのEL式はHTMLエスケープを行わないため、値を出力する場合は `<n:write>` などのカスタムタグを使用すること。

**HTMLテキスト入力を許可する場合（5-(vi)）:**  
Nablarchは未対応です。OSSのHTMLパーサを使って入力値をパースし、使用できないHTMLタグを検出するバリデーションを実装するか、MarkdownでユーザーにHTMLを入力させてクライアントサイドでHTML変換する方式を検討してください。
