**結論**: 入力→確認→完了画面間のセッション変数保持では、**複数タブでの画面操作を許容しないならDBストア**、**許容するならHIDDENストア**を使用する。

---

**根拠**:

Nablarchのセッションストアは3種類あり、用途の選択基準は以下のとおり。

| 用途 | セッションストア |
|---|---|
| 入力〜確認〜完了画面間の入力情報保持（複数タブ不許容） | DBストア |
| 入力〜確認〜完了画面間の入力情報保持（複数タブ許容） | HIDDENストア |

**DBストアの特長**:
- データベース上のテーブルにセッション変数を保存（サーバサイド保持）
- ローリングメンテナンス等でAPサーバが停止してもセッション変数を復元可能
- APサーバのヒープ領域を圧迫しない
- 同一セッションで複数スレッドが処理した場合は後勝ち（先のデータは消失）

**HIDDENストアの特長**:
- `hidden` タグを使用して画面間でセッション変数をクライアントサイドで引き回す
- 複数タブでの画面操作を許容できる
- APサーバのヒープ領域を圧迫しない
- 同一セッションで複数スレッドが処理した場合、セッションデータはそれぞれのスレッドに紐付けて保存される

**HIDDENストア使用時のJSP記述例**:

```jsp
<n:form>
  <!--
    name属性にはコンポーネント設定ファイルに定義した、
    HiddenStoreのparameterNameプロパティの値を設定
  -->
  <n:hiddenStore name="nablarch_hiddenStore" />
  <!-- その他のタグは省略 -->
</n:form>
```

また、セッションストアにはFormではなくEntityを格納することが推奨される。Entityを格納することでセッションから取り出したオブジェクトをそのまま業務ロジックで使用でき、凝集性が高まる。

---

**注意点**:

- **ハンドラの配置順（HIDDENストア）**: HIDDENストア使用時、`SessionStoreHandler`はマルチパートリクエストハンドラより後ろ、かつ内部フォワードハンドラより前に配置する必要がある。内部フォワードハンドラが先にあると最新のセッション変数を取得できない問題が発生する。

- **改竄検知の挙動（HIDDENストア）**: HIDDENストアの改竄を検知した場合、ステータスコード400の `HttpErrorResponse` が送出される（他のストアとは挙動が異なる）。

- **APサーバ冗長化時の暗号化キー設定（HIDDENストア）**: APサーバが冗長化されている場合、デフォルトの自動生成キー（AES）はサーバごとに異なるため復号に失敗する。この場合は `HiddenStore` の `encryptor` プロパティで暗号化キー（`AesEncryptor` + `Base64Key`）を明示的に設定すること。

- **DBストアの期限切れレコード削除**: ブラウザが閉じられた場合などにテーブル上にセッション情報が残ることがあるため、期限切れのセッション情報は定期的に削除する必要がある。
