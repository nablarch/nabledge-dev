# Index Enrichment

`index-llm.md` に検索の手がかりとなるキーワードを付記し、AI-1 (search) の
ページ発見能力を上げるためのワークフロー。

## 検索フロー全体像（前提）

AI-1 の検索は 2 経路で構成される:

1. **index-llm.md 経路（意味マッチ）**: AI-1 が index を読んで `selections` を返す
2. **term_queries 経路（本文 grep）**: スクリプトが質問文から機械抽出した語で
   全ページ本文を grep し、ヒットしたセクションを selections にマージ

本ワークフローは **1 の index-llm.md 経路** を強化するもの。
2 の term_queries は **ASCII 識別子** (CamelCase / camelCase / @Annotation)
のみを機械抽出するので、それら識別子を index-llm.md に追記しても重複する
だけで価値が増えない。

そのため **index-llm.md に入れるのは日本語の概念語** に限定する。
具体的には、**ページ本文に存在しないがそのページの主題を表す日本語語**、
または**ページ本文に出るが AI-1 が質問と結び付けにくい日本語特徴語**。

日本語は term_queries では扱わない。質問の複合語は本文と表記がズレ、
短い語は generic すぎて noise になるため。

## 目的

AI-1 は `index-llm.md` を読んでユーザー質問に合うページを選ぶ。現状の index
はページタイトルとセクションタイトルしか持たないため、質問文のキーワードが
それらに出てこないページは発見されない。

本ワークフローは、各ページの本文から「そのページを特徴づけるキーワード」を
TF-IDF で抽出し、人が審査して index に付記する。

## index 形式 (セクション単位配置)

```
[handlers-transaction_management_handler] トランザクション制御ハンドラ
  s1:ハンドラクラス名 — トランザクション管理 / コールバック / ロールバック
  s2:モジュール一覧
  s3:制約 — 配置順序 / DbConnectionManagementHandler
  s5:特定の例外の場合にトランザクションをコミットさせる — コミット / ロールバック
  s6:トランザクション終了時に任意の処理を実行したい — コールバック / TransactionEventCallback
  s7:アプリケーションで複数のトランザクションを使用する — transactionName
```

### 配置ルール

各採用語 term について:

- **ページ内の全セクション本文を走査**
- **term が本文に出現するセクション全てに配置** (セクション行末に `— ` で付記)
- 複数セクションに出現する語は **重複して配置**
  (AI-1 にとって各セクションでの手がかりになる)

### ヒットなし時はエラー

採用語がページ本文のどのセクションにも出現しない場合、**エラー出力して調査する**。

- 想定される原因:
  - 採用リストが別バージョンのもの、または版ズレ
  - 本文の記号/表記ゆれで grep が通らない
  - タイトル/セクションタイトルにしか出ていない (title_overlap 除外漏れ)
- **黙ってスキップしない**。ビルドログに警告を出し、必要に応じて調査してから本計測へ

## 対象バージョン

開発起点は v6 だが、すべてのスクリプトは `--version` オプションで
他バージョン (v5 / v1.4 / v1.3 / v1.2) に切替可能。

## ディレクトリ構成

```
tools/benchmark/

  build_index.py              # index-llm.md 生成

  docs/
    index-enrichment.md       # 本ドキュメント
```

## 全体フロー

```
1. 候補抽出 (classify_terms.py)
       ↓
2. 自動分類
   - JAVA_STOPLIST (Java 標準クラス名) → 除外
   - stop_title_overlap (top ページのタイトル/セクションタイトルに既に含まれる語)
     → 除外。index に追加しても AI-1 には既に見えているので価値がない
   - @Annotation / CamelCase / camelCase → **index には入れない**
     (term_queries 経路が機械抽出でカバーするため)
   - それ以外 (日本語・漢字 4 字以上) → pending
       ↓
3. 人間判定 (スコア帯ごと、3 エージェント + メインエージェント + ユーザー)
       ↓
4. 採用リスト確定 → build_index.py に組み込み
       ↓
5. 効果測定 (benchmark run.py --search-only / search_coverage.py)
```

**重要**:
- ステップ 2 の自動分類は人間判定の前に必ず完了させる
- `stop_title_overlap` チェックは絶対に省かない (手動判定コスト削減 + 重複記載防止)
- `@Annotation / CamelCase / camelCase` は index-llm.md に入れない。
  これらは term_queries 経路でカバーされる

## 抽出パターン (classify_terms.py)

正規表現で以下を抽出し、TF-IDF の対象トークンとする。

| パターン | 例 | 分類 |
|---|---|---|
| `@Annotation` | `@Published` `@UseToken` | index 対象外 (term_queries で拾う) |
| CamelCase 2-hump | `DataReader` `BatchAction` | index 対象外 (term_queries で拾う) |
| camelCase プロパティ | `connectionFactory` `transactionName` | index 対象外 (term_queries で拾う) |
| カタカナ 4字以上 | `トランザクション` `バリデーション` | pending (人間判定) |
| 漢字 4字以上 | `公開API` `並列実行` | pending (人間判定) |
| 漢字+カタカナ混合 4字以上 | `暗号化処理` `悲観ロック` | pending (人間判定) |

**注**: 3字以下の語は採用しない。3字以下は一般語が多く、4字以上の複合語で
代替できる (例: `暗号化` は `暗号化処理` `暗号化機能` で拾える)。

**term_queries 経路との役割分担**:
- 識別子 (@Annotation / CamelCase / camelCase) は質問文に直接書かれること
  が多く、機械抽出してページ本文 grep するのが確実
- 日本語・漢字の概念語は、質問と本文で表記が揺れやすい (質問「レート制限」/
  本文「流量制御」など)。この層で意味の橋渡しをするのが index-llm.md の役割

## TF-IDF の意味

- TF = そのページ内での語の頻度
- IDF = 全ページ中で何ページに出るかの逆数
- `max_score` = その語の全ページ中の最大 TF-IDF スコア

| スコア帯 | 位置付け |
|---|---|
| 0.8〜1.0 | そのページの看板語 |
| 0.5〜0.8 | そのページの主要テーマ |
| 0.3〜0.5 | 関連はあるが主役ではない |
| 0.1〜0.3 | 脇役 |
| < 0.1 | ほぼ無関係 |

## 判定基準

採用 ○ は以下をすべて満たすこと:

- (A) 開発者が Nablarch に関する質問を書くときに、その語またはごく近い語を使う可能性がある
- (B) 上位ページの主題と意味的に結びついている
- (C) 過度に汎用ではない (df が 20% を超えるなら原則 ×)

× を付ける代表パターン:

- 文書メタ語 / 自己参照: 「本機能」「記載」「補足」「見出し一覧」
- 文書構造の副産物: 「外部サイト」「リンク先」
- 過度に広い一般語: 「設定」「使用」「処理」
- ページ主題とほぼ無関係 (スコア上位でも文脈ズレ)

## 人間判定のコツ

- **左から右**: 高スコア帯から順に判定。高スコア帯ほど明確な ○ が多い
- **× のチェックで十分**: 採用側は TF-IDF が保証している。×（除外すべき）だけ丁寧に見る
- **df を併読**: 高スコアでも df が大きいなら汎用語の可能性
- **snippet で意味確認**: 「コミット」= DB コミットか git コミットか等を判別
- **セクションタイトル確認**: ページ主題との結びつきが分かる

## スコア帯ごとの作業指針

- 一帯あたり 50〜200 語の粒度で人が処理できるよう、**0.05〜0.10 刻み**で
  区切って上から順に ●/× を判定する
- 高スコア帯 (0.5 以上) は 0.10 刻みで十分 (1 帯 15〜40 語)
- 中スコア帯 (0.15〜0.50) は 0.05 刻みで区切る (1 帯 70〜350 語)
- **× だけ重点的に確認すれば十分**。採用側は TF-IDF が保証しているので、
  漏れより誤採用が問題になる
- 0.15 未満は原則見ない。脇役以下でノイズが圧倒的多数。どうしても
  救いたい語があれば df を絞って個別に追加する

## 判定の実施方法 (3エージェント並列方式)

スコア帯ごとの判定は、メインエージェントの単独判定に加え、サブエージェントを
3 並列で起動して独立判定させることで精度を上げる。

### フロー

1. **データ準備**: 対象帯の全語 (term, df, df_pct, top_pages, snippets) を
   1 つの配列にまとめる
2. **3 サブエージェントに並列投げ**: Task ツールで同一プロンプト・同一データを
   3 つのサブエージェントに同時に送る。各エージェントは独立に判定
3. **結果集計**: 3 エージェントの ○/× を照合
   - 3-0 ○: 採用候補 → メインエージェントがスクリーニング
   - 2-1, 1-2, 0-3: ストップリスト確定
4. **メインエージェントのスクリーニング**: 3-0 ○ の中から明らかに OK と
   判断できないものをユーザーに相談
5. **ユーザー最終確認**: 迷う語のみユーザーが判定

### なぜ 3 エージェントか

- 単独判定ではミスが見えない
- 3 エージェントの意見が一致すれば判定が安定
- 分かれた語が「本物の境界例」として自動的に浮上する

### なぜバッチ判定か (1 語ずつ CLI 起動しない)

1 語ずつ `claude CLI` を起動すると、1 語あたり CLI 起動オーバーヘッドで
10〜20 秒かかる (推論時間ではない)。100 語で 20 分以上。

一方、Task ツールでサブエージェントに 100 語を 1 プロンプトで渡せば、推論
1 回で数秒。CLI 起動なし、API 呼び出し 1 回で済む。

### サブエージェント判定プロンプト (テンプレート)

```
あなたは Nablarch フレームワークの知識検索インデックスを審査する判定者です。

## 背景
Nablarch 日本語ドキュメントから生成された知識ベースがあり、各ページには
ID とタイトルが付いています。ユーザー（Nablarch を使う日本人開発者）の
自然言語の質問に対し、下流の AI エージェントが index-llm.md を読んで
関連ページを最大10件選びます。index-llm.md の各行は次の形式です。

  [page_id] ページタイトル — キーワード1 / キーワード2 / ...

あなたの役割は、与えられた「候補キーワード」群をこのキーワード欄に
載せるべきか（○）載せないべきか（×）を独立に判定することです。

## 判定基準

○（採用）を付ける条件は、以下を **すべて** 満たすこと：
- (A) 開発者が Nablarch に関する質問を書くときに、その語または
  ごく近い語（表記ゆれ・略称・対応する英語/カタカナ）を使う
  可能性が現実的にある
- (B) その語が、提示された上位ページの主題と意味的に結びついている
  （タイトルやページ ID から読み取れる範囲で判断）
- (C) その語が過度に汎用ではない。df_pct が 20% を超える語は
  原則 × にする（ただし Nablarch 中核機能名は例外）

× を付ける代表パターン：
- 文書メタ語／自己参照：「本機能」「記載」「以下」「上記」「補足」「本ハンドラ」
- 文書構造の副産物：「見出し一覧」「外部サイト」「リンク先」
- 過度に広い一般語：「設定」「使用」「方法」「対応」「処理」
- ページ主題とほぼ無関係（スコア上位でも文脈的に的外れ）

## 判定のコツ

- 各語には max_score, df_pct, 上位3ページの id と タイトル と セクションタイトル、
  snippet（本文中の該当箇所）が与えられる
- 上位ページのセクションタイトルに語が含まれるなら、主題との結びつきが明確
- snippet の言い回しを reason に引き写すな。語そのものの性質を評価せよ
- 特定のベンチマーク質問に合わせて採否を決めないこと。抽象的な
  「実在する開発者が Nablarch の機能について自然に書きそうな問い」
  という基準で判断する

## 入力

以下は判定対象の語の配列です:

{{TERMS_JSON}}

## 出力

各語について、以下の JSON 配列を返してください:

[
  {"term": "...", "verdict": "○" または "×",
   "reason": "1文。どの基準 (A)(B)(C) または ×パターンに該当するかを明示",
   "confidence": "high" / "medium" / "low"},
  ...
]

迷ったら confidence を medium/low にしてよいが、verdict は必ず
○ か × のいずれかに確定させること。「場合による」「文脈次第」
といった保留は不可。JSON 配列のみを返し、前後に説明文を置かないこと。
```

## 採用リスト・ストップリストの管理

### 保存先 (PR 確定時)

採用リストと Java ストップリストは `classify_terms.py` のコード内に
定数として保持する。将来 YAML / JSON に切り出してもよい。

- `JAVA_STOPLIST` — Java 標準クラス名 (バージョン横断で共通)
- `MANUAL_ALLOWLIST_JA` — 人間判定で採用した日本語・漢字語
  (多くはバージョン横断で共通、一部はバージョン固有)

### 他バージョンへの展開

**共通して流用できるもの:**

- `JAVA_STOPLIST` — Java 言語の標準クラス名、バージョン不変
- `MANUAL_ALLOWLIST_JA` の大部分 — Nablarch コア概念 (「コミット」「バリデーション」
  「セッションストア」「トランザクション」等) はバージョン間で共通

**バージョン固有のため再判定が必要なもの:**

- `@Annotation` / CamelCase / camelCase — パターン自動許可なので自動で再抽出される
- 一部の日本語語 — Jakarta EE 移行関連など v5→v6 固有のもの

**展開手順:**

1. 既存の `JAVA_STOPLIST` / `MANUAL_ALLOWLIST_JA` をそのまま利用
2. `classify_terms.py --version 5` を走らせる
3. パターン自動許可は自動で反映される
4. pending で 0.30 以上のうち、許可リストにまだ無い語だけを人間判定
5. バージョン固有で落とすべき語があればストップリストに追加

## term_queries (機械抽出、本文 grep 経路)

`search_ids.py` は AI-1 の `selections` 応答に加え、質問文から機械抽出した
**識別子** で本文 grep を行う。抽出対象は ASCII のみ:

| パターン | 例 |
|---|---|
| `@Annotation` (4+ 字) | `@Published` `@UseToken` |
| CamelCase 2-hump (4+ 字) | `TransactionManagementHandler` |
| camelCase (4+ 字) | `connectionFactory` `transactionName` |

**日本語は対象外**。実測ですべて次のいずれかに落ちるため:
- 質問の複合語 (`明細レコード` `夜間バッチ`) は本文に出ない (表記ズレ)
- 抽出できた短い語 (`レコード` `チェック` `クライアント`) は generic すぎて
  noise にしかならず、per-term/total cap を食い尽くす

日本語概念語は **index-llm.md の section-level keyword** で扱う
(役割分担を崩さない)。

term_queries は AI-1 に生成させず **スクリプトが機械抽出する**。これで:
- AI-1 のプロンプトから term_queries 生成タスクを削除できる
- 決定的で再現性が高い
- AI-1 が抽出語を選ぶ際のバイアスが入らない

grep 側のフィルタ:
- df_pct が 20% を超える語は grep 対象から外す (候補爆発を防ぐ)
  ASCII 識別子のみなので v6 では 2 語 (`artifactId`, `groupId`)
- per-term ヒット上限、total ヒット上限で selections に merge する量を制御

## 注意事項

- **ベンチマークシナリオに合わせて採用語を選ばない** (出来レース防止)。
  「現場の開発者が質問しそうか」という抽象基準で判断する
- **hints フィールドは使わない**。knowledge JSON にはセクションごとの `hints`
  があるが、これはベンチマークシナリオに合わせて調整された疑いがあり、
  現場の質問には効かない可能性がある
- **ページ単位 TF-IDF を採用**。セクション単位は短い本文で TF-IDF が不安定
  なので、ページ単位で計算した語を index に付ける。セクション単位の
  enrichment は別途検討 (エキスパート推奨の段階導入)
- **index-llm.md と term_queries の役割分担を崩さない**。同じ語を両方に
  入れると冗長で、AI-1 の attention を分散させる
