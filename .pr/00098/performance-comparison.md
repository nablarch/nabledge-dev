## Baseline Results (Old Workflows)

### Overall Statistics

| Metric | Value |
|--------|-------|
| Scenarios tested | 5 (knowledge-search only) |
| Detection rate | 93.9% (31/33 items) |
| Average tool calls | 7.6 calls |
| Average execution time | 73.1 seconds |
| Time range | 14-121.9 seconds |
| Tool call range | 3-10 calls |

### Individual Scenario Results

| Scenario | Question | Detection | Time | Tool Calls |
|----------|----------|-----------|------|------------|
| ks-001 | バッチの起動方法を教えてください | 7/7 (100%) | 88.6s | 7 |
| ks-002 | UniversalDaoでページングを実装したい | 4/5 (80%) | - | 3 |
| ks-003 | データリードハンドラでファイルを読み込むには？ | 7/7 (100%) | 68s | 9 |
| ks-004 | バッチのエラーハンドリングはどうすればいいですか？ | 6/6 (100%) | 121.9s | 9 |
| ks-005 | バッチアクションの実装方法は？ | 7/7 (100%) | 14s | 10 |

### Key Findings

**Strengths:**
- High detection accuracy (93.9%)
- L1/L2 keyword strategy provides flexible search
- Section judgement effectively filters relevance
- Early termination reduces unnecessary reads

**Weaknesses:**
- Large variance in tool calls (3-10, 3.3x difference)
- Large variance in execution time (14-122s, 8.7x difference)
- Section content reading is sequential (bottleneck at 21.2% of time)
- Early termination may miss context sections (ks-002: 80% detection)

**Bottlenecks:**
- Step 6 (Section content reading): 21.2% of total time
- Sequential section reads limit parallelization

## New Workflow Architecture

### Design Goals

1. **Fallback Strategy**: Full-text search → Index-based search
2. **Unified Section Judgement**: Common workflow for both routes
3. **Batch Section Reading**: One tool call for multiple sections
4. **5-Field Index**: Extended index.toon with section-level hints

### Expected Improvements

**From Design Document (§7.2):**
- **Route 1 (Full-text search hits)**: 5-8 tool calls
- **Route 2 (Index-based fallback)**: 7-12 tool calls
- **Section reading**: Batch mode via read-sections.sh

**Improvement Hypotheses:**
1. **20% faster via batch section reading** - Reduce sequential bottleneck
2. **Better context coverage** - Improved early termination logic
3. **Higher accuracy** - Section-level hints in knowledge files

## Knowledge File Format Issue

### Problem

**Old format** (existing 17 files):
```json
{
  "sections": {
    "overview": [
      {"type": "paragraph", "content": "..."},
      {"type": "code", "language": "java", "content": "..."}
    ]
  }
}
```

**New format** (design document):
```json
{
  "sections": {
    "overview": "# Overview\n\n後続のハンドラ及びライブラリで使用する..."
  },
  "index": [
    {"id": "overview", "title": "概要", "hints": ["..."]  }
  ]
}
```

### Impact

- `full-text-search.sh` expects `.sections | to_entries[] | select(.value | test(...))`
- Old format: `.value` is array → `test()` fails
- New format: `.value` is string → `test()` works

### Resolution

Wait for new knowledge files to be generated by nabledge-creator and merged to main. The work instruction document (§2) explicitly states:

> 今回の作業では知識ファイル（JSON）の作成は行わない。別ツール（nabledge-creator）で生成する。
