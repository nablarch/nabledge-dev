## 🔬 仮説と改善提案

### 仮説 1: `n:select` は意図的な省略ではなくプロンプト誘導の問題

**根拠**: v1.3 の知識ファイルには `n:select` が存在する。しかし質問「コードリストのプルダウン入力を実装するには？」に対して、skill は `n:codeSelect` を主軸に回答を構成し、一般的な `n:select` タグへの言及を省略する。

**検証方法**: qa-001 シナリオの期待値を v5/v6 同様に `n:codeSelect`/`codeId` のみに変更し、`n:select` を期待値から削除した場合の検出率変化を確認する（100% になるはず）。

**期待アウトカム**: 期待値修正により qa-001 が 100% 達成。v1.4 ベースラインでも同様のパターン（75%）なので一括修正が適切。

### 仮説 2: `TransactionAbnormalEnd`/`ParameterizedSqlPStatement` は Nablarch Usage に追加できる

**根拠**: ca-002 の生成ドキュメントにはこれらのクラスへの言及が複数箇所に存在する（Processing Flow、Component Summary）が、`## Nablarch Framework Usage` の `###` セクションとして独立していない。

**検証方法**: CA テンプレートまたは workflow prompt に「エラー処理系クラス（TransactionAbnormalEnd 等）も Nablarch Framework Usage に `###` セクションとして記載すること」という指示を追加して再ベースライン。

**期待アウトカム**: ca-002 が 27/34 → 29/34 (85.3%) に改善。

### 仮説 3: CA Overview の詳細度は workflow prompt で制御可能

**根拠**: ca-002 の Overview には `B11AC014Action` と `FileBatchAction` しか言及されず、`ValidatableFileDataReader`/`BusinessDateUtil`/`UserInfoTempEntity` が省略される。一方 ca-001（web 系）の Overview 検出率は高い。

**検証方法**: workflow の Overview セクション指示に「主要な依存クラス・エンティティ名を明示すること」を追加し、変化を確認。

**期待アウトカム**: Overview の 3 未検出項目のうち 1-2 項目が改善し、ca-002 が 79.4% → 82-85% 程度に向上。
