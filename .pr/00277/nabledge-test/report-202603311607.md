## 💡 主要な発見

### 1. ca-002 の系統的な低スコア（67.6%）とファイルパス問題

ca-002「B11AC014Action」は全3試行で67.6%（23/34）と完全に一致。これは全3エージェントが同一の出力ファイルを参照したため（並列実行で同じパスに上書き）。検出できないのは：
- Overview: `ValidatableFileDataReader`/`BusinessDateUtil`/`UserInfoTempEntity`（高レベル要約で省略）
- Processing Flow: `getValidatorAction`（概念的記述にとどまり、メソッド名が出ない）
- Nablarch Framework Usage: 全5見出し（`###` ヘッダーとして独立セクション化されない）

また、エージェントはターゲットファイルを `nablarch-example-batch/` から見つけられず、`tutorial/.../B11AC014Action.java` を代替使用した。これはファイルパスの変更が原因と考えられる。

### 2. ca-001 の Nablarch Framework Usage 見出し問題

ca-001（31/36 = 86.1%）の未検出5項目はすべて Nablarch Framework Usage の `###` 見出し（DbAccessSupport, OnDoubleSubmission, MailRequester, MessageSender, SyncMessage）。これは v5 および前回 v1.4 ベースラインとも一致する既知パターン。v1.4 スキルは Nablarch 利用を一括記述するが、各クラスを独立した `###` セクションとして出力しない。

### 3. qa-001 の `n:select` 検出不安定（75%/100%/75%）

qa-001「コードリストのプルダウン入力」はベンチマーク3試行で 75%/100%/75%（mean=83.3%）。試行2では `n:select` が検出されたが試行1/3では未検出。v1.4 スキルは `n:codeSelect` を主軸に回答し、汎用セレクトタグへの言及は試行依存。

## 🔬 仮説と改善提案

### 仮説1: B11AC014Action ファイルパス不一致が ca-002 精度低下の根本原因
**証拠**: 全3エージェントが `nablarch-example-batch/` パスで失敗し、`tutorial/...` の代替ファイルを使用。前回ベースライン（branch: 125-..）では 82.4% だったが今回は 67.6% に低下。
**検証方法**: `.lw/nab-official/v1.4/` 配下で `B11AC014Action.java` の実際の場所を確認する。
**期待される結果**: 正しいパスを scenario.json に設定するか skill でパス探索を改善することで前回水準（82.4%）を回復できる。

### 仮説2: Nablarch Framework Usage の `###` 見出し問題はプロンプト改善で解決できる
**証拠**: v1.4/v5 の全 CA シナリオで一貫して発生。CA ワークフローがNablarch利用を箇条書きまたはサブセクションとして記述し、`### ClassName` 形式を使用しない。
**検証方法**: code-analysis ワークフローの「Nablarch Framework Usage」生成ガイドラインに「各クラスを `### ClassName` 見出しで独立したセクションとして記述すること」を追加。
**期待される結果**: ca-001 が 86.1% → 100%（5項目改善）、ca-002 が 67.6% → 82%+ に改善。
