## 🔬 仮説と改善提案

### 仮説 1: qa-001 の `n:select` は期待値として不適切

**根拠**: `n:codeSelect` がコードリスト専用タグ、`n:select` は任意リスト用。「コードリストのプルダウン」質問に対して `n:select` は回答に出ないのが正しい。RST にも `n:select` は存在するが、別トピック（カスタムリスト表示）。
**検証**: 他バージョン（v5, v6）の qa-001 と比較し、同様に `n:select` が未検出か確認。
**提案**: 期待値から `n:select` を除外し、代わりに `コードID` や `CODE_PATTERN` などより的確なキーワードを追加。

### 仮説 2: ca-002 の Processing Flow メソッド名省略は設計上の問題

**根拠**: スキルの Processing Flow 生成テンプレートが概念説明優先でメソッド名を省略する傾向。`doData`, `initialize` は出るが `doHeader`, `doTrailer`, `doEnd` は出ない。
**検証**: v1.4 ca-002 の実際の baseline response を参照し、同じ省略が発生しているか確認。
**提案**: ca-002 シナリオの期待値を更新 — Processing Flow から `doHeader`/`doTrailer`/`doEnd`/`getValidatorAction` を削除し、代わりに `doData`, `initialize` などで確認する期待値に変更。

### 仮説 3: Nablarch Usage の `TransactionAbnormalEnd` / `ParameterizedSqlPStatement` は見出しレベルで出ない

**根拠**: スキルが主要 Nablarch クラスのみを独立セクション見出しとして扱い、補助クラスは本文言及にとどめる。
**検証**: 生成された output ファイルを確認し、見出しになっているクラスの選定基準を分析。
**提案**: 期待値の `nablarch_usage` から `TransactionAbnormalEnd`, `ParameterizedSqlPStatement` を削除し、見出しになる可能性が高い `ValidationUtil` などに変更。
