## 💡 主要な発見

### 1. qa-001 が唯一の低検出シナリオ（37.5%）

全10シナリオのうち qa-001「バッチの起動方法」のみが低検出（trial 1: 37.5%、ベンチマーク平均: 55.0%）となった。検出できなかったキーワードは「コマンドライン引数」「アクションのクラス名」「リクエストID」「javaコマンド」「ハンドラキュー」の5つ。これらはすべて Nablarch バッチ起動の具体的な手順・用語であり、nabledge-5 の知識ファイルにこの粒度の情報が不足している可能性がある。また、ベンチマーク std=0.188 と全シナリオ中最大の分散があり、応答品質が大きく安定しない。

### 2. CA シナリオは QA より安定して高い検出率（98.3% vs 86.1%）

5つの CA シナリオは平均 98.3%（ca-003: 96.8%、ca-005: 94.3%、他3件: 100%）と高水準で安定している。一方 QA は qa-001 の低検出が引き下げ、平均 86.1% となった。CA シナリオはコード解析ワークフローが明確に構造化されており（クラス図・シーケンス図・Nablarch Framework Usage など固定セクション）、知識検索結果の良し悪しに関わらず必要な情報が出力に含まれやすい設計が機能している。

### 3. CA ベンチマークに共通する弱点: Processing Flow の SessionUtil/BeanUtil

ca-004（平均 91.5%）・ca-005（平均 90.6%）のベンチマークにおいて、Processing Flow セクションで `SessionUtil` や `BeanUtil` をクラス名として明示しないケースが頻発する。エージェントが「セッションストア」「フォームをコピー」のように機能を日本語で説明する場合に未検出となり、`SessionUtil.delete()` や `BeanUtil.copy()` のようにメソッド呼び出し形式で記述した場合のみ検出される。Overview セクションの method name 明示（`confirmUpdate`/`update`）にも同様の揺らぎがある（trial 1: 5/5、他多くの trial: 2-3/5）。

## 🔬 仮説と改善提案

### 仮説1: qa-001 の低検出は知識ファイルの記述粒度不足が原因

**根拠**: バッチ起動に関する missing keywords（`コマンドライン引数`、`javaコマンド`、`ハンドラキュー`）はすべて起動コマンドの具体的な書き方に関するもの。nabledge-5 の知識ファイルを確認すると、バッチ起動は概念的に説明されているが具体的な `java -jar` コマンド例や引数仕様が含まれていない可能性がある。

**検証方法**: `.claude/skills/nabledge-5/` 以下のバッチ関連知識ファイルを検索し、「コマンドライン引数」「requestPath」「javaコマンド」「ハンドラキュー」のキーワードが含まれるかを確認する。

**期待効果**: 起動コマンドの具体例・引数仕様を知識ファイルに追加することで qa-001 の検出率を 80%+ に改善できると予想される。std も下がり安定性が向上する見込み。

### 仮説2: CA Processing Flow での utility class 名未記述はテンプレートの誘導不足

**根拠**: Processing Flow セクションで BeanUtil/SessionUtil がクラス名として記述されるかどうかは、エージェントの裁量に依存している。trial 3（ca-004）では `BeanUtil.copy` という明示的記述で検出でき、trial 6（ca-005）では `BeanUtil#createAndCopy`・`SessionUtil#delete` の記述で 5/5 を達成した。一方、他の多くの trial では日本語での機能説明に留まった。

**検証方法**: code-analysis ワークフローのテンプレートまたはガイドに「Processing Flow では使用する Nablarch ユーティリティクラス名（BeanUtil, SessionUtil 等）を明示すること」という指示を追加し、再ベンチマークを実施する。

**期待効果**: ca-004/ca-005 の Processing Flow 検出率が平均 3-4/5 → 5/5 に改善し、ベンチマーク mean が 0.94+ に向上する見込み。
