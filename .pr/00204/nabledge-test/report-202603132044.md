## 💡 主要な発見

### QA検出率の二極化

QAシナリオは77.3%（17/22）とCAシナリオ97.1%（67/69）の間に大きな差がある。これはQAの構造的な問題を示している。qa-001（コードリスト）が2/5=40%と最低で、`n:optionItems`・`コードリスト`・`listId` の3キーワードを一貫して検出できない。nabledge-5のコード機能ナレッジが`n:codeSelect`/`codeId`中心で書かれており、`n:select`のプルダウン向け属性名（listId/optionItems）がカバーされていない可能性がある。

### CA品質の安定性

ca-001（ProjectAction）が37/37=100%、ca-002（ImportZipCodeFileAction）が30/32=93.8%と、CAシナリオは高い検出率を達成した。ca-002で一貫して欠落するのは `Processing Flow includes 'createReader'` と `Sequence diagram includes message 'createReader'` の2項目のみで、`createReader`はコンポーネント説明に記載されるが処理フロー/シーケンス図では省略されるパターンがある。これはドキュメントテンプレートのProcessing Flow節がバッチ実行フロー（`handle`）を中心に記述する設計になっているためと考えられる。

### 実行時間：QA vs CA

QAの平均48秒に対し、CAは平均278秒で約5.8倍の差がある。CA内でもca-001の「Build and write complete documentation」ステップが単独で220秒（79%）を占めており、ドキュメント生成が圧倒的なボトルネックになっている。qa-001のトークン推定値が0となっているのはagentのmetrics形式が非標準だったための推定エラーで、実際の実行時間44秒は正常範囲。

## 🔬 仮説と改善提案

### 仮説1: qa-001の低検出率はナレッジファイルのカバレッジ不足に起因する

**根拠**: 3試行全てで `n:optionItems`・`コードリスト`・`listId` が未検出（0%）。`n:select`・`codeId` は100%検出されている。
**実装参照**: `.claude/skills/nabledge-5/knowledge/component/libraries/libraries-tag.json` の `n:select` 節と `n:codeSelect` 節の記述密度を比較。
**検証方法**: `grep -r "optionItems\|コードリスト\|listId" .claude/skills/nabledge-5/knowledge/` でナレッジファイル内の出現頻度を確認。
**予測**: `n:select` の `listName`/`elementLabelProperty` は記載されているが `optionItems`/`listId` の記載が薄いと予想される。

### 仮説2: ca-002の `createReader` 欠落はテンプレート設計の問題

**根拠**: Processing Flowセクションにおいて、バッチフレームワークが `DataReadHandler` → `read()` のフローを説明するが、アクションの `createReader()` 呼び出しは「コンポーネント詳細」節にのみ記述される。
**実装参照**: `.claude/skills/nabledge-5/assets/code-analysis-template.md` の Processing Flow セクションのガイダンス。
**検証方法**: テンプレートの Processing Flow ガイダンスに「バッチアクションでは `createReader()` を含めること」の指示があるかチェック。
**予測**: テンプレートにその指示がない（または `handle()` フローのみが例示されている）ため、全エージェントが同じパターンで省略している。
