## 🔬 仮説と改善提案

### 仮説1: ca-005の質問文が「実装方法を教える」用途に解釈されやすい

**根拠**: ca-001「ExportProjectsInPeriodActionのコードを理解したい」、ca-004「ProjectCreateActionの実装を理解したい」は一貫してコード分析にルーティングされた。ca-005「ProjectUpdateActionの実装を**理解したい**」は同じ構文だが、ProjectUpdateActionというクラス名がSampleProjectのサンプルコードに実在するため、「実装パターンの説明」と「既存コードの分析」が混在しやすい。

**検証方法**: SKILL.mdのコード分析ルーティング条件を確認し、「理解したい」でトリガーされる条件がca-001〜004と一致するか比較する。

**期待効果**: ルーティング条件に「Actionクラス名 + 理解したい」を明示的にコード分析パターンとして追加することで試行安定性が向上し、平均47.1%→90%以上が見込まれる。

### 仮説2: UniversalDaoはProjectServiceを通じた間接依存のため見出し生成対象から除外されている

**根拠**: コード分析ワークフローはアクションクラスが直接importしているNablarchクラスをNablarch Framework Usageセクションに記述する設計になっていると推測される。ProjectUpdateAction/ProjectCreateActionはUniversalDaoをProjectService経由でのみ使用しており、直接依存としてカウントされていない。

**検証方法**: コード分析ワークフロー（`.claude/skills/nabledge-6/workflows/code-analysis.md`）の「Nablarch Framework Usage」生成ルールを確認し、直接import vs 間接依存の扱いを検証する。

**期待効果**: ガイドラインに「サービス経由でも重要なNablarchクラスは記述する」と明記することで、UniversalDao検出率が0%→80%以上に改善される見込み。
