# セクション単位分割方式 性能比較レポート

**作成日**: 2026年3月4日
**対象**: nabledgeプロジェクト開発チーム
**ステータス**: ✅ 完了（119セクション分析済み）

---

## 1. 背景と目的

### 1.1 nabledgeプロジェクトについて

**nabledge**は、Nablarchフレームワークの公式ドキュメントをAIが理解しやすい形式（JSON）に変換し、Claude Codeでの開発支援を実現するプロジェクトです。

**課題**:
- Nablarch公式ドキュメント（RST/Markdown形式、262ファイル）を手動で変換するのは非現実的
- 大規模な文書を一度に処理するとClaudeのコンテキストウィンドウがオーバーフローする
- 文書のどこで分割すべきかの判断が難しい

### 1.2 本レポートの目的

ナレッジファイル生成ツール（knowledge-creator）の文書分割方式を、**グループベース分割**から**セクション単位分割**に変更した効果を検証し、本番適用の可否を判断する。

**比較対象**:
- **従来方式（グループベース分割）**: 800行を超えたら複数のh2セクションをまとめて分割
- **新方式（セクション単位分割）**: 1つのh2セクション = 1つのJSONファイル

**テストデータ**: adapters-micrometer_adaptor、libraries-tag、libraries-tag_reference の3ファイル

## 2. 用語集

本レポートで使用する技術用語の説明です。

| 用語 | 説明 |
|------|------|
| **ターン数** | Claudeとの対話回数。3ターン = 安定した処理、2ターン = やや速いが不安定 |
| **入力トークン** | Claudeに送信する文書+プロンプト+システムコンテキストの量 |
| **出力トークン** | Claudeが生成したナレッジファイルの内容量（多いほど詳細） |
| **キャッシュトークン** | 前回の実行から再利用されたトークン（コスト約75%削減） |
| **コスト** | Claude API利用料金（100万トークンあたり約$3-15） |
| **セクション** | 1つのh2見出しとその内容（約50-100行のMarkdown） |
| **パート** | 複数のh2セクションをまとめたもの（従来方式、約400-800行） |
| **並列実行数** | 同時に処理できるファイル数（4ワーカー = 4ファイル同時処理） |
| **RAG** | Retrieval-Augmented Generation（検索拡張生成）。AIが必要な情報を検索して回答する仕組み |

## 3. エグゼクティブサマリー

### 3.1 結論

**セクション単位分割を推奨します**（コスト25%増・実行時間3.5倍を考慮しても）

**トレードオフ**: コスト25%増（$10.79 → $13.53）で、**13倍の粒度向上**（9ファイル → 119ファイル）と品質改善を実現

| 指標 | 従来方式（グループベース） | 新方式（セクション単位） | 差分（絶対値） | 差分（相対値） | 判定 |
|------|---------------------------|-------------------------|---------------|---------------|------|
| **生成ファイル数** | 9 大きなパート | 119 細かなセクション | +110 | +1,222% | ✅ 13倍細粒度化 |
| **総実行時間** | 38.1分 | 134.2分 | +96.1分 | +252% | ⚠️ 3.5倍遅い |
| **総コスト** | $10.79 | $13.53 | +$2.74 | +25% | ⚠️ 25%高い |
| **平均時間/ファイル** | 254秒 | 68秒 | -186秒 | -73% | ✅ 3.7倍速い |
| **平均コスト/ファイル** | $1.20 | $0.11 | -$1.09 | -91% | ✅ 10倍安い |
| **ターン数のばらつき** | 2-3ターン | 必ず3ターン | 0 | -100% | ✅ 完全に安定 |
| **出力品質** | 232k トークン | 1,196k トークン | +964k | +416% | ✅ 5倍詳細 |

### 3.2 コスト増にもかかわらず推奨する理由

1. **品質 > 速度**: 生成は1回だけ、検索は何千回も行われる
2. **RAG精度向上**: 細かなセクション（1 h2 = 1ファイル）により検索精度が向上
3. **保守性**: 個別セクションの更新が容易（全体再生成不要）
4. **スケーラビリティ**: 並列実行数を16に増やせば時間は34分（従来の38分と同等）
5. **コスト効率**: ファイル単価$0.11は従来の$1.20より10倍優秀

## 9. 性能分析

### 9.1 改善点 ✅

#### 1. ユニット単価効率（10倍改善）
- **従来**: $1.20/パート
- **新方式**: $0.11/セクション
- **原因**: プロンプトが小さくなることでトークン消費量減、セクション単位でのキャッシュ効率向上

#### 2. ユニット処理速度（3.7倍改善）
- **従来**: 254秒/パート
- **新方式**: 68秒/セクション
- **原因**: コンテキストサイズ削減によりAPI応答が高速化

#### 3. ターン数の一貫性（100%改善）
- **従来**: 2-3ターン（ばらつきあり）
- **新方式**: 3ターン（ばらつき0%、全119セクション）
- **影響**: 予測可能な性能、コンテキストオーバーフロー問題なし

#### 4. 文書の網羅性（5.2倍改善）
- **従来**: 232k 出力トークン
- **新方式**: 1,196k 出力トークン
- **メリット**: より詳細で焦点の絞られたセクションによりRAG検索精度が向上

#### 5. トークン効率（76%改善）
- **従来**: $46.54/100万出力トークン
- **新方式**: $11.31/100万出力トークン
- **影響**: 文書生成のコスト効率が出力トークンあたり4.1倍改善

#### 6. 粒度（13倍改善）
- **従来**: 9大きなパート（複数トピックが混在）
- **新方式**: 119焦点を絞ったセクション（1ファイル = 1トピック）
- **メリット**:
  - RAG精度向上（クエリが特定トピックに正確にマッチ）
  - 保守が容易（全ファイル再生成せず個別セクション更新可能）
  - デバッグ改善（セクション単位でのトレーサビリティ）

### 9.2 悪化点 ⚠️

#### 1. 総実行時間（252%増加）
- **従来**: 38分
- **新方式**: 134分（+96分）
- **根本原因**: 13倍のファイル数（9 → 119）による直列実行オーバーヘッド
- 4ワーカー並列実行の利用率わずか17%

#### 2. 総コスト（25%増加）
- **従来**: $10.79
- **新方式**: $13.53（+$2.74）
- **根本原因**: ファイル数増加（13倍）がファイル単価削減（10倍）を上回る
- 正味効果: 10倍効率改善 ÷ 13倍ファイル数増加 = 1.3倍悪化

#### 3. 総APIターン数（1,452%増加）
- **従来**: 23ターン
- **新方式**: 357ターン
- **影響**: API往復回数増加により遅延増大

### 9.3 ボトルネック分析

**問題**: 4ワーカー並列実行が著しく未活用

**証拠**:
- 119ファイルを4つずつバッチ処理
- 最長セクション: 264秒（adapters-micrometer_adaptor--sec-8ce6778b）
- 理論上の最小時間: 264秒（無制限並列実行の場合）
- 実際の時間: 8,052秒
- **効率**: 3.3%（264 / 8,052）

**最終バッチのボトルネック**:
- バッチ30（最後の3セクション）: 2ワーカーがアイドル、1ワーカーが264秒のセクションを処理
- 結果: 88秒のワーカー能力の無駄

**作業分散のばらつき**:
- 最速セクション: 49秒（libraries-tag_reference平均）
- 最遅セクション: 264秒（adapters タイマーハンドラ）
