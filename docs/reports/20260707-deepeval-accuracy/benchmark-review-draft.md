## 概要

nabledge-6 は Nablarch 6 フレームワークに関する質問に回答する AI アシスタント（Claude Code スキル）である。知識ファイルをエージェンティック検索して回答を生成する。回答品質の客観的な担保手段として DeepEval による自動評価ベンチマークを導入し、34 シナリオ × 3 run（計 102 件）の実行結果を手動で検証した。本レポートは、その検証結果をもとに現行ベンチマーク設計の妥当性を評価し、エキスパートとの認識合わせを通じて改善アクションを導出することを目的とする。

---

## 評価対象のベンチマーク設計

[ベンチマーク設計書 §DeepEval RAGメトリクス](../../benchmark-design.md#deepeval-ragメトリクス) で定義されたベンチマーク設計を評価対象とする。主要要素は以下の通り。

- **評価指標**: DeepEval の3指標（answer_correctness / answer_relevancy / faithfulness）
- **評価方式**: LLM-as-Judge（ジャッジ LLM: Amazon Bedrock 上の Claude Sonnet 4.6）
- **閾値**: answer_correctness = 0.99、answer_relevancy = 0.95、faithfulness = 0.99
- **規模**: 34 シナリオ × 3 run = 102 件
- **シナリオ設計**: 質問・期待事実（`must.facts`）・参照セクションを宣言的に定義
- **シナリオ定義**: [ベンチマークシナリオ](../../../tools/benchmark/scenarios/qa.json)（[ベンチマーク設計書 §シナリオ定義](../../benchmark-design.md#シナリオ定義)）
- **ベンチマーク結果**: [ベンチマーク精度検証レポート](result.md)

指標選定は nabledge-6 の目的（Nablarch 実装支援）とエキスパートへの相談をもとに決定した。

## 指標・閾値の妥当性

### 指標選定

**3指標の役割分担**:

| 指標 | DeepEval クラス | 評価の焦点 |
|------|-----------------|------------|
| answer_correctness | `GEval` | 回答が期待事実（must.facts）を網羅しているか |
| answer_relevancy | `AnswerRelevancyMetric` | 回答が質問に対して的外れでないか |
| faithfulness | `FaithfulnessMetric` | 回答の主張が検索コンテキストで裏付けられているか |

nabledge-6 では「正しい情報が含まれているか（correctness）」「的外れでないか（relevancy）」「根拠なく作られていないか（faithfulness）」の3軸でアシスタントの回答品質を評価することにした。

`answer_relevancy` は旧指標では捉えられなかった「的外れな回答」を検出できる点で新たな品質軸を追加している。

[要確認]
以下の理由で3指標を選定しました。選定の妥当性と不足がないかご確認をお願いします。

| 指標 | 選定理由 |
|------|----------|
| answer_correctness | Nablarch 実装支援の核心。期待事実の欠落は誤実装に直結するため、事実網羅性の評価は必須 |
| answer_relevancy | 的外れな回答（質問と無関係な説明）を検出する軸。旧指標では捉えられなかった品質リスクを補完 |
| faithfulness | ハルシネーション検出。根拠のない記述はミッションクリティカル用途では許容できない |

一方、以下は意図的に除外しました:

- **context_precision / context_recall**（検索精度・検索網羅性）: 検索品質ではなく回答品質を評価することを優先したため。ナレッジ検索の品質はキーワード検索ランナーで別途評価している。

この3指標セットで、一般的に使われる組み合わせと比べて不足はないでしょうか。

### 閾値設定

**現在の閾値と設定根拠**:

| 指標 | 閾値 | DeepEval デフォルト | 比率 | 設定根拠 |
|------|------|----------------------|------|----------|
| answer_correctness | 0.99 | 0.5 | 約2倍 | 実装に必要な事実の欠落は誤実装に直結する |
| answer_relevancy | 0.95 | 0.5 | 約1.9倍 | 多少の冗長・脱線は許容するが大きな逸脱は不可 |
| faithfulness | 0.99 | 0.5 | 約2倍 | ハルシネーション（根拠なし記述）は誤実装に直結する |

これらの閾値は、Nablarch が大規模金融システム等のミッションクリティカルな領域で使われているという要件に基づく。誤った実装情報の提供は本番障害に直結するため、DeepEval のデフォルト（0.5）より大幅に高い基準を設定している。

**実証データからの提案閾値**:

手動検証（→ [ベンチマーク精度検証レポート](result.md)）の結果、誤検知が多い2指標について閾値の見直しを検討している:

| 指標 | 現行閾値 | 提案閾値 | 提案根拠 |
|------|----------|----------|----------|
| answer_relevancy | 0.95 | 0.70 | 誤検知率 60.9%（14/23件）。主因は出典メタデータ行が無関係コンテンツと判定されること。閾値を下げても見逃しリスクはなし（見逃し 0/79件）。 |
| faithfulness | 0.99 | 0.90 | 誤検知率 94.3%（33/35件）。主因は要点のみ説明する省略スタイルを DeepEval が矛盾と解釈すること。見逃しは 1.5%（1/67件）と低い。 |
| answer_correctness | 0.99 | 0.99（変更なし） | 誤検知率 88.9%（8/9件）だが、見逃し 0%。正確性の要求が最も高いため現行閾値を維持。 |

[要確認]
answer_relevancy を 0.95 → 0.70 へ引き下げることを提案します。根拠は以下の通りです:

- **誤検知率 60.9%**（14/23件）で、主因が出典メタデータ行（「参照: ファイル名」）の誤判定と特定済み
- **見逃し 0 件**（0/79件）: 0.70 に下げても「的外れな回答」の見逃しリスクは実証データ上ゼロ

この根拠での引き下げについて、実務上の見解をいただけますか。特に「見逃しゼロ」だけでは不十分な理由があればご指摘ください。

[要確認]
0.70 前後の閾値は RAG 評価の実務でも使われる範囲と推測していますが、根拠となる実績値を持っていません。answer_relevancy（または類似指標）の運用閾値について、参考になる実績・設計基準をご教示いただけますか。上記 0.70 提案の妥当性を業界知見と照合したいと考えています。

[要確認]
faithfulness の誤検知（誤検知率 94.3%）への対処として、「閾値を 0.99 → 0.90 へ引き下げる」+ 「DeepEval のプロンプトカスタマイズで省略スタイルを許容する評価基準を明示する」の組み合わせが現時点での最善と考えています。誤検知の主因は「要点のみを説明する省略スタイルを DeepEval が矛盾と解釈する」ことです。

この判断について実務上の見解をご教示いただけますか。特に閾値調整とプロンプトカスタマイズを組み合わせることへのリスクや、より効果的なアプローチがあればご指摘ください。

## DeepEval の信頼性

### 誤検知の傾向

手動検証（→ [ベンチマーク精度検証レポート](result.md)）で確認された主な誤検知パターン:

**answer_relevancy の誤検知（誤検知率 60.9%）**:
- 主因: 回答末尾の出典メタデータ行（「参照: ファイル名」等）を DeepEval が「無関係なコンテンツ」と判定する
- DeepEval は文全体ではなく文の一部を個別のステートメントとして評価するため、出典行単独でスコアを下げることがある

**faithfulness の誤検知（誤検知率 94.3%）**:
- 主因: nabledge-6 の回答スタイル（要点のみを簡潔に説明する省略スタイル）を DeepEval が「retrieval_context に存在しない主張」と解釈する
- 実際にはハルシネーションではなく、正確な情報の要約であるケースが大多数

**answer_correctness の誤検知（誤検知率 88.9%）**:
- 主因は詳細確認中（→ [ベンチマーク精度検証レポート](result.md) の各シナリオの reason フィールドを参照）

[要確認]
answer_correctness の誤検知 8 件について、私たちの仮説は「reason フィールドの自然言語説明が一貫していないため、8 件ではパターン分類に十分なサンプルがない」です。[ベンチマーク精度検証レポート](result.md) の reason フィールドを確認しましたが、明確な共通パターンを特定できていません。

この仮説は正しいでしょうか。また GEval（LLM-as-Judge による事実網羅評価）で誤検知が起きやすいシナリオタイプの傾向（例: 長い回答、複数事実の組み合わせ、手順系の回答 等）があれば、原因特定のアプローチと合わせてご教示いただけますか。

### 見逃しリスク

手動検証では、3指標すべてで見逃し率が低い結果となった:

| 指標 | 見逃し率 | 内訳 |
|------|----------|------|
| answer_correctness | 0.0% | 0/93件 |
| answer_relevancy | 0.0% | 0/79件 |
| faithfulness | 1.5% | 1/67件 |

「OK は信用に値する、NG は要確認」という運用方針（NG が出た場合に人間が確認する）は、見逃し率の低さから一定の根拠を持つ。

[要確認]
34 シナリオ × 3 run = 102 件という規模で、見逃し率 0〜1.5% という結果が得られています。この結果から「OK は信用に値する」という運用方針を採用したいと考えています。

ただし 102 件というサンプルサイズが「見逃しなし」を主張するのに十分かどうかの確信が持てていません。実務上の観点から:

- このサンプルサイズは「信頼できる」という判断に十分でしょうか?
- 「見逃しなし」を主張するために必要なサンプルサイズや検証方法の業界基準があればご教示いただけますか。

faithfulness の見逃し 1 件（run-2-qa-11）のベンチマークスコアは 1.0 であり、閾値 0.99 → 0.90 変更後も検知されない（閾値変更の影響を受けない）。

## 現アプローチの最適性

### LLM-as-Judge の特性と既知の限界

LLM-as-Judge 方式は、LLM の出力を別の LLM が評価する方式であり、以下の特性を持つ:

**利点**:
- Ground truth の定義が柔軟（厳密な期待回答ではなく「含まれるべき事実」で定義できる）
- 意味的な一致を評価できる（キーワードマッチではなく意味的な正確さを判定）

**既知の限界（研究知見）**:
- **誤検知率の高さ**: LLM ジャッジの誤検知は構造的な課題として知られている。Wang et al. (2023) では position bias による逆転が 82.5% のケースで確認されており、一般的なハルシネーション検出精度は 30〜60% 程度という報告もある（Eugene Yan）
- **ベンチマーク vs 実務の乖離**: 自動評価スコアと実際の業務品質が乖離することが報告されている。コード生成タスクでは合成ベンチマーク 84〜89% に対し実世界コードベースでは 25〜34% という乖離事例がある（arxiv:2510.26130）
- **高リスク用途での失敗**: 教育介入推薦タスクでの研究では G-Eval（DeepEval と同等の仕組み）が業務判断の品質を保証できず 43% の偽陽性を見逃した事例がある（arxiv:2606.29280）。判断の質ではなく流暢さを評価しているため
- **非決定性**: 同一入力でもスコアが変動する（LLM の確率的性質）

nabledge-6 で観測された誤検知率（answer_relevancy 60.9%、faithfulness 94.3%）はこれらの既知の限界の範囲内と見なせる可能性がある。ただしミッションクリティカル用途では許容できない水準であり、改善が必要。

### Ground truth 設計

nabledge-6 では、`must.facts`（回答に含まれるべき事実リスト）+ `section`（参照ナレッジセクション）という設計を採用している。これにより:
- answer_correctness: 期待事実と実際の出力を LLM が比較
- faithfulness: 実際の出力と参照セクション内容を LLM が比較

この設計は「どの情報を参照したか」と「何を出力したか」を分離して評価できる点で、nabledge-6 の品質評価として理にかなっている。

[要確認]
Ground truth として `must.facts`（回答に含まれるべき事実リスト）+ `section`（参照ナレッジセクション）という設計を採用しています。利点として「期待事実を宣言的に定義でき、メンテナンスしやすい」と考えています。

「期待事実を箇条書きで定義する」という方式について、標準的な設計と比較した場合の欠点や盲点（例: 事実の粒度が評価精度に影響する等）があればご教示いただけますか。

[要確認]
ジャッジ LLM として Claude Sonnet 4.6（Amazon Bedrock）を使用していますが、回答生成 LLM も同じモデルです。自己評価バイアス（同一モデルが自身の出力を甘く評価する）のリスクを認識しており、改善案として別モデルへの変更を検討しています。

この同一モデル問題が実務上どの程度深刻か、および現実的な対処策（別モデルの選定基準、代替アプローチ等）について知見をご共有いただけますか。

### 実務品質監視のベストプラクティス（世間の知見との比較）

自動評価だけに依存せず、実務品質を継続的に担保するための多層構造が世間では推奨されている:

| 層 | 手法 | nabledge-6 での対応状況 |
|---|---|---|
| 開発時の自動評価 | LLM-as-Judge（DeepEval 等）| ✅ 実施中（本ベンチマーク） |
| 定期的な人間サンプリング | 50〜100 件を専門家がレビュー（月次/四半期）| ❌ 未実施 |
| 失敗ケースの継続蓄積 | NG シナリオをベンチマークに追加してキャリブレーション | ❌ 未実施 |
| 本番ログ監視 | ユーザーフィードバック・失敗トレースの収集 | ❌ 未実施 |

> 複数の実務ガイド（Hamel Husain, Eugene Yan, Databricks）が独立して「LLM-as-Judge は専門家とのキャリブレーションなしでは信頼できない」「代表サンプルへの人間レビューを定期実施することが前提」という結論に至っている。

[要確認]
自動評価だけでは「ベンチマーク vs 実務の乖離」を検出できないため、定期的な人間サンプリングレビューが必要と認識しています。私たちの初期案: 月次 30〜50 件を開発者 1〜2 名でレビューし、誤検知の多い指標（faithfulness・answer_relevancy）を中心にサンプリングする。

この運用規模は現実的でしょうか。社内向けアシスタントでの実績や、より効果的なアプローチがあればご教示いただけますか。

[要確認]
本番ログ監視（ユーザーフィードバックの収集・失敗トレースの蓄積）は重要と考えていますが、nabledge-6 は Claude Code スキルとして動作しており、標準的な「サーバーログ収集」の仕組みが使えません。私たちの初期案: 回答末尾に「この回答は役に立ちましたか？（はい/いいえ）」というフィードバック依頼を組み込み、会話ログを定期的にエクスポートして集計する。

この方向性は有効でしょうか。Claude Code スキル環境での本番品質監視について、より現実的なアプローチがあればご教示いただけますか。

## 改善アクション

### 実証データから直接導ける改善

1. **answer_relevancy 閾値の調整**: 0.95 → 0.70 への変更（誤検知率 60.9% の主因が出典メタデータ行であることが特定されているため）
2. **faithfulness 閾値の調整**: 0.99 → 0.90 への変更（誤検知率 94.3% の主因が省略スタイルの誤解釈であることが特定されているため。ただし faithfulness の見逃し 1 件への影響を事前確認要）
3. **誤検知シナリオの reason フィールド分析の継続**: answer_correctness の誤検知 8 件について、原因分類と対処方針の検討
