## 概要

nabledge-6 は Nablarch 6 フレームワークに関する RAG ベースの AI アシスタントである。回答品質の客観的な担保手段として DeepEval による自動評価ベンチマークを導入し、34 シナリオ × 3 run（計 102 件）の実行結果を手動で検証した。本レポートは、その検証結果をもとに現行ベンチマーク設計の妥当性を評価し、エキスパートとの認識合わせを通じて改善アクションを導出することを目的とする。

---

## 評価対象のベンチマーク設計

`docs/benchmark-design.md` の DeepEval セクションで定義されたベンチマーク設計を評価対象とする。主要要素は以下の通り。

- **評価指標**: DeepEval の3指標（answer_correctness / answer_relevancy / faithfulness）
- **評価方式**: LLM-as-Judge（ジャッジ LLM: Amazon Bedrock 上の Claude Sonnet 4.6）
- **閾値**: answer_correctness = 0.99、answer_relevancy = 0.95、faithfulness = 0.99
- **規模**: 34 シナリオ × 3 run = 102 件
- **シナリオ設計**: 質問・期待事実（`must.facts`）・参照セクションを宣言的に定義

指標選定の経緯として、従来の独自 LLM ジャッジ指標（C-claim ジャッジ・ハルシネーションジャッジ）との相関分析を実施し、`answer_correctness` と `accuracy` が 96.4% 一致、`faithfulness` と `hallucination` が 88.5% 一致することを確認した上で、DeepEval 標準指標に完全移行した。

→ 詳細な実証データ（手動検証結果）: [result.md](result.md)

## 指標・閾値の妥当性

### 指標選定

**3指標の役割分担**:

| 指標 | DeepEval クラス | 評価の焦点 |
|------|-----------------|------------|
| answer_correctness | `GEval` | 回答が期待事実（must.facts）を網羅しているか |
| answer_relevancy | `AnswerRelevancyMetric` | 回答が質問に対して的外れでないか |
| faithfulness | `FaithfulnessMetric` | 回答の主張が検索コンテキストで裏付けられているか |

RAG システムの品質は「正しい情報が含まれているか（correctness）」「的外れでないか（relevancy）」「根拠なく作られていないか（faithfulness）」の3軸で捉えるのが一般的であり（RAGAS フレームワーク等も同様の観点で設計されている）、この選定は標準的なアプローチと整合している。

`answer_relevancy` は旧指標では捉えられなかった「的外れな回答」を検出できる点で新たな品質軸を追加している。

[要確認: answer_correctness / answer_relevancy / faithfulness の3指標セットは、大企業社内向け AI チャット開発で一般的に使われる組み合わせと比べて十分か。]

[要確認: context_precision（検索精度）や context_recall（検索網羅性）を今回の設計では外しているが、この判断は妥当か。]

### 閾値設定

**現在の閾値と設定根拠**:

| 指標 | 閾値 | DeepEval デフォルト | 比率 | 設定根拠 |
|------|------|----------------------|------|----------|
| answer_correctness | 0.99 | 0.5 | 約2倍 | 実装に必要な事実の欠落は誤実装に直結する |
| answer_relevancy | 0.95 | 0.5 | 約1.9倍 | 多少の冗長・脱線は許容するが大きな逸脱は不可 |
| faithfulness | 0.99 | 0.5 | 約2倍 | ハルシネーション（根拠なし記述）は誤実装に直結する |

これらの閾値は、Nablarch が大規模金融システム等のミッションクリティカルな領域で使われているという要件に基づく。誤った実装情報の提供は本番障害に直結するため、一般的な RAG システムより大幅に高い基準を設定している。

**実証データからの提案閾値**:

手動検証（→ [result.md](result.md)）の結果、誤検知が多い2指標について閾値の見直しを検討している:

| 指標 | 現行閾値 | 提案閾値 | 提案根拠 |
|------|----------|----------|----------|
| answer_relevancy | 0.95 | 0.70 | 誤検知率 60.9%（14/23件）。主因は出典メタデータ行が無関係コンテンツと判定されること。閾値を下げても見逃しリスクはなし（見逃し 0/79件）。 |
| faithfulness | 0.99 | 0.90 | 誤検知率 94.3%（33/35件）。主因は要点のみ説明する省略スタイルを DeepEval が矛盾と解釈すること。見逃しは 1.5%（1/67件）と低い。 |
| answer_correctness | 0.99 | 0.99（変更なし） | 誤検知率 88.9%（8/9件）だが、見逃し 0%。正確性の要求が最も高いため現行閾値を維持。 |

[要確認: answer_relevancy を 0.95 → 0.70 に下げることで、実務上「的外れな回答」を見逃すリスクをどう評価するか。]

[要確認: 大企業社内向け AI チャット開発での answer_relevancy（または類似指標）の運用閾値について、参考になる実績値や設計基準があれば教えてほしい。]

[要確認: faithfulness の 0.99 → 0.90 への変更について。省略スタイルの回答が誤検知される問題は、閾値調整で対処するのが適切か、それとも DeepEval の設定（プロンプトカスタマイズ等）で対処すべきか。]

## DeepEval の信頼性

### 誤検知の傾向

手動検証（→ [result.md](result.md)）で確認された主な誤検知パターン:

**answer_relevancy の誤検知（誤検知率 60.9%）**:
- 主因: 回答末尾の出典メタデータ行（「参照: ファイル名」等）を DeepEval が「無関係なコンテンツ」と判定する
- DeepEval は文全体ではなく文の一部を個別のステートメントとして評価するため、出典行単独でスコアを下げることがある

**faithfulness の誤検知（誤検知率 94.3%）**:
- 主因: nabledge-6 の回答スタイル（要点のみを簡潔に説明する省略スタイル）を DeepEval が「retrieval_context に存在しない主張」と解釈する
- 実際にはハルシネーションではなく、正確な情報の要約であるケースが大多数

**answer_correctness の誤検知（誤検知率 88.9%）**:
- 主因は詳細確認中（→ [result.md](result.md) の各シナリオの reason フィールドを参照）

[要確認: answer_correctness の誤検知 8 件について、result.md の reason フィールドを確認したが明確なパターンが特定できていない。エキスパートの経験から、GEval（LLM-as-Judge による事実網羅評価）で誤検知が起きやすいシナリオタイプはあるか。また、誤検知原因の特定・対処に推奨するアプローチはあるか。]

### 見逃しリスク

手動検証では、3指標すべてで見逃し率が低い結果となった:

| 指標 | 見逃し率 | 内訳 |
|------|----------|------|
| answer_correctness | 0.0% | 0/93件 |
| answer_relevancy | 0.0% | 0/79件 |
| faithfulness | 1.5% | 1/67件 |

「OK は信用に値する、NG は要確認」という運用方針（NG が出た場合に人間が確認する）は、見逃し率の低さから一定の根拠を持つ。

[要確認: 34シナリオ × 3 run = 102件という規模での見逃し率 0〜1.5% という数値について、実務上の判断として十分な信頼性を持つか。大企業社内向け AI チャットでの知見から、「見逃しなし」を主張するために必要なサンプルサイズや検証方法の基準があれば教えてほしい。]

faithfulness の見逃し 1 件（run-2-qa-11）のベンチマークスコアは 1.0 であり、閾値 0.99 → 0.90 変更後も検知されない（閾値変更の影響を受けない）。

## 現アプローチの最適性

### LLM-as-Judge の特性

LLM-as-Judge 方式は RAG 評価の標準的なアプローチであり、以下の特性を持つ:

- **利点**: Ground truth の定義が柔軟（厳密な期待回答ではなく「含まれるべき事実」で定義できる）
- **利点**: 意味的な一致を評価できる（キーワードマッチではなく意味的な正確さを判定）
- **課題**: ジャッジ LLM 自体の変動性（同一入力でスコアが変動する可能性）
- **課題**: ジャッジ LLM の判定基準がブラックボックスであり、誤検知の原因特定に時間がかかる

今回の実証データで確認された誤検知（faithfulness・relevancy）は、LLM-as-Judge 方式特有の課題（LLM が暗黙の前提を持つ）を示している。

### Ground truth 設計

nabledge-6 では、`must.facts`（回答に含まれるべき事実リスト）+ `section`（参照ナレッジセクション）という設計を採用している。これにより:
- answer_correctness: 期待事実と実際の回答を LLM が比較
- faithfulness: 実際の回答と参照セクション内容を LLM が比較

この設計は「どの情報を参照したか」と「何を回答したか」を分離して評価できる点で、RAG システムの品質評価として理にかなっている。

[要確認: Ground truth（must.facts + section）の定義アプローチは、大企業社内向け AI チャット開発での標準的なアプローチと比較してどうか。特に「期待事実を箇条書きで定義する」という方式の利点・欠点があれば教えてほしい。]

[要確認: ジャッジ LLM として Claude Sonnet 4.6（Amazon Bedrock）を使用している。ジャッジ LLM と回答生成 LLM が同一モデルである点はバイアスになりうるか。実務上の対処策（別モデルを使う等）はあるか。]
