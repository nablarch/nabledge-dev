## RAG 品質評価の基本設計

### 指標の選び方

RAG システムの品質は、最低でも以下の3軸で評価することが推奨される（RAGAS フレームワーク、DeepEval など主要な RAG 評価ライブラリが共通して採用している観点）:

| 軸 | 問う内容 | DeepEval での指標 |
|----|----------|-------------------|
| 正確性（Correctness） | 期待する情報が回答に含まれているか | `GEval`（answer_correctness） |
| 関連性（Relevancy） | 質問に対して的外れな内容がないか | `AnswerRelevancyMetric` |
| 忠実性（Faithfulness） | 回答がコンテキスト外の情報を作り出していないか | `FaithfulnessMetric` |

**nabledge-6 での採用理由**: 従来の独自 LLM ジャッジ指標（C-claim ジャッジ・ハルシネーションジャッジ）との相関分析を実施し、`answer_correctness` と `accuracy` が 96.4% 一致、`faithfulness` と `hallucination` が 88.5% 一致することを確認した上で移行。`answer_relevancy` は旧指標では捉えられなかった「的外れな回答」を検出する新しい品質軸として追加した。

**指標を追加する場合の候補**:
- `context_precision`: 検索されたコンテキストのうち、実際に回答に使われた割合（RAG の検索精度を評価）
- `context_recall`: 期待される情報を検索で取得できているか（検索漏れの検出）

ただし指標を増やすほど評価コストと誤検知の複雑さが増す。最初は3軸から始めて、運用で不足を感じたら追加する方針が現実的。

[要確認: 大企業社内向け AI チャット開発での標準的な RAG 評価指標セットはどのようなものか。]

[要確認: 指標を段階的に追加していくアプローチと、最初から全指標を入れるアプローチのどちらが推奨か。]

### Ground truth（期待事実）の定義

Ground truth の定義は RAG 評価設計の核心であり、最も設計コストがかかる部分でもある。

**nabledge-6 でのアプローチ**:

```json
{
  "then": {
    "must": [
      {"fact": "回答に含まれるべき事実", "section": "path/to/file.json:sN"}
    ]
  }
}
```

- `fact`: 回答に含まれるべき事実を自然文で記述（「〜である」「〜する場合は〜を使う」等）
- `section`: その事実の出所となるナレッジセクション（RAG が参照すべきセクション）

この設計の特徴:
- **柔軟性**: 「回答の完全一致」ではなく「事実の網羅」を評価するため、表現の揺れに強い
- **追跡可能性**: `section` で参照元を明示することで、評価失敗時の根本原因（知識ファイルの問題か、RAG の検索問題か、回答生成の問題か）を切り分けやすい
- **作成コスト**: 1シナリオあたり2〜5個の事実を定義するため、シナリオ設計に相応の工数がかかる

**一般化できるベストプラクティス**:
- 事実は「ユーザーが誤実装・誤判断しうる情報単位」を基準に定義する（網羅しすぎると閾値が厳しくなりすぎる）
- アウトオブスコープシナリオ（「〜はサポート外」という回答が正解）も含めることで、過剰な回答を検出できる

[要確認: 大規模プロジェクトでの Ground truth（期待事実）の定義粒度について、1シナリオあたりの事実数の目安はあるか。]

[要確認: Ground truth の事実定義が「多すぎる・少なすぎる」を判断する基準はあるか。]

[要確認: ドメイン知識が必要な事実定義は誰が担当するのが現実的か（開発者・ドメインエキスパート・レビュー体制など）。]

### 閾値の設定

閾値の設定は「品質要件の定量化」であり、プロジェクトのリスク許容度と直接対応する。

**閾値設定の一般的な考え方**:

1. まず「見逃し率（False Negative Rate）」を先に決める: どの程度の品質問題を見逃せるか
2. 次に「誤検知率（False Positive Rate）」のトレードオフを確認: 閾値が高すぎると誤検知が多くなり、人間レビューの負荷が増大する
3. 実証データで「閾値 X でどれだけの誤検知・見逃しが発生するか」を測定して調整する

プロジェクトの実証データに基づいて見逃し率を確認した上で判断することが原則である。nabledge-6 での具体的な閾値設定根拠・提案閾値・実測値は → [benchmark-review-draft.md](benchmark-review-draft.md) の「閾値設定」セクションを参照。

[要確認: 「見逃しなし」が要件として成立するのは、サンプルサイズ 102 件という規模で妥当か。大企業社内向け AI チャットで閾値を設定する場合、どのような水準を目安とするか（例: ミッションクリティカル用途 / 一般業務用途での違い）。]

## LLM-as-Judge の特性と対処

### 基本的な特性

LLM-as-Judge は「LLM が LLM の出力を評価する」方式であり、RAG 評価では広く使われている。

**利点**:
- Ground truth の定義が柔軟（完全一致ではなく意味的な正確さで評価できる）
- 自然文での事実定義を評価できる
- 人間によるアノテーションに近い評価が可能

**課題**:
- 同一入力でスコアが変動する可能性（LLM の確率的な性質）
- 判定基準がブラックボックスであり、誤検知の原因特定に時間がかかる
- ジャッジ LLM と回答生成 LLM が同一の場合のバイアスリスク

### 誤検知パターンの例

LLM-as-Judge では以下のようなパターンで誤検知が発生することがある:

**1. 出典メタデータ行による relevancy 誤検知**: 回答に付与したメタデータ（出典情報等）を「無関係なコンテンツ」と判定するパターン。対処は閾値の調整または後処理による除外。

**2. 省略スタイルによる faithfulness 誤検知**: 要点のみを簡潔に説明する回答を「コンテキストにない主張」と解釈するパターン。省略は矛盾ではないが、LLM ジャッジが区別できないことがある。

nabledge-6 での具体例（誤検知件数・スコア分布・原因分析）は → [benchmark-review-draft.md](benchmark-review-draft.md) の「誤検知の傾向」セクションを参照。

### 推奨される運用方針

「OK は信用に値する、NG は要確認」という非対称な運用が合理的:

| ジャッジ結果 | 対応 | 根拠 |
|------------|------|------|
| PASS（閾値以上） | 自動承認 | プロジェクトの実証データで見逃し率が低いことを確認した上で適用する |
| FAIL（閾値未満） | 人間が確認 | 誤検知率が高い指標では自動否定は不適切（nabledge-6 での実測値は → [benchmark-review-draft.md](benchmark-review-draft.md)） |

この運用方針は、LLM-as-Judge の「見逃しは少ないが誤検知は多い」という特性を活かしたものである。

[要確認: LLM-as-Judge の誤検知率（answer_relevancy で 60.9%、faithfulness で 94.3%）は、大企業社内向け AI チャット開発での許容範囲に収まるか。]

[要確認: DeepEval の評価プロンプトをカスタマイズすることで誤検知率を下げた実績はあるか。]

[要確認: ジャッジ LLM と回答生成 LLM が同一モデル（Claude Sonnet 4.6）である点のバイアスリスクについて、実務上どう評価するか。別モデルをジャッジに使う場合のトレードオフ（コスト・信頼性・管理コスト）はどうか。]

## サンプルサイズと検証の設計

### nabledge-6 での設計

- **シナリオ数**: 34（ユースケース・カテゴリ・フェーズを網羅）
- **実行回数**: 3 run（LLM の確率的変動を観測するため）
- **総評価件数**: 102 件
- **手動検証**: 全102件を手動で検証（誤検知・見逃しを特定）

34 シナリオの設計基準: nabledge-6 の主要ユースケース（処理方式ごと・ハンドラ種別ごと）を網羅しつつ、実行コストとのバランスを取った規模。

3 run の意義: LLM-as-Judge の変動性（同一シナリオでスコアが変わりうる）を観測し、「ランダムな閾値割れ」と「継続的な品質問題」を区別するため。

## 別プロジェクトへの適用チェックリスト

### このガイドで使った設計の前提条件

nabledge-6 での設計は以下の前提に基づいている。自プロジェクトへの適用前に確認すること:

| 前提条件 | nabledge-6 での状態 | 自プロジェクトでの確認 |
|----------|---------------------|----------------------|
| RAG ベースの Q&A システムである | ✅ | |
| Ground truth（期待事実）をドキュメントから定義できる | ✅（Nablarch 公式ドキュメントが存在する） | |
| 質問が特定ドメインの技術情報に絞られる | ✅（Nablarch フレームワーク） | |
| 回答生成 LLM が Claude 系（Amazon Bedrock）である | ✅ | |
| ミッションクリティカルな業務への適用 | ✅（金融系） | |

### Nablarch 固有の要因 vs. 一般化できる要因

**Nablarch 固有（他プロジェクトでは調整が必要）**:
- 閾値の絶対値（0.99 / 0.95）: Nablarch の要件（誤実装 = 本番障害）から設定。一般的な業務用途では 0.7〜0.8 程度が現実的な出発点になりうる
- シナリオの設計（処理方式別・ハンドラ種別）: ドメイン固有の分類

**一般化できる要因**:
- 3指標の組み合わせ（correctness / relevancy / faithfulness）: RAG 評価の標準アプローチ
- Ground truth 設計（事実 + 参照セクション）: 多くの RAG システムで適用可能
- 「OK は信用に値する、NG は要確認」の運用方針: LLM-as-Judge の特性から導かれる
- 3 run による変動観測: LLM の確率的変動への対処として一般的

[要確認: ミッションクリティカル以外の用途（例: 一般的な社内FAQ、ヘルプデスク支援）で RAG 精度評価を設計する場合、閾値の基準値（出発点）をどう設定するか。実務上の目安があれば教えてほしい。]

[要確認: このガイドの適用範囲として「RAG ベースの Q&A」を想定しているが、RAG 以外の形式（ファインチューニング、Function Calling 中心等）では評価設計がどう変わるか。大企業社内向け AI チャット開発では複数の形式が混在することもあると思うが、評価設計の考え方が根本的に異なる点があれば教えてほしい。]
