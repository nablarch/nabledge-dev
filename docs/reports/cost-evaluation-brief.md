# Nabledge コスト最適化評価 — 評価前提・背景

**作成日**: 2026-06-19
**対象**: Nabledge（`nablarch/nabledge` 配布版 / `nablarch/nabledge-dev` 開発版）

---

## 1. 背景と目的

全社的に開発PJでのAI利用コストが問題視されている。Nabledge についても、現在の作りが **コスト最適化の観点で妥当か** を有識者レビューにかけられる形で評価したい。

最終ゴールは「有識者がフラットにレビューできる評価結果」を出すこと。評価は推測でなく事実ベースで、バイアスなくフラットに行う。

---

## 2. 評価の前提（クローン確認済みの事実）

| 項目 | 事実 |
|---|---|
| 利用想定モデル | AWS Bedrock 上の Claude Sonnet 4.6 |
| 配布形態 | GitHub 経由の Claude Code / GitHub Copilot プラグイン。各PJの開発環境に取り込んで利用 |
| アーキテクチャ方式 | 埋め込みベクトルを使わない、LLM駆動の2段階決定的選定（`index.md`/`classes.md` でページ選定 → 該当 JSON セクションのみ精読）+ スクリプト全文一致のキーワード検索 |
| 知識生成 | RBKC（Rule-Based Knowledge Creator）が公式ドキュメント（RST/Markdown/Excel）から知識JSONを自動生成。verify が品質ゲート |
| 対象バージョン | Nablarch 6/5/1.4/1.3/1.2 |
| 対象処理方式 | Nablarchバッチ、RESTfulウェブサービス（JSR352バッチ・MOM・ウェブアプリは対象外） |

**要件の根拠**: RBKC verify 品質保証設計書（`tools/rbkc/docs/rbkc-verify-quality-design.md`）を中心に、`docs/nabledge-design.md`・`docs/search-design.md`・`docs/benchmark-design.md`・`docs/development-status.md` を参照して Nabledge に求められる要件を抽出する。

**Nablarch 側の更新頻度前提**: Nablarch 自体は頻繁な機能追加・変更を行わない。v6 リリースノート（6 / 6u1 / 6u2 / 6u3）で確認できる更新は、最新技術への追随・不具合対応・小規模な機能追加が中心。この前提を評価に織り込む（知識更新の発生頻度は低いとみなす）。

**Nablarch固有性・LLM未学習の前提（知識ファイル確認済みの事実）**: Nablarch は汎用LLMの学習データに十分含まれていない、固有用語の多いフレームワークである。これは評価の土台となる重要な前提。
- **【事実】** `classes.md` に登録された Nablarch 固有クラス名は **1,025件**、Javadoc化された固有クラスのドキュメントは **582ファイル**。
- **【事実】** 固有概念が知識ファイル全体に高頻度で登場（`ExecutionContext` 約700回、`ハンドラキュー` 約151回、`UniversalDao` 約134回ほか、`インターセプタ`/`サービス提供可否`/`SystemRepository`/`BatchAction`/`RoutesMapping` 等）。これらは Nablarch 独自の命名・概念体系。
- **【事実】** Nabledge 設計書自身が「AI単体ではNablarch固有知識がなく開発作業の代行精度が低い」を解決課題として明記している。

**この前提が評価に効く点**:
- LLMが未学習であるため、Nabledge・RAG いずれの方式でも**知識を外から注入しないと答えられない**。知識ファイルのトークン注入が必須となり、これがコスト構造の土台になる。
- 固有語彙の密度が高いため、**RAGの汎用埋め込みモデルが意味検索で外しやすい**可能性がある（精度・方式比較の論点に直結）。

**評価スタンス**: 推測を排し、クローンした実物に基づく事実ベース。バイアスなくフラットに。

## 3. 評価軸

以下の評価軸で評価する。必要な軸があれば「追加提案軸」として明示し、提案理由を添える。

### 利用者側
- 回答精度
- 回答コスト（1問あたりのトークンコスト）
- 回答時間
- ランニングコスト
- 初期導入のやりやすさ
- アップデートのやりやすさ

### 開発側
- 新規開発 / 改善の速度
- ランニングコスト
- 提供形態

> 上記に加え、評価の過程で必要と判断した軸があれば「追加提案軸」として明示し、提案理由を添える。

## 4. 利用想定（コスト試算の土台）

数字遊びを避けるため、二層構成で試算する。

### 4-1. ベースコスト（1回あたり）
1回の利用あたりのトークン消費を、**入力 / 出力 / キャッシュ書き込み / キャッシュ読み取り** の4種別に分解する。

**キャッシュ前提**: デフォルト5分TTL（**事実**: Nabledge/CC側コードに `ttl` 明示指定はなく、ベンチマーク実トレースでも全件 `ephemeral_5m` に計上、`ephemeral_1h = 0` を確認済み）。
- 5分以内の連続利用 → キャッシュ読み取り（ベース入力単価の約10%）でヒット
- 5分超の散発利用 → キャッシュ書き込み（約25%単価）が毎回再発生

この「連続利用か散発利用か」でコストが大きく変わるため、ベースコストはこの差を明示して示す。

### 4-2. 規模試算（パターン）
ベースコスト × 利用量で規模感を示す。1人1日20回を基準に、人数でパターン化する（**この人数設定自体は規模感を掴むための仮置きであり試算であることを明示する**）。

| パターン | 想定 | 1日あたり呼び出し |
|---|---|---|
| 小規模 | 10名 × 20回/日 | 200回/日 |
| 中規模 | 50名 × 20回/日 | 1,000回/日 |
| 大規模 | 200名 × 20回/日 | 4,000回/日 |

## 5. 試算の方針（事実の使い方とラベリング）

実環境での完全な実測はできないため、Nabledge・RAG ともに最終的には「試算」になる。これは前提として受け入れる。**ただし試算の土台は、適当な推測ではなく、できる限り既存の事実で固める。**

### 5-1. 事実を土台にする方法
知識ファイルは既に存在するため、その実トークン量を土台に使う。

- **Nabledge 側**: 実在する知識JSON（毎回ロードされる `index.md` 約3.4万トークン・`classes.md` 約1.1万トークン、および精読対象の該当 JSON セクション）の実トークン量を数えて土台にする。
- **RAG 側**: 同じ知識を RAG で持つ場合のチャンク元として、**Nabledge内の閲覧用MD（`docs/` 配下）を使う**。
  - **【事実】** `docs/` 配下のMDは **936ファイル・7.4MB**で知識JSONと1:1対応。**解説書を網羅し、RST記法は除去済み・内部リンクは解決済み**のクリーンなMarkdown（verify設計書で「docs MDはJSONの人間可読レンダリング、JSON⊂MDの内容一致」が要件化されている）。
  - **このMDを選ぶ理由**: RAGの正攻法は「RST/HTMLを前処理してクリーンな可読テキストを作りチャンク化」だが、**その前処理済み成果物が既にこのdocs MDとして存在する**。RAGソースにこれを使えば、チャンク化前提や前処理品質といった推測を最小化でき、試算の土台が事実で固まる。
  - **一次ソース形式の事実**: Nablarch解説書の一次ソースはRST（37群）、システム開発ガイドはMarkdown（3件）、セキュリティ対応表・リリースノートはExcel。公開HTMLはこれらをSphinxでビルドした派生物。RAGをゼロから組む場合これらの前処理が必要になるが、docs MDはその前処理を済ませた形である。

> **ベンチマークのトレースの扱い**: `tools/benchmark/results/` のトレースは実行解析用に本番より多くの情報を出力しているため、トークン数を額面どおりベースコストには使えない。**傾向・構造の参考としてのみ**利用し、一次根拠にはしない。

### 5-2. ラベリング（最重要）
最終文書では、記述の性質を区別できるようにタグで明示する。

- **【事実】**: 計測・確認できたもの（実ファイルのトークン量、リリースノートの内容、コードの仕様など）
- **【試算】**: 事実を土台に単価・回数を掛けて算出したもの
- **【推測】**: 前提として置いた仮説（規模パターンの人数、RAG構成の想定など）

読み手がどれが事実でどれが置いた前提かを判別できる状態を必須とする。

## 6. 評価の対象範囲

### (1) アーキテクチャ面の評価
- RAG（ベクトル検索）等の他方式と比較する。**最新動向を調査した上で**評価する。
- コスト最適化の観点を主軸に置く。例えば RAG であれば「開発側が全社向けにサービスとして提供する」といった、配布形態・運用主体の違いまで含めて比較する。
- 現方式（決定的選定 + 全量配布）の構造的なコスト特性を、他方式と並べて評価する。

### (2) Nabledge 単体のコスト最適化評価
- 現アーキテクチャを前提に、Nabledge 自体でさらにコスト削減できる余地がないかを評価する。
- 利用者側・開発側それぞれのコスト構造に踏み込む。

## 7. 読み手と記述レベル

本文書は有識者レビューだけでなく、**他のメンバーやリーダー層も読む**。

- 読み手は「AIの基礎知識はあり普段使っているが、仕組みや詳細は知らない一般的なエンジニア」を想定する。
- Nabledge・RAG・その他の方式について、仕組みや専門的な部分は**一般的なエンジニアが理解できるように**かみ砕いて説明する（用語は初出時に簡潔な説明を添える）。
- 専門性と平易さを両立させ、レビュー層には論拠の精度を、リーダー層には全体像の理解を提供する。
