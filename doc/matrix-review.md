# 成果物とアクティビティのマトリックス - アーキテクトレビュー

## レビュー観点

1. 各アクティビティで必要なインプットは全て揃っているか
2. 各アクティビティが生成するアウトプットは正しく設定されているか
3. 前工程のOUTが次工程のINとして流れているか
4. 一貫性を保つための参照成果物（方式設計書、開発標準、開発ガイド）が適切か

## 改善提案

### 設計工程

| アクティビティ | 現状の問題 | 改善案 |
|--------------|----------|--------|
| **既存設計書の確認** | 設計書だけ見ても実態が分からない | ソースコードもIN追加（設計と実装の乖離を確認） |
| **機能設計** | 既存機能を参照せずに設計できない | ソースコードもIN追加（既存実装パターンを参照） |
| **データ設計** | 既存テーブル定義を参照しない | ソースコード（DDL、Entity）をIN追加<br/>開発標準（命名規則）をIN追加 |
| **既存パターンの踏襲** | アーキテクチャパターンの参照が不足 | 方式設計書をIN追加 |
| **SQL検証** | 既存SQLと比較検証が必要 | ソースコードをIN追加（既存SQL参照） |
| **設計レビュー** | テスト設計のレビューが抜けている | テスト仕様書/報告書をIN追加（単体テストケース設計の成果物） |

### PG・UT工程

| アクティビティ | 現状の問題 | 改善案 |
|--------------|----------|--------|
| **既存コード理解** | 設計書との突き合わせが必要 | 設計書をIN追加 |
| **影響範囲分析** | 設計レベルの影響も分析が必要 | 設計書をIN追加 |
| **テストデータ作成** | テストケース仕様なしで作れない | テスト仕様書/報告書をIN追加 |
| **コード実装** | 既存パターン参照が必要 | ソースコード、方式設計書をIN追加 |
| **パターン踏襲実装** | 設計との整合性確保が必要 | 設計書、方式設計書をIN追加 |
| **クラス単体テスト作成** | テスト仕様と既存テストコード参照が必要 | 設計書、ソースコード（既存テストコード参照）をIN追加 |
| **コードレビュー** | 設計との整合性チェックが必要 | 設計書、方式設計書をIN追加 |

### 結合テスト工程

| アクティビティ | 現状の問題 | 改善案 |
|--------------|----------|--------|
| **結合テストケース作成** | 業務要件の理解が必要 | 要件定義書をIN追加 |
| **リグレッションテスト観点抽出** | 要件・方式・標準レベルの影響分析が必要 | 要件定義書、方式設計書、開発標準をIN追加 |
| **結合テスト実行** | テスト環境の使い方参照が必要 | 開発ガイドをIN追加 |
| **不具合調査** | 原因特定に設計情報が必要 | 設計書、方式設計書、開発標準、開発ガイドをIN追加 |

### システムテスト工程

| アクティビティ | 現状の問題 | 改善案 |
|--------------|----------|--------|
| **システムテストケース作成** | テスト標準の参照が必要 | 開発標準をIN追加 |
| **システムテスト実行** | テスト環境の使い方参照が必要 | 開発ガイドをIN追加 |
| **性能テスト** | 性能要件・設計との照合が必要 | 要件定義書、設計書、方式設計書をIN追加 |
| **不具合調査** | 結合テストと同様に設計情報が必要 | 設計書、方式設計書、開発標準、開発ガイドをIN追加 |

## 改善の理由

### 1. 既存資産の参照
保守開発では既存の設計書とソースコードの両方を参照して、実態を把握することが不可欠。設計書だけでは実装の乖離が見えない。

### 2. 設計と実装の整合性
各工程で設計書を参照することで、設計意図を正しく理解し、整合性を保つ。

### 3. 一貫性の担保
方式設計書、開発標準、開発ガイドを適切に参照することで、プロジェクト全体の一貫性を保つ。

### 4. 不具合調査の効率化
テスト工程での不具合調査では、要件・設計・実装の全レイヤーの情報が必要。設計情報がないと原因特定に時間がかかる。
