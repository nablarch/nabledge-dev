# 保守開発の工程とアクティビティ（AsIs/ToBe共通）

## ロール定義

| ロール | 主な担当工程 | 担当アクティビティ概要 |
|--------|------------|---------------------|
| **アーキテクト** | 要件定義、設計、PG・UT、結合テスト | 技術検証、基盤設計、開発標準策定、環境構築、アーキテクチャレビュー |
| **要件定義者／設計者** | 要件定義、設計、結合テスト | 要求分析、機能設計（画面・バッチ・Webサービス等）、テストケース設計 |
| **データモデラー** | 設計 | データ定義、DB設計（論理・物理） |
| **データアナリスト** | 設計 | データパターン洗い出し、テストデータ作成 |
| **アプリケーションエンジニア** | PG・UT、結合テスト | アプリ実装、単体テスト作成・実施、障害調査 |

## 工程別アクティビティ

### 1. 要件定義

| アクティビティ | 作業者 | 説明 | IN（インプット） | OUT（アウトプット） |
|--------------|--------|------|----------------|------------------|
| **変更要求分析** | 要件定義者／設計者 | 何を変更するか整理 | - | 要件定義書 |
| **既存機能の影響範囲調査** | 要件定義者／設計者<br/>アーキテクト | 変更により影響を受ける既存機能・設計書・ソースコードの特定 | 要件定義書、既存の設計書、既存のソースコード | 調査/検証結果 |
| **技術的実現可能性検証** | アーキテクト | 既存パターンで実装可能か、新規実装が必要かの判断 | 要件定義書、方式設計書、既存のソースコード | 調査/検証結果 |
| **工数見積** | 要件定義者／設計者<br/>アーキテクト | 変更作業の工数を見積もる（技術的複雑さを含む） | 要件定義書、調査/検証結果 | 工数見積 |

### 2. 設計

| アクティビティ | 作業者 | 説明 | IN（インプット） | OUT（アウトプット） |
|--------------|--------|------|----------------|------------------|
| **既存設計書の確認** | 要件定義者／設計者 | 類似機能の設計書を参照 | 既存の設計書 | 調査/検証結果 |
| **機能設計** | 要件定義者／設計者 | 画面・バッチ・Webサービス等の詳細設計 | 要件定義書、既存の設計書 | 設計書 |
| **データ設計** | データモデラー | テーブル・項目の追加・変更設計 | 要件定義書、既存の設計書 | 設計書 |
| **既存パターンの踏襲** | 要件定義者／設計者 | 既存機能の処理パターンに合わせた設計 | 既存の設計書、既存のソースコード | 設計書 |
| **整合性チェック** | 要件定義者／設計者 | 既存の命名規則・設計パターンとの整合性確認 | 設計書、開発標準 | 調査/検証結果 |
| **設計レビュー** | アーキテクト | 設計内容の確認・承認 | 設計書 | チェックリスト/レビュー結果 |

### 3. PG・UT

| アクティビティ | 作業者 | 説明 | IN（インプット） | OUT（アウトプット） |
|--------------|--------|------|----------------|------------------|
| **環境構築** | アーキテクト | 開発環境のセットアップ | 開発ガイド | - |
| **既存コード理解** | アプリケーションエンジニア | 変更対象の既存コードの処理フローを理解 | 既存のソースコード | - |
| **影響範囲分析** | アプリケーションエンジニア | 変更により影響を受ける他の機能・コードを特定 | 既存のソースコード | 調査/検証結果 |
| **単体テストケース設計** | 要件定義者／設計者 | 変更部分の単体テスト観点を設計 | 設計書 | テスト仕様書/報告書 |
| **SQL検証** | 要件定義者／設計者 | SQLの実現性・性能確認 | 設計書 | 調査/検証結果 |
| **テストデータ作成** | データアナリスト | 単体テスト用のテストデータ作成 | 設計書 | テストデータ |
| **コード実装** | アプリケーションエンジニア | 設計に基づいて変更コードを実装 | 設計書 | ソースコード |
| **パターン踏襲実装** | アプリケーションエンジニア | 既存コードのパターンに合わせて実装 | 既存のソースコード | ソースコード |
| **クラス単体テスト作成** | アプリケーションエンジニア | テストコードを作成 | テスト仕様書/報告書、ソースコード | ソースコード（テストコード） |
| **クラス単体テスト実施** | アプリケーションエンジニア | テストを実行し結果を確認 | ソースコード、テストデータ | テスト仕様書/報告書 |
| **取引単体テスト実施** | アプリケーションエンジニア | 取引単位の打鍵テスト実施 | ソースコード、テストデータ | テスト仕様書/報告書 |
| **コードレビュー** | アーキテクト | コード確認・承認 | ソースコード、開発標準 | チェックリスト/レビュー結果 |

### 4. 結合テスト

| アクティビティ | 作業者 | 説明 | IN（インプット） | OUT（アウトプット） |
|--------------|--------|------|----------------|------------------|
| **結合テスト環境構築** | アーキテクト | テスト環境のセットアップ、デプロイ | 開発ガイド、ソースコード | - |
| **結合テストケース作成** | 要件定義者／設計者 | 機能間連携のテストケース作成 | 設計書 | テスト仕様書/報告書 |
| **リグレッションテスト観点抽出** | 要件定義者／設計者 | 既存機能への影響を確認するテスト観点を抽出 | 設計書、既存のソースコード | テスト仕様書/報告書 |
| **結合テスト実行** | 要件定義者／設計者 | テストを実行 | ソースコード、テスト仕様書/報告書、テストデータ | テスト仕様書/報告書 |
| **不具合調査** | アプリケーションエンジニア<br/>要件定義者／設計者 | テスト失敗時の原因特定（コード不具合、IF不整合、データ連携ミス等） | テスト仕様書/報告書、ソースコード | 不具合票 |

### 5. システムテスト

| アクティビティ | 作業者 | 説明 | IN（インプット） | OUT（アウトプット） |
|--------------|--------|------|----------------|------------------|
| **システムテストケース作成** | 要件定義者／設計者 | 業務フロー全体のテストケース作成 | 要件定義書、設計書 | テスト仕様書/報告書 |
| **システムテスト実行** | 要件定義者／設計者 | テストを実行 | ソースコード、テスト仕様書/報告書、テストデータ | テスト仕様書/報告書 |
| **性能テスト** | アーキテクト | 処理時間・スループットの確認、ボトルネック特定 | ソースコード、テスト仕様書/報告書 | テスト仕様書/報告書、調査/検証結果 |
| **不具合調査** | アプリケーションエンジニア<br/>要件定義者／設計者 | テスト失敗時の原因特定 | テスト仕様書/報告書、ソースコード | 不具合票 |

## 成果物分類

| 分類 | 含まれる成果物 |
|------|--------------|
| **1. 要件定義書** | 変更要求仕様書 |
| **2. 設計書** | システム機能設計書、テーブル定義書 |
| **3. 方式設計書** | 基盤設計、アーキテクチャ設計 |
| **4. 開発標準** | コーディング規約、UI標準、テスト標準 |
| **5. 開発ガイド** | プロジェクト用開発ガイド |
| **6. ソースコード** | Java/JSP/SQL、テストコード |
| **7. テスト仕様書/報告書** | テストケース、テスト結果、性能テスト結果 |
| **8. テストデータ** | 共通テストデータ、テストケース用データ |
| **9. 不具合票** | 障害票、設計修正指示 |
| **10. チェックリスト/レビュー結果** | レビューチェックリスト、セルフチェックリスト、レビュー指摘事項 |
| **11. 調査/検証結果** | 影響範囲調査書、技術検証結果、SQL検証結果、性能改善案 |
| **12. 工数見積** | 見積データ、見積書 |

## 備考

- このアクティビティ定義はAsIs（従来）とToBe（Nabledge活用後）で共通
- ToBeではNabledgeがアクティビティを支援・自動化するが、アクティビティ自体は変わらない
- 成果物分類により、Nabledgeがどの成果物にどう影響するかを可視化可能
